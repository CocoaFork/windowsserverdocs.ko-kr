# <a name="windows-authentication-concepts"></a>Windows 인증 개념

>적용 대상: Windows Server (세미콜론 연간 채널) Windows Server 2016

참조 개요 여기서 Windows 인증 삼고 개념에 설명 합니다.

인증은 프로세스 개체 또는 사용자의 신원을 확인 합니다. 개체 인증할 때 정품 인지 확인 하기 위해 목표가입니다. 사람 인증할 때 사람 방법으로든 되었는지 확인 하는 됩니다.

네트워킹 컨텍스트에서 인증 신원을 네트워크 응용 프로그램이 나 리소스에 증명 하는 작업입니다. 일반적으로 공유 키 또는 (와 마찬가지로 키 공개 암호화)를 키 사용자만 아는 사용 하 여 신원은 암호화 작업 하 여 입증입니다. 인증 거래소의 서버 쪽 인증 시도 유효성을 검사 하기 알려진된 암호화 키를 서명된 된 데이터를 비교 합니다.

확장 하 고 관리 하기 쉬운 인증 프로세스를 사용 하면 암호화 키 안전 하 게 한곳에 저장 합니다. 키를 포함 하 여 암호화 id 정보를 저장 하기 위한 기본 기술 및 active Directory 권장 되는 사용자의 자격 증명은 합니다. Active Directory NTLM 기본 및 Kerberos 구현 필요 합니다.

인증 기술을 범위 간단 하 게 로그온에서 운영 체제 또는 사용자를 식별 하는 응용 프로그램 또는 서비스에 로그인 하는 사용자가 알고 있는, 암호를 등 토큰 공개 키 인증서도 사용자 has'such 사진 하는 것을 사용 하는 강력한 보안 메커니즘 또는 생물학적 특성 항목에 따라 합니다. 비즈니스 환경에서 사용자 여러 응용 프로그램에서 다양 한 유형의 서버 또는 한곳에서 여러 위치에서 액세스할 수 있습니다. 이러한 이유로 인증 환경 및 다른 Windows 운영 체제에 대 한 다른 플랫폼에 대 한 지원 해야 합니다.

Windows 클라이언트/서버가 아키텍처에서 네트워크 보안을 위해 전이. 보안 관계 도메인 밤나무 등의 도메인의 세트 전체 및 도메인 하 고 해당 도메인 신뢰 하는 모든 도메인 간의 관계를 구성 합니다. 예를 들어 A 도메인에 전이 도메인 B와 및 도메인 B 도메인 C 신뢰 하는 경우 다음 도메인 A 신뢰 c

## <a name="credentials"></a>자격 증명
Passport 및 줄 관련된 visas는 여행자에 대해 승인 된 자격 증명 됩니다. 그러나 해당 자격 증명 입력 하거나 액세스 모든 리소스를 국가 내의 여행자를 수 없습니다. 예를 들어 추가 자격 증명은 회의 참석 하는 데 필요한 합니다. Windows에서 계정 소유자 반복적으로 자격 증명을 제공 하지 않고도 네트워크 리소스에 액세스할 수 있게 하 자격 증명을 관리할 수 있습니다. 이러한 종류의 액세스 한 번 모든 응용 프로그램에 액세스할 수 있도록 시스템에서 인증 하는 사용자가 수와 데이터 다른 식별자 계정 또는 암호를 입력 하지 않고 사용할 권한이 있는 원본의 합니다. Windows 플랫폼 로컬 사용자의 운영 체제의 LSA 로컬 보안 기관을 () 자격 증명을 캐시 하 여 네트워크를 통해 (유지 Active Directory) 단일 사용자 id를 사용 하는 기능을 활용 합니다. 사용자가 도메인 로그온, Windows 인증 패키지 네트워크 리소스를 자격 증명 인증 하는 경우 단일 로그인을 제공 하기 위해 자격 증명을 투명 하 게 사용 합니다. 자격 증명에 대 한 자세한 내용은 참조 [Windows 인증에서 자격 증명 프로세스](credentials-processes-in-windows-authentication.md)합니다.

양식의는 여행자 다단계 인증을 수행 하 고 여러 문서를 passport와 회의 등록 정보 등 그의 id를 인증에 존재 하는 요구 사항을 수도 있습니다. Windows에는이 양식 또는 인증 스마트 카드, 가상 스마트 카드 생체 인식 기술을 통해 구현합니다. 

## <a name="security-principals-and-accounts"></a>보안 사용자 및 계정
Windows에서 사용자, 서비스, 그룹 또는 작업을 시작할 수 있는 컴퓨터 보안 사용자입니다. 보안 사용자 계정이, 로컬 컴퓨터에 하거나 도메인 기반 될 수 있는 합니다. 예를 들어 Windows 클라이언트 도메인에 가입 컴퓨터 로그온 한 휴먼 사용자가 때도 도메인 컨트롤러와 통신 하 여 네트워크 도메인에 참여할 수 있습니다. 통신을 시작 하려면 컴퓨터 도메인에 있는 현재 계정이 있어야 합니다. 컴퓨터의 통신을 받기 전에 도메인 컨트롤러에서 로컬 보안 기관을 컴퓨터의 id를 인증 및 인간의 보안 사용자에 대 한 것 처럼 후 컴퓨터의 보안 컨텍스트 정의 합니다. 이 보안 컨텍스트 특정 컴퓨터 또는 사용자, 서비스, 그룹 또는 네트워크에서 컴퓨터 id 및 사용자 또는 서비스의 기능이 정의합니다. 예를 들어, 파일 공유 하거나 프린터에 액세스할 수 있는 및 읽기, 쓰기 또는 사용자, 서비스 또는 해당 리소스에는 컴퓨터에서 수행할 수 있는 수정 등의 작업을 리소스를 정의 합니다. 자세한 내용은 참조 [보안 사용자](https://technet.microsoft.com/itpro/windows/keep-secure/security-principals)합니다.

각 버전의 Windows에 기본 제공 된 계정 및은 구성원이는 보안 그룹 정의 됩니다. 보안 그룹을 사용 하 여 지정할 수 있습니다와 동일한 보안 권한 많은 사용자에 게 성공적으로 인증 액세스 관리를 간소화 하는. ## <a name="delegated-authentication"></a>위임된 인증
여행 비유를 사용 하려면 국가 수 동일한 액세스에 발급 공식 정부 위임의 모든 구성원이 것으로 유명는 위임 하는 합니다. 가정 한 구성원이이 위임 다른 멤버의 기관에 적용 됩니다. Windows에서 위임된 인증 네트워크 서비스에서 사용자에 게 서 인증 요청을 승인 하 고에서는 해당 사용자의 id를 사용 하기 위해 네트워크 서비스 두 번째에 새 연결을 시작할 때 발생 합니다. 위임된 인증을 지원 하기 위해 클라이언트 인증 요청을 처리할 담당 하는 웹 서버 등 프런트 또는 1 단계 서버 및 대규모 데이터베이스 정보를 저장 하는 일을 담당 하는 등 백 엔드 또는 n 티어 서버 설정 해야 합니다. 관리자의 관리자 로드 줄이기 위해 조직에는 사용자에 게 위임된 인증을 설정 하 고 오른쪽을 위임 수 있습니다.

위임 신뢰할 수 있는로 서비스 또는 컴퓨터를 구축 하 여 해당 서비스 또는 위임된 인증을 완료을 요청 하 고 있는 사용자에 대 한 티켓 수신 하 고 해당 사용자에 대 한 정보에 액세스 한 다음 컴퓨터를 사용 합니다. 이 모델 제한 백 엔드 서버에 데이터 액세스 해당 사용자 또는 서비스 하 여 올바른 액세스 제어 토큰와 현재 해당 자격 증명 합니다. 또한 백 엔드 리소스에 액세스 감사 수 있게 합니다. 클라이언트 대신 사용할 수 있도록 서버에 위임 하는 자격 증명을 사용 하 여 모든 데이터에 액세스할 수 있다는 함으로써 서버 손상 되지 및 기타 서버에 저장 되는 중요 한 정보에 액세스할 수를 확인 합니다. 위임된 인증 여러 컴퓨터 단일 인식 기능을 사용 하도록 설계 된 계층 응용 프로그램에 유용 합니다.

### <a name="authentication-in-trust-relationships-between-domains"></a>신뢰 하는 도메인 관계에는 인증
둘 이상의 도메인 합법적가 있는 대부분의 조직 리소스에 액세스 공유를 다른 도메인에 있는 여행자 여행 국가 서로 다른 영역을 허용 하는 것 처럼 사용자가 필요 합니다. 이 액세스를 제어 하려면는 사용자가 도메인에도 고 인증할 수 있는 다른 도메인 리소스를 사용할 수 있는 권한이 필요 합니다. 다른 도메인 수정 하는 클라이언트와 인증과 기능을 제공 하려면 두 도메인 신뢰를 이어야 합니다. 신뢰는 기본 기술이 보안된 Active Directory 통신 발생 하 고 있는 Windows Server 네트워크 아키텍처 가장 중요 한 보안 구성 됩니다.

신뢰 두 도메인에 있는 경우 각 도메인 인증 메커니즘 신뢰 다른 도메인에서 제공한 것임 인증 합니다. 신뢰 수신 인증을 확인 하 여 리소스 도메인-신뢰 도메인-의 공유 리소스에 액세스 제어에 대해 제공을 지원 요청 신뢰할 수 있는 권한을-신뢰할 수 있는 도메인에서 제공 됩니다. 이렇게에서만 수 있게 해 주는 다리 도메인 간에 요청 여행 인증의 유효성을 검사 신뢰 작동 합니다.

특정 신뢰 전달 인증 요청 어떻게 구성에 따라 다릅니다. 신뢰 관계 각 도메인 다른 도메인 리소스에에서 액세스 하 여 단방향 신뢰할 수 있는 도메인 신뢰할 도메인 리소스에에서 액세스 하 여 또는 양방향 될 수 있습니다. 신뢰 하는 대/소문자 신뢰 두 신뢰 파트너 도메인 간만 또는 전이 존재 하는 중 비 전이적도, 보안 파트너 중 하나를 신뢰 하는 다른 모든 도메인에 자동으로 확장 있는 경우 합니다.

신뢰를 작동 하는 방법에 대 한 정보를 참조 하세요. [도메인 어떻게 및 숲 신뢰 작업](https://technet.microsoft.com/library/cc773178(v=ws.10).aspx)합니다.

### <a name="protocol-transition"></a>프로토콜 전환
프로토콜 전환 사용자 인증 층에서 다른 인증 메커니즘을 지 원하는 응용 프로그램을 하 고 보안 기능을 상호 인증 Kerberos 프로토콜으로 전환 하 여 응용 프로그램 디자이너 수 있도록 지 원하는 및 위임 계층 후속 응용 프로그램에서에서 제한 됩니다.

프로토콜 전환에 대 한 자세한 내용은 참조 [Kerberos 프로토콜 전환 하 고 제한 위임](https://technet.microsoft.com/library/cc758097(v=ws.10).aspx)합니다.

### <a name="constrained-delegation"></a>제한 된 위임
제한 된 위임 관리자 지정 하 고 사용자를 대신 하 여 응용 프로그램 서비스 역할 할 수 있는 범위를 제한 하 여 응용 프로그램 신뢰 경계를 적용 하는 기능을 제공 합니다. 위임 신뢰할 수 있는 컴퓨터를 리소스를 요청할 수 있는 특정 서비스를 지정할 수 있습니다. 서비스에 대 한 권한의 제한 하는 유연성 신뢰할 수 없는 서비스에 의해 손상 될 확률이 줄여 보안 디자인 응용 프로그램을 개선 하는 데 도움이 됩니다.

제한 된 위임에 대 한 자세한 내용은 참조 [제한 위임 개요 Kerberos](../kerberos/kerberos-constrained-delegation-overview.md)합니다.
