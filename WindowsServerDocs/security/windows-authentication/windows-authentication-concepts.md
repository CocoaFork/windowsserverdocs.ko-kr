# <a name="windows-authentication-concepts"></a>Windows 인증 개념

>적용 대상: Windows Server (반기 채널), Windows Server 2016

이 참조 개요 항목에서는 Windows 인증의 기반이 되는 개념을 설명 합니다.

인증은 개체 또는 사용자의 id를 확인 하는 프로세스입니다. 개체 인증의 목표는 개체가 올바른지 확인하는 데 있습니다. 사용자를 인증할 때의 목표는 사용자는 가짜 아닌지 확인입니다.

네트워킹 컨텍스트에서 인증은 네트워크 응용 프로그램이나 리소스에 대해 ID를 증명하는 행위입니다. 일반적으로 id가 암호화 작업을 통해 증명 (마찬가지로 공개 키 암호화) 키 사용자만 알고 사용 하 여 또는 공유 키입니다. 인증 교환의 서버 쪽에서는 서명된 데이터를 알려진 암호화 키와 비교하여 인증 시도의 유효성을 검사합니다.

암호화 키를 안전한 중앙 위치에 저장하면 인증 프로세스를 쉽게 확장하고 유지 관리할 수 있습니다. Active Directory는 권장 되는 키 암호화를 포함 하는 id 정보를 저장 하기 위한 기본 기술 및 사용자의 자격 증명이 있습니다. 기본 NTLM 및 Kerberos 구현에는 Active Directory가 필요합니다.

운영 체제 또는 서비스 또는 사용자만 아는, 암호와 같은 항목을 사용 하는 더 강력한 보안 메커니즘으로는 항목을 기반으로 하는 사용자를 식별 하는 응용 프로그램에 로그인 하는 간단한 로그온에서 인증 기술을 범위는 토큰, 공개 키 인증서, 그림 또는 생물학적 특성으로 사용자 has'such 합니다. 사용자는 비즈니스 환경에서 다양 한 유형의 단일 위치 내에서 또는 여러 위치에서 서버에서 여러 응용 프로그램을 액세스할 수 있습니다. 이러한 이유로, 인증은 다른 플랫폼 및 다른 Windows 운영 체제의 환경을 지원해야 합니다.

## <a name="authentication-and-authorization-a-travel-analogy"></a>인증 및 권한 부여: 여행 비유
여행 비유는 인증의 작동 원리를 설명 하는 데 도움이 됩니다. 몇 가지 준비 작업은 일반적으로 필요한 작업을 시작 합니다. 휴대용 해당 호스트 기관에 true 신원을 증명 해야 합니다. 이 증명 citizenship, 출생 내부, 개인 바우처, 사진 또는 호스트 국가의 법률에 필요한 모든 개념 증명의 형태로 수 있습니다. 휴대용의 id가 발급 하 고 조직-보안 주체 관리 시스템 계정을 비슷합니다는 passport 발급 하 여 유효성을 확인 합니다. Passport와 의도 한 대상 규칙 및 정부 기관에서 발급 한 규정 집합을 기반으로 합니다.

**과정**

휴대용 국제 테두리에 도착 하는 경우 테두리 가드를 자격 증명을 요청 하 고는 휴대용 자신의 passport를 제공 합니다. 프로세스는 두 가지가 있습니다.

-   가드는 passport 수정 되지 않았는지 확인 하 고 지방 정부 트러스트 (트러스트의 경우 적어도 여권 발급)는 보안 기관에서 발급 된를 확인 하 여 passport를 인증 합니다.

-   가드는 다른 필수 자격 증명이 올바르게에서 및 얼굴 passport에서 나온 사람의 얼굴 일치 하는지 확인 하 여는 휴대용을 인증 합니다.

Passport 입증 유효 하는 휴대용 입증 해당 소유자 여야 하 고 인증에 성공 하면 및를 휴대용 테두리에서 액세스 허용 될 수 있습니다.

보안 기관 간의 전이적 트러스트는 인증의 기초 국제 경계에서 발생 하는 인증 유형 신뢰를 기반으로 합니다. 지방 정부는 휴대용 알지 하지만 호스트 government 않은 것을 신뢰 합니다. 호스트 government passport에 실행 하면 해당 모르는 휴대용 하거나 합니다. 이 신뢰 출생 증명서 또는 기타 설명서를 발급 한 기관. 출생 증명서를 발급 한 기관에 인증서를 서명한 의사를 신뢰할 수 있습니다. 의사는 휴대용의 생년월일을 경험 및 newborn의 공간을 사용 하 여 직접 증빙 id 사용 하 여 인증서를이 경우에 자동 삽입 합니다. 이러한 방식으로 신뢰할 수 있는 매개 자를 통해 전송 되는 트러스트는 전이적입니다.

전이적 트러스트에 Windows 클라이언트/서버 아키텍처의 네트워크 보안을 위한 기초입니다. 트러스트 관계를 진행 하는 도메인 트리를 같은 도메인 집합 되면서 및 도메인과 해당 도메인을 신뢰 하는 모든 도메인 간의 관계를 구성 합니다. 예를 들어 도메인 A 도메인 B 전이적 트러스트 관계가 도메인 B C 도메인을 신뢰 하는 경우 다음 도메인은 도메인을 신뢰 3.

인증 및 권한 부여 간에 차이가 있습니다. 인증을 사용 하 여 시스템 시겠습니까 사람의 입증 합니다. 권한 부여를 사용 하 여 시스템 작업을 수행 하 시겠습니까 권한이 있는지 확인 합니다. 다음 단계로 테두리 비유를 사용 하려면 단순히는 휴대용이 유효한 passport 적절 한 소유자를 인증 반드시 권한을 부여 하지 않습니다 국가 입력 하는 휴대용입니다. 특정 국가 거주자는 단순히 입력 되는 국가 입력에 해당 특정 국가의 모든 시민 들에 게 무제한 권한을 부여 하는 있는 경우에만 passport를 제공 하 여 다른 국가 입력 수 있습니다.

마찬가지로, 모든 사용자는 특정 도메인 권한 리소스에 액세스를 부여할 수 있습니다. 캐나다 있습니다 미국 시민 캐나다를 입력 하는 것 처럼 해당 도메인에 속한 모든 사용자를 리소스에 액세스할 수 있습니다. 그러나 브라질 또는 인도 진입 하려고 하는 미국 시민 찾을 유효한 visa 할 미국 시민 방문 해당 국가의 모두 필요 하기 때문에 passport를 제공 하 여 단순히 해당 국가의 입력할 수는 없습니다. 따라서 인증 리소스 또는 리소스를 사용 하도록 권한 부여에 대 한 액세스를 보장 하지 않습니다.

## <a name="credentials"></a>자격 증명
Passport 관련이 있을 visas와 허용 되는 휴대용 자격 증명입니다. 그러나 해당 자격 증명을 휴대용 입력 하거나 단일 국가 내 모든 리소스에 액세스할 수 없는 경우도 있습니다. 예를 들어 추가 자격 증명을 컨퍼런스에 참가할 필요 합니다. Windows를 반복적으로 자격 증명을 지정 하지 않고 네트워크를 통해 리소스에 액세스 하려면 계정 소유자에 대 한 수 있도록 자격 증명을 관리할 수 있습니다. 다른 계정 id 또는 암호를 입력 하지 않고 사용 권한이 있는 데이터 원본 및 이러한 유형의 액세스 사용자 시스템 모든 응용 프로그램에 액세스 하는 데 한 번에 인증할 수 있습니다. 로컬 사용자 자격 증명에서 운영 체제의 로컬 보안 기관 (LSA)를 캐시 하 여 네트워크를 통해 단일 사용자 id (Active Directory에서 유지 관리 됨)를 사용 하는 기능 Windows 플랫폼을 활용 합니다. 사용자 도메인에 로그온 하면 Windows 인증 패키지 네트워크 리소스에 자격 증명을 인증 하는 경우 single sign on 제공 자격 증명을 투명 하 게 사용 합니다. 자격 증명에 대 한 자세한 내용은 참조 하세요. [Windows 인증에서 자격 증명 프로세스](credentials-processes-in-windows-authentication.md)합니다.

두 가지 형태의 휴대용 용 multi-factor authentication 수행 하 고 등록 정보를 passport 및 회의 같은 id를 인증 하는 여러 문서를 제공 하는 요구 사항을 수도 있습니다. Windows는이 형식 또는 스마트 카드, 가상 스마트 카드 및 생체 인식 기술을 통해 인증을 구현합니다. 

## <a name="security-principals-and-accounts"></a>보안 주체 및 계정
Windows, 사용자, 서비스, 그룹 또는 작업을 시작할 수 있는 컴퓨터는 보안 주체입니다. 보안 주체 계정을 컴퓨터에 로컬 또는 도메인 기반 수 있는 경우 예를 들어, Windows 클라이언트 도메인에 가입 된 컴퓨터는 네트워크 도메인에 없는 사용자가 로그온 하는 경우에 도메인 컨트롤러와 통신 하 여 참여할 수 있습니다. 통신을 시작 하려면 컴퓨터를 도메인에는 활성 계정이 있어야 합니다. 컴퓨터에서 통신을 수락 하기 전에 도메인 컨트롤러에서 로컬 보안 기관 컴퓨터의 id를 인증 하 고 사용자 보안 주체에 대 한 것 처럼 컴퓨터의 보안 컨텍스트를 정의 합니다. 이 보안 컨텍스트에서 특정 컴퓨터 또는 사용자, 서비스, 그룹 또는 네트워크의 컴퓨터에서 id 및 사용자 또는 서비스의 기능을 정의합니다. 예를 들어, 파일 공유 또는 액세스할 수 있는 프린터와 같은 읽기, 쓰기 또는 수정, 사용자, 서비스 또는 해당 리소스에 대해 컴퓨터에서 수행할 수 있는 동작을 같은 리소스를 정의 합니다. 자세한 내용은 [보안 주체](https://technet.microsoft.com/itpro/windows/keep-secure/security-principals)합니다.

계정이는 사용자 또는 서비스 액세스 또는 리소스가 요청 보험금 청구-방식의 식별 수단을 됩니다. 인증 된 passport를 보유 하는 휴대용 호스트 국가 사용 하 여 계정을 있습니다. 사용자, 그룹 사용자, 개체 및 서비스의 모든 계정이 있습니다 개별 계정을 공유 합니다. 계정 그룹의 구성원 수 및 특정 권한 및 사용 권한을 할당할 수 있습니다. 계정에는 로컬 컴퓨터, 작업 그룹, 네트워크를 제한할 수 있습니다 또는 도메인 멤버 자격을 할당할 수입니다.

기본 제공 계정 및 보안 그룹, 멤버는 자신이 Windows의 각 버전에 정의 됩니다. 보안 그룹을 사용 하 여 권한을 할당할 수 있습니다는 동일한 보안을 성공적으로 인증 하는 많은 사용자에 게 액세스 관리를 간소화 하는 합니다. 여권 실행에 대 한 규칙은 휴대용 여행자, 비즈니스 또는 government와 같은 특정 그룹을 할당할 수 있음이 필요할 수 있습니다. 이 프로세스는 그룹의 모든 구성원 간의 일관 된 보안 권한의 보장합니다. 사용 권한을 할당 하려면 보안 그룹을 사용 하 여 리소스의 제어에 액세스 하는 방법에는 지속적이 고 쉽게 관리 및 감사 유지 됩니다. 를 추가 하 고 필요에 따라 적절 한 보안 그룹에서 액세스를 필요로 하는 사용자를 제거 하 여 액세스 제어 목록 (Acl) 변경 빈도 최소화할 수 있습니다.

독립 실행형 관리 서비스 계정 및 가상 계정 자체 도메인의 격리를 사용 하 여 Microsoft Exchange Server 인터넷 정보 서비스 (IIS) 등의 필요한 응용 프로그램을 제공 하는 Windows Server 2008 R2 및 Windows 7에 도입 합니다. 이러한 계정에 대 한 자격 증명 및 서비스 주체 이름 (SPN)을 수동으로 관리 하려면 관리자에 대 한 필요를 제거 하는 동안 계정입니다. 그룹 관리 서비스 계정 도메인 내에서 동일한 기능을 제공 하며 여러 서버를 통해 해당 기능을 확장 하는 또한 Windows Server 2012에서 도입 되었습니다. 네트워크 부하 분산과 같이 서버 팜에서 호스트된 서비스에 연결하는 경우 상호 인증을 지원하는 인증 프로토콜을 사용하려면 모든 서비스 인스턴스가 동일한 사용자를 사용해야 합니다.

## <a name="delegated-authentication"></a>위임 된 인증
여행 비유를 사용 하려면 국가 발생할 동일한 액세스는 공식 정부 위임의 모든 멤버에만으로 대리자는 잘 알려진 합니다. 이 위임에는 하나의 멤버가 다른 멤버의 권한을 작업할 수 있습니다. Windows, 위임 된 인증 네트워크 서비스 사용자의 인증 요청을 수락 하 고 두 번째 네트워크 서비스에 새 연결을 시작 하려면 해당 사용자의 id를 가정 하는 경우 발생 합니다. 위임 된 인증을 지원 하려면 클라이언트 인증 요청을 처리 하는 일을 담당 하는 웹 서버와 같은 프런트 엔드 또는 첫 번째 계층 서버 및 담당 하는 대규모 데이터베이스와 같은 백 엔드 또는 n 계층 서버를 설정 해야 합니다. 정보를 저장 합니다. 관리자의 관리 로드를 줄이려면 사용자 조직에 게 위임 된 인증 설정 수 있는 권한을 위임할 수 있습니다.

위임용으로 서비스 또는 컴퓨터를 설정, 하 여 해당 서비스 또는 위임 된 인증을 완료 하는 요청 하는 사용자에 대 한 티켓을 받고 그런 다음 해당 사용자에 대 한 정보는 컴퓨터는 것이 있습니다. 이 모델 데이터 액세스 제한 백 엔드 서버에서 해당 사용자 또는 서비스에만 올바른 access control 토큰에 있는 해당 자격 증명 합니다. 또한 이러한 백 엔드 리소스의 액세스 감사 수 있습니다. 클라이언트를 대신 하 여 사용을 위해 서버에 위임 된 자격 증명을 사용 하 여 모든 데이터에 액세스할 수 있는지를 요구 하 여 서버를 손상 될 수 없습니다 및 다른 서버에 저장 되는 중요 한 정보에 액세스할 수 있는지 확인 합니다. 위임 된 인증 여러 컴퓨터 간에 single sign-on 기능을 사용 하도록 설계 된 다중 계층 응용 프로그램에 유용 합니다.

### <a name="authentication-in-trust-relationships-between-domains"></a>도메인 간의 트러스트 관계에서 인증
타당 한 이유가 둘 이상의 도메인에 있는 대부분의 조직에서는 휴대용 여행 국가에서 서로 다른 지역에 허용 되는 다른 도메인에 있는 공유 리소스에 액세스 하는 사용자에 대 한 필요 합니다. 이 액세스 제어는 한 도메인의에서 사용자 인증 및 수도 다른 도메인의 리소스를 사용할 수 있는 권한이 필요 합니다. 클라이언트와 서버의 서로 다른 도메인에서 인증 및 권한 부여 기능을 제공 하는 두 도메인 간에 트러스트를 여야 합니다. 트러스트는 Active Directory의 보안된 통신 발생 하는 Windows Server 네트워크 아키텍처의 필수적인 보안 구성 요소는 기본 기술입니다.

두 도메인 간에 트러스트가 있으면 각 도메인에 대 한 인증 메커니즘에 다른 도메인에서 오는 인증 신뢰 합니다. 트러스트 들어오는 인증을 확인 하 여 리소스 도메인--트러스팅 도메인의 공유 리소스에 대 한 액세스 제어를 제공 하는 데 도움이 되는 신뢰할 수 있는 기관-트러스트 된 도메인에서에서 요청 제공 합니다. 이러한 방식으로 트러스트 프록시로 있도록 브리지 도메인 간의 인증 요청이 이동의 유효성을 검사 합니다.

특정 신뢰 인증 요청을 전달 하는 방법을 구성 하는 방법에 따라 달라 집니다. 각 도메인에서 다른 도메인의 리소스에 액세스를 제공 하 여 신뢰할 수 있는 도메인에서 트러스팅 도메인의 리소스에 액세스를 제공 하 여 단방향 또는 양방향 트러스트 관계 수입니다. 사례 트러스트는 두 가지 신뢰 파트너 도메인 사이만 또는 전이적 존재 하는 중 비 전이적도, 트러스트 파트너 중 하나에서 신뢰 하는 다른 도메인에 자동으로 확장 경우 신뢰 합니다.

트러스트를 작동 하는 방법에 대 한 자세한 내용은 참조 하세요. [어떻게 도메인 및 포리스트 트러스트 작업](https://technet.microsoft.com/library/cc773178(v=ws.10).aspx)합니다.

### <a name="constrained-delegation"></a>제한된 위임
제한 된 위임 관리자를 지정 하 고 응용 프로그램 서비스 사용자를 대신할 수 있는 범위를 제한 하 여 응용 프로그램 트러스트 경계를 적용 하는 기능을 제공 합니다. 위임을 위해 신뢰할 수 있는 컴퓨터 리소스를 요청할 수는 특정 서비스를 지정할 수 있습니다. 서비스에 대 한 권한 부여 권한을 제한 하는 유연성에 신뢰할 수 없는 서비스에서 손상에 대 한 기회를 줄여 응용 프로그램 보안 설계를 개선 하는 데 도움이 됩니다.
