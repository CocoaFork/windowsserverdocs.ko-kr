# <a name="plan-a-software-defined-network-infrastructure"></a>소프트웨어 정의 네트워크 인프라 계획

>적용 대상: Windows Server (반기 채널), Windows Server 2016

하드웨어 및 소프트웨어 필수 구성 요소를 포함 하 여 소프트웨어 정의 네트워크 인프라를 계획 하는 배포에 알아봅니다. 


## <a name="prerequisites"></a>사전 요구 사항
이 항목에서는 다양을 한 하드웨어 및 소프트웨어 필수 구성 요소를 포함 하 여 설명 합니다.

-   **보안 그룹, 로그 파일 위치 및 동적 DNS 등록 구성** 하나 이상의 컴퓨터 또는 Vm 및 하나의 컴퓨터 또는 VM을 해야 하는 네트워크 컨트롤러 배포에 대 한 데이터 센터를 준비 해야 합니다. 네트워크 컨트롤러를 배포 하기 전에 보안 그룹, 로그 파일 위치 (필요한 경우) 및 동적 DNS 등록을 구성 해야 합니다.  네트워크 컨트롤러 배포에 대 한 데이터 센터를 하지 준비한 경우 참조 [설치 및 네트워크 컨트롤러 배포에 대 한 준비 요구 사항](Installation-and-Preparation-Requirements-for-Deploying-Network-Controller.md) 세부 정보에 대 한 합니다.

-   **실제 네트워크** 실제 네트워크 장치를 구성 Vlan, 라우팅, BGP, 데이터 센터 브리징 (ETS)는 RDMA 기술을 사용 하는 경우에 액세스 해야 하 고 데이터 센터 브리징 (PFC)는 RoCE를 사용 하는 경우 RDMA 기술을 기반으로 합니다. 이 항목에서는 3 계층 스위치에서 BGP 피어 링 뿐만 아니라 수동 전환 구성 표시 / 라우터 또는 라우팅 및 원격 액세스 서버 (RRAS) 가상 컴퓨터.   

-   **물리적 호스트 계산** 이러한 호스트에서 Hyper-v를 실행 하 고 SDN 인프라 및 테 넌 트 가상 컴퓨터를 호스트 하는 데 필요한 합니다.  특정 네트워크 하드웨어의 뒷부분에서 설명 하는 최상의 성능을 위해 이러한 호스트에 필요 합니다 **네트워크 하드웨어** 섹션입니다.  


## <a name="physical-network-and-compute-host-configuration"></a>물리적 네트워크 및 계산 호스트 구성

각 물리적 계산 호스트에는 물리적 스위치 포트에 연결 하는 하나 이상의 네트워크 어댑터를 통해 네트워크 연결이 필요 합니다.  계층 2 [VLAN](https://en.wikipedia.org/wiki/Virtual_LAN) 여러 논리 네트워크 세그먼트로 분리 하는 네트워크를 지원 합니다. 

>[!TIP]
>액세스 모드에서 논리 네트워크에 대 한 VLAN 0을 사용 하거나 태그 없음. 

>[!IMPORTANT]
>Windows Server 2016 소프트웨어 정의 네트워킹 언더레이 및 오버레이 대 한 IPv4 주소 지정 하는 작업을 지원 합니다. IPv6이 지원되지 않음

### <a name="logical-networks"></a>논리 네트워크

#### <a name="management-and-hnv-provider"></a>관리 및 HNV 공급자

DHCP 서버 관리 네트워크에 대 한 IP 주소를 자동으로 할당할 수 또는 고정 IP 주소를 수동으로 할당할 수 있습니다. SDN 스택을 네트워크 컨트롤러에서 HNV 공급자 논리 네트워크 IP 풀에서 개별 Hyper-v 호스트를 통해 지정 및 관리에 대 한 IP 주소를 자동으로 할당 합니다.

## <a name="routing-infrastructure"></a>라우팅 인프라  

스크립트를 사용 하 여 SDN 인프라를 배포 하는 경우 관리, HNV 공급자, 전송 중 및 VIP 서브넷 실제 네트워크에서 서로 라우팅 가능 해야 합니다.     

라우팅 정보 \(예를 들어 다음 홉\) VIP에 대 한 서브넷은 통해 보급 된 SLB/MUX 및 RAS 게이트웨이 내부 BGP 피어 링을 사용 하 여 실제 네트워크에 있습니다. VIP 논리 네트워크 할당 된 VLAN이 없는 되며 미리 구성 된 계층 2 스위치 (예:-Top-of-rack 스위치)에 없습니다.  

라우터에서 SLB/Mux 및 RAS 게이트웨이 통해 보급 된 VIP 논리 네트워크에 대 한 경로 수신 하 여 SDN 인프라에서 사용 되는 BGP 피어를 생성 해야 합니다. 한 가지 방법은 (SLB/MUX 또는 외부 BGP 피어에 RAS 게이트웨이)에서 발생 해야만 BGP 피어 링 합니다.  그러나 라우팅을의 첫 번째 계층 위에 사용할 수 있으며 고정 경로 OSPF 같은 다른 동적 라우팅 프로토콜 설명한 것 처럼에서는 VIP 논리 네트워크 IP 서브넷 접두사 외부 BGP 피어에 실제 네트워크에서 라우팅 가능 해야 합니다.   

BGP 피어 링은 일반적으로 관리 되는 스위치 또는 라우터는 네트워크 인프라의 일부로 구성 됩니다. BGP 피어만 라우팅 모드에서 설치 된 원격 액세스 서버 (RAS) 역할을 사용 하 여 Windows Server 에서도 구성할 수 있습니다. 네트워크 인프라에서이 BGP 라우터 피어 ASN 자체 있고 SDN 구성 요소에 할당 된 ASN에서 피어 링을 허용 하도록 구성 되어야 합니다 \(SLB/MUX 및 RAS 게이트웨이에\)합니다. 실제 라우터, 또는 해당 라우터의 컨트롤에서 네트워크 관리자에서 다음 정보를 구해야 합니다.

- 라우터 ASN  
- 라우터 IP 주소  
- SDN 구성 요소 (개인 ASN 범위에서 AS 숫자 수 있음) 사용에 대 한 ASN

>[!NOTE]
>SLB/MUX에서 4 바이트 Asn은 사용할 수 없습니다. SLB/MUX와 라우터에 2 바이트 Asn을 할당 해야 wo 연결입니다. 사용자 환경에서 다른 곳에서 4 바이트 Asn을 사용할 수 있습니다.  

사용자 또는 네트워크 관리자는 BGP 라우터 피어 ASN 및 IP 주소 또는 RAS 게이트웨이 및 SLB/Mux 사용 하는 전송 논리 네트워크의 서브넷 주소에서 연결을 허용 하도록 구성 해야 합니다.

자세한 내용은 [프로토콜 BGP (경계 게이트웨이)](../../../remote/remote-access/bgp/Border-Gateway-Protocol-BGP.md)합니다.

## <a name="default-gateways"></a>기본 게이트웨이
물리적 호스트 및 게이트웨이 가상 컴퓨터와 같은 여러 네트워크에 연결 하도록 구성 된 컴퓨터만 하나의 기본 게이트웨이가 구성 되어 있어야 합니다. 인터넷에 연결 하는 데 사용 되는 어댑터에서 기본 게이트웨이 구성 합니다.

Virtual machines에 대 한 기본 게이트웨이로 사용 하는 네트워크를 결정 하기 위해 이러한 규칙을 수행 합니다.

1. 전송 네트워크에 연결 하는 가상 컴퓨터 또는 멀티 호 밍 전송 네트워크나 다른 네트워크에 있으면 기본 게이트웨이로 전송 논리 네트워크를 사용 합니다.
2. 가상 컴퓨터만 관리 네트워크에 연결 하는 경우 기본 게이트웨이로 관리 네트워크를 사용 합니다. 
3. SLB/Mux 및 RAS 게이트웨이에 대 한 HNV 공급자 네트워크를 사용 합니다. 기본 게이트웨이 HNV 공급자 네트워크를 사용 하지 마세요. 
4. Storage1 Storage2, 공용 VIP, 개인 VIP 네트워크에 직접 가상 컴퓨터를 연결 하지 마십시오.

Hyper-v 호스트에 저장소 노드에 기본 게이트웨이로 관리 네트워크를 사용 합니다.  저장소 네트워크를 사용 하는 기본 게이트웨이가 할당 있어서는 안 됩니다.


## <a name="network-hardware"></a>네트워크 하드웨어

다음 섹션에서는 네트워크 하드웨어 배포 계획을 사용할 수 있습니다.

### <a name="network-interface-cards-nics"></a>NIC(네트워크 인터페이스 카드)

Hyper-v 호스트 및 저장소 호스트에서 사용 되는 네트워크 인터페이스 카드 (Nic)에 최상의 성능을 얻으려면 특정 기능이 필요 합니다. 

원격 직접 메모리 액세스 (RDMA) 커널 바이패스 호스트 cpu에서 다른 작업을 수행 하는 CPU를 사용 하지 않고 많은 양의 데이터를 전송할 수 있도록 하는 기술입니다. 

스위치 포함 된 팀 (SET)는 Windows Server 2016에 Hyper-v 및 네트워킹 SDN (소프트웨어) 스택을 포함 하는 환경에서 사용할 수 있는 NIC 팀 솔루션의 대안입니다. 일부 NIC 팀 기능은 Hyper-v 가상 스위치에 통합 하는 설정 합니다. 

자세한 내용은 [원격 직접 메모리 액세스 (RDMA) 및 스위치 포함 된 팀 (SET)](../../../virtualization//hyper-v-virtual-switch/RDMA-and-Switch-Embedded-Teaming.md)합니다.   

테 넌 트 가상 네트워크 트래픽은 NVGRE 또는 VXLAN 캡슐화 헤더에 의해 발생 하는 오버 헤드를 위해서 계층 2 패브릭 네트워크 (스위치 및 호스트)의 MTU 설정 해야에 보다 크거나 같은 1674 바이트 \(계층 2 이더넷을 포함 하 여 헤더\)합니다. 

새 지 Nic *EncapOverhead* 고급 어댑터 키워드 MTU 네트워크 컨트롤러 호스트 에이전트를 통해 자동으로 설정 합니다. 지원 하지 않는 새 Nic *EncapOverhead* 키워드를 사용 하 여 각 실제 호스트에서 MTU 크기를 수동으로 설정 해야 합니다 *JumboPacket* \(또는 그와 동등한\) 키워드. 


### <a name="switches"></a>스위치

물리적 스위치와 사용자 환경에 대 한 라우터를 선택할 때 다음 기능 집합을 지원 하 고 있는지 확인 합니다.  

- Switchport MTU 설정 \(필요\)  
- MTU 설정 > = 1674 바이트 \(L2 이더넷 헤더를 포함 하 여\)  
- L3 프로토콜 \(필요\)  
- ECMP  
- BGP \(IETF RFC 4271\)\-ECMP를 기반 합니다.

구현을 다음 IETF 표준을에서 반드시 문을 지원 해야 합니다.

- RFC 2545: "IPv6 Inter-domain Routing에 대 한 BGP 4 멀티 프로토콜 확장"  
- RFC 4760: "BGP 4에 대 한 멀티 프로토콜 확장"  
- RFC 4893: "4 16-8 진수 AS에 대 한 BGP 지원 공간을 숫자"  
- RFC 4456: "BGP 경로 리플렉션: 대안을 전체 망상 조직 내부 BGP (IBGP) "  
- RFC 4724: "BGP 정상적으로 다시 시작 메커니즘"  

다음 태그 지정 프로토콜은 필수입니다.

- VLAN-다양 한 유형의 트래픽에 대해 격리
- 802.1q 트렁크

다음 항목이 링크 컨트롤을 제공합니다.

- 서비스 품질 \(PFC RoCE를 사용 하는 경우에 필요\)
- 트래픽을 선택 향상 \(802.1Qaz\)
- 우선 순위 기반 흐름 제어 \(802.1 p/Q 및 802.1Qbb\)

다음 항목이 가용성 및 중복성을 제공 합니다.

- 스위치 가용성 (필수)
- 항상 사용 가능한 라우터 게이트웨이 기능을 수행 해야 합니다. 다중 섀시 switch\ 라우터 또는 VRRP 같은 기술을 사용 하 여이 수행할 수 있습니다.

다음 항목 관리 기능을 제공 합니다.

**모니터링**

- SNMP v1 또는 SNMP v2 (물리적 스위치 모니터링에 대 한 네트워크 컨트롤러를 사용 하는 경우 필수)  
- SNMP Mib \(물리적 스위치 모니터링에 대 한 네트워크 컨트롤러를 사용 하는 경우 필요 합니다.\)  
- MIB-II (RFC 1213) LLDP, 인터페이스 MIB \(RFC 2863\), IF MIB, IP MIB, IP 전달 MIB, 질문 및 답변-브리지-MIB, 브리지 MIB, LLDB MIB, 엔터티 MIB, IEEE8023 지연 MIB  

다음 다이어그램은 샘플 4 개 노드 설정을 보여 줍니다. 명확성을 위해 첫 번째 다이어그램은 네트워크 컨트롤러만, 두 번째는 네트워크 컨트롤러 및 소프트웨어 부하 분산 장치를 보여 줍니다. 및 세 번째 다이어그램은 네트워크 컨트롤러, 소프트웨어 부하 분산 장치 및 게이트웨이 보여 줍니다.  

이러한 다이어그램에는 저장소 네트워크 및 Vnic 표시 되지 않습니다. SMB 기반 저장소를 사용 하려는 경우 이러한 작업은 필요 합니다.

모든 물리적 계산 호스트 (올바른 논리 네트워크에 대 한 올바른 네트워크로 연결 되어 있는 것으로 가정)에서 인프라 및 테 넌 트 가상 컴퓨터를 재배포할 수 있습니다.

## <a name="extending-your-infrastructure"></a>인프라를 확장합니다.  
인프라의 크기 조정 및 리소스 요구 사항이 호스트 하려는 테 넌 트 워크 로드 가상 컴퓨터에 종속 됩니다. CPU, 메모리 및 인프라 virtual machines 디스크 요구 사항 (예: 네트워크 컨트롤러, SLB, 게이트웨이 등) 이전 표에 나열 되어 있습니다. 필요에 따라 확장 하기 위해 이러한 인프라 가상 컴퓨터를 더 추가할 수 있습니다. 그러나 Hyper-v 호스트에서 실행 중인 모든 테 넌 트 가상 컴퓨터 자체 CPU, 메모리 및 디스크 요구 사항을 고려해 야 할 수 있습니다.   

테 넌 트 워크 로드 가상 컴퓨터는 실제 Hyper-v 호스트에서 너무 많은 리소스를 소비 하기 시작, 물리적 호스트를 추가 하 여 인프라를 확장할 수 있습니다. Virtual Machine Manager 사용 하 여 또는 PowerShell 스크립트 (배포한 방법에 따라 처음에 인프라)를 사용 하 여이 수행할 수 있습니다 네트워크 컨트롤러를 통해 새 서버 리소스를 만듭니다. HNV 공급자 네트워크에 대 한 추가 IP 주소를 추가 해야 할 경우 (해당 IP 풀)과 호스트를 사용할 수 있는 새 논리 서브넷을 만들 수 있습니다.
