# <a name="plan-nps-as-a-radius-server"></a>RADIUS 서버로 작동하는 NPS 계획

>적용 대상: Windows Server (반기 채널), Windows Server 2016

네트워크 정책 서버를 배포 하는 경우 \(NPS\) 전화 접속 사용자 서비스 RADIUS (Remote Authentication) 서버로 NPS 인증, 권한 부여 및 계정에 대해 수행 도메인 및 로컬 도메인에 대 한 연결 요청 로컬 도메인을 신뢰 하는 합니다. RADIUS 배포를 간소화 하기 위해 다음 계획 지침을 사용할 수 있습니다.

다음 계획 지침에서 RADIUS 프록시로 NPS를 배포 하려는 경우를 포함 하지 않습니다. RADIUS 프록시로 NPS를 배포 하면 NPS는 원격 도메인 이나 트러스트 되지 않은 도메인에서 NPS 또는 다른 RADIUS 서버를 실행 하는 서버에 연결 요청을 전달 합니다. 

NPS를 RADIUS 서버로 네트워크에 배포 하기 전에 배포를 계획 하려면 다음 지침을 따르세요.

- NPS 구성을 계획 합니다.

- RADIUS 클라이언트를 계획 합니다.

- 인증 방법의 사용을 계획 합니다.

- 네트워크 정책을 계획 합니다.

- NPS 계정 관리를 계획 합니다.

## <a name="plan-nps-configuration"></a>NPS 구성 계획

도메인에서 NPS가 멤버인 결정 해야 합니다. 다중 도메인 환경에 대 한 NPS의 멤버 이기는 도메인의 사용자 계정 및 로컬 도메인을 NPS 신뢰 하는 모든 도메인에 대 한 자격 증명을 인증할 수 있습니다. 권한 부여 프로세스 중 사용자 계정의 전화 접속 속성 읽기를 NPS를 허용 하려면 각 도메인에 대 한 RAS 및 NPSs 그룹 NPS의 컴퓨터 계정을 추가 해야 합니다.

NPS의 도메인 멤버 자격을 확인 한 후 서버는 RADIUS 프로토콜을 사용 하 여 네트워크 액세스 서버 라고도 하는 RADIUS 클라이언트와 통신 하도록 구성 되어야 합니다. 또한 구성할 수 있습니다. 있습니다 이벤트 유형을 NPS에서 기록 하는 이벤트 로그에 서버에 대 한 설명을 입력할 수 있습니다.

### <a name="key-steps"></a>주요 단계

NPS 구성에 대 한 계획 하는 동안 다음 단계를 사용할 수 있습니다.

- NPS RADIUS 클라이언트에서 RADIUS 메시지를 수신 하는 데 사용 하는 RADIUS 포트를 결정 합니다. 기본 포트는 UDP 포트 1812 및 RADIUS 인증 메시지에 대 한 1645 및 포트 1813 및 1646에서 RADIUS 계정 메시지입니다.

- NPS를 여러 네트워크 어댑터를 사용 하 여 구성 된 경우 RADIUS 트래픽을 허용 해야 하는 원하는 어댑터를 확인 합니다.

- 이벤트 로그에 기록 하기 위해 NPS를 원하는 이벤트 유형을 결정 합니다. 거부 된 인증 요청, 성공적으로 인증 요청 또는 두 가지 유형의 요청을 기록할 수 있습니다.

- 둘 이상의 NPS를 배포 하는 여부를 결정 합니다. RADIUS 기반 인증 및 계정에 대 한 내결함성을 제공 하려면 두 개 이상의 NPSs를 사용 합니다. NPS 주 RADIUS 서버로 사용 됩니다 하 고 다른 백업으로 사용 됩니다. 각 RADIUS 클라이언트는 다음 두 NPSs에서 구성 됩니다. 기본 NPS를 사용할 수 없는 경우 RADIUS 클라이언트는 다음 액세스 요청 메시지 대체 NPS를 보냅니다.

- 관리 오버 헤드에 저장 하 고 서버의 잘못 된 구성 하지 않으려면 다른 NPSs에 NPS 구성 중 하나를 복사 하는 데 스크립트를 계획 합니다. NPS는 다른 NPS에 가져오기에 대 한 NPS 구성의 일부나 전부를 복사할 수 있도록 하는 Netsh 명령을 제공 합니다. 명령 프롬프트에서 Netsh 수동으로 실행할 수 있습니다. 그러나 스크립트에 명령 시퀀스를 저장 하면 서버 구성에 변경 하려는 경우 나중에 스크립트를 실행할 수 있습니다.

## <a name="plan-radius-clients"></a>RADIUS 클라이언트를 계획 합니다.

RADIUS 클라이언트는 무선 액세스 지점과 같은 네트워크 액세스 서버를 가상 사설망 (VPN) 서버에 802.1 X 가능 스위치 및 전화 접속 서버. 연결 RADIUS 서버에 요청 메시지를 전달 하는 RADIUS 프록시를 RADIUS 클라이언트 이기도 합니다. NPS는 모든 네트워크 액세스 서버를 지원 하 고 RADIUS 프록시는 RADIUS를 준수 하는 RFC 2865 "원격 인증 전화 접속 사용자 서비스 (RADIUS)"에 설명 된 대로 프로토콜 및 RFC 2866 "RADIUS 계정입니다."

>[!IMPORTANT]
>클라이언트 컴퓨터와 같은 액세스 클라이언트는 RADIUS 클라이언트가 아닙니다. 네트워크 액세스 서버 및 RADIUS 프로토콜을 지 원하는 프록시 서버는 RADIUS 클라이언트입니다.

또한 무선 액세스 지점 및 스위치 802.1 X 인증을 수행할 수 여야 합니다. Extensible Authentication Protocol을 배포 하려는 경우 \(EAP\) 또는 Protected Extensible Authentication Protocol \(PEAP\), 액세스 지점 및 스위치 EAP의 사용을 지원 해야 합니다.

무선 액세스 지점에 대 한 PPP 연결에 대 한 기본 상호 운용성을 테스트 하려면 액세스 지점 및 인증 프로토콜이 PAP (암호)를 사용 하는 액세스 클라이언트를 구성 합니다. PEAP 등의 추가 PPP 기반 인증 프로토콜을를 사용 하 여 네트워크 액세스에 사용 하려는 것을 테스트 합니다.

### <a name="key-steps"></a>주요 단계

RADIUS 클라이언트에 대 한 계획 하는 동안 다음 단계를 사용할 수 있습니다.

- 문서 공급 업체별 특성 (Vsa) NPS에 구성 해야 합니다. 네트워크 액세스 서버가 Vsa가 필요 하는 경우 NPS에서 네트워크 정책을 구성할 때 나중에 사용할 VSA 정보를 기록 합니다. 

- RADIUS 클라이언트와에 NPS 모든 장치의 구성을 단순화 하기 위해 IP 주소를 문서화 합니다. RADIUS 클라이언트를 배포할 때 인증 서버와 입력 한 NPS IP 주소를 사용 하 여 RADIUS 프로토콜을 사용 하도록 구성 해야 합니다. 및 NPS 스냅인에 RADIUS 클라이언트 IP 주소를 입력 해야 RADIUS 클라이언트와 통신 하는 NPS를 구성한 경우.

- NPS 스냅인에서 RADIUS 클라이언트에 구성에 대 한 공유 암호를 만듭니다. 공유 암호 또는 암호와 NPS에서 RADIUS 클라이언트를 구성 하는 동안 NPS 스냅인에 입력 하는 RADIUS 클라이언트를 구성 해야 합니다.

## <a name="plan-the-use-of-authentication-methods"></a>인증 방법의 사용 계획

NPS에는 모두 암호 및 인증서 기반 인증 방법을 지원합니다. 그러나 모든 네트워크 액세스 서버와 동일한 인증 방법을 지원합니다. 경우에 따라 네트워크 액세스의 유형에 따라 다른 인증 방법을 배포 하는 것이 좋습니다.

예를 들어, 다음 무선와 조직에 대 한 VPN 액세스를 배포 하지만 각 유형의 액세스에 대 한 다른 인증 방법을 사용 하는 것이 좋습니다. EAP with Transport Layer Security (EAP-TLS)이 제공 하는 강력한 보안으로 인해 VPN 연결 및 802.1x 무선 연결용 PEAP-2(MS-CHAP v2에 대 한 EAP-TLS 합니다.

PEAP 빠른 이라는 기능을 제공 하는 버전 2 (PEAP-2(MS-CHAP v2) 하는 Microsoft Challenge Handshake 인증 프로토콜을 사용 하 여 휴대용 컴퓨터 및 기타 무선 장치를 사용 하 여 사용 하기 위해 특별히 설계 된를 다시 연결 합니다. 빠른 무선 클라이언트가 새 액세스 지점에 연결할 때마다 다시 인증 받이 필요 없이 동일한 네트워크에 무선 액세스 지점 사이 이동 하려면 다시 연결 합니다. 이 무선 사용자에 대 한 더 나은 환경을 제공 하 고 자격 증명을 다시 입력 하지 않고도 액세스 지점 사이 이동할 수 있습니다.
인해 빠른 다시 연결 하 고 PEAP-2(MS-CHAP v2 보안 PEAP-2(MS-CHAP v2에서 제공 하는 무선 연결에 대 한 인증 방법으로 논리적으로 선택 됩니다.

VPN 연결에 대 한 EAP-TLS는 가정 이나 모바일 컴퓨터에서 인터넷 조직 VPN 서버에 전송 될 때에 네트워크 트래픽을 보호 하는 강력한 보안을 제공 하는 인증서 기반 인증 메서드입니다.

### <a name="certificate-based-authentication-methods"></a>인증서 기반 인증 방법

인증서 기반 인증 방법이 있는 강력한 보안을 제공 합니다. 있고 단점이 중인 암호 기반 인증 방법을 보다 배포 하기가 더 어렵습니다.

PEAP-MS-CHAP v2와 EAP-TLS는 인증서 기반 인증 방법도 서 배포 되는 방식으로 사이의 여러 차이점이 있습니다.

### <a name="eap-tls"></a>EAP-TLS

EAP-TLS 클라이언트 및 서버 인증 인증서를 사용 하 여 및 조직에서 공개 키 인프라 (PKI)를 배포 해야 합니다. 복잡할 수 있습니다 하 고 NPS RADIUS 서버로 사용 하기 위한 계획의 독립 된 계획 단계를 필요한 PKI를 배포 합니다.

EAP-TLS를 사용 하 여 NPS 인증 기관에서 서버 인증서를 등록 \(CA\), 인증서를 인증서 저장소에 로컬 컴퓨터에 저장 됩니다. 서버 인증에는 인증 프로세스 중 NPS 액세스 클라이언트에 해당 id를 증명 하기 위해 access 클라이언트를 해당 서버 인증서를 보내면 발생 합니다. Access 클라이언트 인증서 유효 및 서버 인증 하는 동안 사용 하기에 적합 여부를 결정 하는 다양 한 인증서 속성을 검사 합니다. 서버 인증서의 최소 서버 인증서 요구 사항을 충족 하 고 액세스 클라이언트 신뢰 하는 CA에서 발급 한 경우 NPS 클라이언트에서 성공적으로 인증 됩니다.

마찬가지로, 클라이언트 인증 클라이언트 NPS로 id를 증명 하기 위해 NPS를 해당 클라이언트 인증서를 보내면 인증 프로세스 중 발생 합니다. NPS는 인증서를 검사 하 고 access 클라이언트는 NPS에서 성공적으로 인증 하 고 클라이언트 인증서를 최소 클라이언트 인증서 요구 사항을 충족 하 고 NPS 신뢰 하는 CA에서 발급 한 경우.

이지만 서버 인증서를 NPS에서 인증서 저장소에 저장 되도록 하는 데 필요한 클라이언트 또는 사용자 인증서에에서 저장할 수 스마트 카드 또는 클라이언트에서 인증서 저장소의 중 하나입니다.

성공 하려면이 인증 프로세스에 대 한 것이 필수는 모든 컴퓨터는 로컬 컴퓨터와 현재 사용자에 대 한 신뢰할 수 있는 루트 인증 기관 인증서 저장소에 있는지 조직의 CA 인증서입니다.

### <a name="peap-ms-chap-v2"></a>PEAP-MS-CHAP v2

PEAP-2(MS-CHAP v2 서버 인증과 사용자 인증에 대 한 암호 기반 자격 증명에 대 한 인증서를 사용합니다. 인증서는 서버 인증에 대해서만 사용 되므로, PEAP-2(MS-CHAP v2를 사용 하려면 PKI 배포에 필요 하지 않습니다. PEAP-2(MS-CHAP v2를 배포할 때에 다음 두 가지 방법 중 하나에 NPS에 대 한 서버 인증서를 가져올 수 있습니다.

- NPSs에 Active Directory 인증서 서비스 (AD CS) 한 다음 자동 등록 인증서를 설치할 수 있습니다. 이 메서드를 사용 하는 경우 NPS로 발급 된 인증서를 신뢰할 수 있도록 네트워크에 연결 하는 클라이언트 컴퓨터에 CA 인증서도 등록 해야 합니다.

- VeriSign과 같은 공용 CA에서 서버 인증서를 구입할 수 있습니다. 이 메서드를 사용 하는 경우 클라이언트 컴퓨터에서 이미 신뢰할 수 있는 CA를 선택 해야 합니다. 클라이언트 컴퓨터는 CA를 신뢰 하는지 여부를 결정할 클라이언트 컴퓨터에 인증서 Microsoft Management Console (MMC) 스냅인 열고 로컬 컴퓨터 및 현재 사용자에 대해 신뢰할 수 있는 루트 인증 기관 저장소를 확인 합니다. 이러한 인증서 저장소에 CA에서 발급 한 인증서 인 경우 클라이언트 컴퓨터는 CA를 신뢰 하며는 CA에서 발급 된 모든 인증서를 신뢰 하므로

서버 인증에는 PEAP-2(MS-CHAP v2 인증 프로세스 중 NPS가 클라이언트 컴퓨터에 해당 서버 인증서를 보내면 발생 합니다. Access 클라이언트 인증서 유효 및 서버 인증 하는 동안 사용 하기에 적합 여부를 결정 하는 다양 한 인증서 속성을 검사 합니다. 서버 인증서의 최소 서버 인증서 요구 사항을 충족 하 고 액세스 클라이언트 신뢰 하는 CA에서 발급 한 경우 NPS 클라이언트에서 성공적으로 인증 됩니다.

사용자 인증의 네트워크 형식 암호 기반 자격 증명에 연결 하려고 하는 사용자 나타나고 로그온 하려고 시도 합니다. NPS는 자격 증명을 수신 하 고 인증 및 권한 부여를 수행 합니다. 사용자는 인증 하 고 성공적으로 권한이 부여 되 고 클라이언트 컴퓨터 NPS 인증 되 면 연결 요청이 허용 됩니다.

### <a name="key-steps"></a>주요 단계

인증 방법의 사용에 대 한 계획 하는 동안 다음 단계를 사용할 수 있습니다.

- 같은 무선, VPN, 802.1 X 호환 스위치를 제공 하려면 네트워크 액세스 및 전화 접속 액세스의 형식을 식별 합니다.

- 인증 방법에는 각 유형의 액세스에 대해 사용 하려는 메서드를 결정 합니다. 강력한 보안을 제공 하는 인증서 기반 인증 방법을 사용 하는 것이 좋습니다. 그러나 아닐 다른 인증 방법의 네트워크에 필요한 것 보다 나은 균형을 제공할 수 있습니다, PKI를 배포할 수 있습니다.

- EAP-TLS를 배포 하는 경우 PKI 배포를 계획 합니다. 여기에 인증서 템플릿에 서버 인증서 및 클라이언트 컴퓨터 인증서를 사용 하는 것을 계획 합니다. 또한 도메인 구성원 및 비-도메인 구성원 컴퓨터, 스마트 카드를 사용할 것인지 여부를 결정 하는 인증서를 등록 하는 방법을 결정도 포함 됩니다.

- PEAP-2(MS-CHAP v2를 배포 하는 경우에 NPSs 또는 VeriSign과 같은 공용 CA에서 서버 인증서를 구입할 것인지에 서버 인증서를 발급 하도록 AD CS를 설치 하려는 지 여부를 결정 합니다.

### <a name="plan-network-policies"></a>네트워크 정책 계획

네트워크 정책은 NPS에서 RADIUS 클라이언트에서 받은 연결 요청 된 권한이 있는지 확인 사용 됩니다. 또한 NPS는 권한 부여 결정을 사용자 계정 전화 접속 속성을 사용 합니다.

네트워크 정책은 NPS 스냅인에서 나타나는 순서 대로 처리 되므로, 정책 목록에서 먼저 가장 제한적인 정책을 배치 하려고 합니다. 각 연결 요청에 대 한 NPS 연결 요청 속성을 사용 하 여 정책의 조건과 일치 하도록 시도 합니다. NPS는 일치 항목을 찾을 때까지 순서 대로 각 네트워크 정책을 검사 합니다. 일치 항목을 찾지 못하면, 연결 요청이 거부 됩니다.

### <a name="key-steps"></a>주요 단계

네트워크 정책에 대 한 계획 하는 동안 다음 단계를 사용할 수 있습니다.

- 네트워크 정책에 가장 낮은 가장 제한적인에서 기본 NPS 처리 순서를 결정 합니다.

- 정책 상태를 확인 합니다. 정책 상태를 사용 하거나 사용 하지 않도록 설정의 값을 가질 수 합니다. 정책을 사용 하는 경우 NPS 정책 권한 부여를 수행 하는 동안 평가 합니다. 정책을 사용 하지 않는 경우 평가 되지 않습니다.

- 정책 유형을 결정 합니다. 정책 연결 요청 정책 연결 요청 정책의 조건과 일치 하는 경우 액세스를 거부 하도록 고안 된 여부, 또는 정책의 조건과 일치 하는 경우 액세스 권한을 부여할 기능이 있는지 여부를 결정 합니다. 예를 들어, 무선 액세스는 Windows 그룹의 멤버를 명시적으로 거부 하려는 경우 그룹, 무선 연결 메서드를 갖는 액세스 거부의 설정을 입력 하는 정책을 지정 하는 네트워크 정책을 만들면 됩니다.

- 정책을 기준으로 하는 그룹의 구성원 인 사용자 계정 전화 접속 속성을 무시 하도록 NPS 여부를 결정 합니다. 이 설정을 사용 하지 않으면, 사용자 계정 전화 접속 속성 네트워크 정책에서 구성한 설정을 재정의 합니다. 예를 들어, 사용자에 게 액세스 권한을 부여 하는 네트워크 정책이 구성 된 경우 해당 사용자에 대 한 사용자 계정의 전화 접속 속성 액세스를 거부로 설정 됩니다, 사용자는 액세스가 거부 됩니다. 그러나 정책 유형 무시 사용자 계정 전화에 대 한 설정 속성을 사용 하도록 설정 하는 경우 동일한 사용자 네트워크에 액세스 권한이 부여 합니다.

- 정책 정책 원본 설정을 사용 하는지 여부를 결정 합니다. 이 설정을 사용 하면 모든 액세스 요청에 대 한 소스를 쉽게 지정할 수 있습니다. 가능한 소스는 터미널 서비스 게이트웨이 (TS 게이트웨이), (VPN 또는 전화 접속) 원격 액세스 서버, DHCP 서버, 무선 액세스 지점 및 상태 등록 기관 서버는 합니다. 또는 공급 업체별 소스를 지정할 수 있습니다.

- 네트워크 정책 적용 하기 위해에서 일치 해야 하는 조건을 결정 합니다.

- 연결 요청을 통해 네트워크 정책의 조건과 일치 하는 경우 적용 되는 설정을 확인 합니다.

- 사용, 수정 또는 기본 네트워크 정책을 삭제할 것인지 여부를 결정 합니다.

## <a name="plan-nps-accounting"></a>NPS 계정 계획

NPS는 세 가지 형식으로 사용자 인증 및 계정 요청 등의 RADIUS 계정 데이터를 기록 하는 기능을 제공 합니다. IAS 형식, 데이터베이스 호환 형식 및 Microsoft SQL Server 로깅 

IAS 형식 및 데이터베이스와 호환 가능한 형식의 텍스트 파일 형식으로 로컬 NPS에서 로그 파일을 만듭니다. 

SQL Server 로깅 SQL Server 2000 또는 SQL Server 2005 XML 규격 데이터베이스, 관계형 데이터베이스에 대 한 로깅의 장점을 활용 하 여 RADIUS 계정을 확장에 로그인 하는 기능을 제공 합니다.

### <a name="key-steps"></a>주요 단계

NPS 계정에 대 한 계획 하는 동안 다음 단계를 사용할 수 있습니다.

- SQL Server 데이터베이스 또는 로그 파일에서 NPS 계정 데이터를 저장할 것인지 여부를 결정 합니다.

### <a name="nps-accounting-using-local-log-files"></a>로컬 로그 파일을 사용 하 여 NPS 계정

연결 분석 및 청구 목적에 주로 사용 되 고 후 악의적인 사용자의 활동을 추적 하는 방법을 제공 하는 보안 조사 도구로 서도 유용 사용자 인증을 기록 하 고 계정 로그 파일에 대 한 요청을 공격입니다.

### <a name="key-steps"></a>주요 단계

로컬 로그 파일을 사용 하 여 NPS 계정에 대 한 계획 하는 동안 다음 단계를 사용할 수 있습니다.

- NPS 로그 파일에 대해 사용 하려는 텍스트 파일 형식을 결정 합니다.

- 로그인 하려는 정보의 유형을 선택 합니다. 계정 요청, 인증 요청 및 정기적으로 상태를 기록할 수 있습니다.

- 로그 파일을 저장 하려는 하드 디스크 위치를 결정 합니다.

- 로그 파일 백업 솔루션을 디자인 합니다. 하드 디스크 위치에 로그 파일이 저장 된 데이터를 쉽게 백업할 수 있는 위치에 있어야 합니다. 또한 하드 디스크 위치를 로그 파일을 저장할 폴더에 대 한 액세스 제어 목록 (ACL)를 구성 하 여 보호 되어야 합니다.

- 만들려는 새 로그 파일 하려는 빈도 결정 합니다. 파일 크기에 따라 만들 로그 파일을 원한다 면 NPS를 통해 새 로그 파일이 생성 되기 전에 허용 되는 최대 파일 크기를 결정 합니다.

- 하드 디스크 공간이 부족 한 경우 이전 로그 파일을 삭제 하려면 NPS 여부를 결정 합니다.

- 응용 프로그램 또는 계정 데이터 보기 및 보고서를 생성 하는 데 원하는 응용 프로그램을 확인 합니다.

### <a name="nps-sql-server-logging"></a>NPS SQL Server 로깅

NPS SQL Server 로깅 세션 상태 정보를 보고서 만들기 및 데이터 분석을 위해 및 중앙 집중화 하 고 관리 계정 데이터를 간소화 해야 하는 경우에 사용 됩니다.

로그인 레코드 사용자 인증 하 고 Microsoft SQL Server Desktop Engine을 실행 하는 컴퓨터에서 데이터 원본에 하나 이상의 네트워크 액세스 서버에서 받은 요청을 계정에 SQL Server를 사용 하는 기능을 제공 하는 NPS \(MSDE2000\), 또는 SQL Server 2000 이상 SQL Server의 모든 버전.

계정 데이터는 XML 형식으로 NPS에서 모두 구조적된 쿼리 언어를 지 원하는 데이터베이스에 저장된 프로시저에 전달 됩니다 \(SQL\) 및 XML \(SQLXML\)합니다. 사용자 인증을 기록 하 고 XML 규격 SQL Server 데이터베이스에서 요청을 계정에 하나의 데이터 원본이 여러 NPSs 수 있습니다.

### <a name="key-steps"></a>주요 단계

계획 하는 동안 NPS 계정에 대 한 NPS SQL 서버 로깅을 사용 하 여, 다음 단계를 사용할 수 있습니다.

- 사용자 또는 조직의 다른 구성원이 SQL Server 2000 또는 SQL Server 2005 관계형 데이터베이스 개발 환경을 만들고, 수정 하 고, 관리 하 고 SQL Server 데이터베이스를 관리 하려면 이러한 제품을 사용 하는 방법을 알고 있는지 확인 합니다.

- NPS 또는 원격 컴퓨터에서 SQL Server에 설치 되어 있는지 확인 합니다.

- NPS 계정 데이터를 포함 하는 들어오는 XML 파일을 처리 하는 데 사용할 SQL Server 데이터베이스에서 저장된 프로시저를 디자인 합니다.

- SQL Server 데이터베이스 복제 구조 및 흐름을 디자인 합니다.

- 응용 프로그램 또는 계정 데이터 보기 및 보고서를 생성 하는 데 원하는 응용 프로그램을 확인 합니다.

- 모든 계정 요청에서 클래스 특성을 전송 하는 네트워크 액세스 서버를 사용 하려고 합니다. 클래스 특성 액세스 허용 메시지를 RADIUS 클라이언트로 보내고 인증 세션을 사용 하 여 계정 요청 메시지를 상호 연결 하는 데 유용 합니다. 클래스 특성은 계정 요청 메시지의 네트워크 액세스 서버에 의해 전달 되 면 계정 및 인증 레코드에 맞게 사용할 수 있습니다. 서비스-재부팅-시간 및 서버 주소 특성 Unique-일련 번호를 조합에는 서버에서 허용 하는 각 인증에 대 한 고유 하 게 식별 해야 합니다.

- 중간 계정을 지 원하는 네트워크 액세스 서버를 사용 하려고 합니다.

- 계정 설정 및 계정 끄기 메시지를 보내는 네트워크 액세스 서버를 사용 하려고 합니다.

- 저장 하 여 지 원하는 네트워크 액세스 서버 및 계정 데이터 전달을 사용 하도록 계획 합니다. 네트워크 액세스 서버를 NPS와 통신할 수 없는 경우이 기능을 지 원하는 네트워크 액세스 서버에서 계정 데이터를 저장할 수 있습니다. NPS를 사용할 수 있는 네트워크 액세스 서버를 전달 저장 된 레코드를 NPS로이 기능을 제공 하지 않는 네트워크 액세스 서버를 통해 계정에서 향상 된 안정성을 제공 합니다.

- 항상 네트워크 정책에서 Acct-중간-간격 특성을 구성 하려고 합니다. Acct-중간-간격 특성 사이의 각 중간 네트워크 액세스 서버를 전송 하는 초 단위로 간격을 설정 합니다. 한번에 따라 Acct-중간-간격 특성의 값 60 초 또는 1 분 미만의 아니어야 하며 600 이며 10 분 보다 작아서는 안 됩니다. 자세한 내용은 RFC 2869, "RADIUS 확장입니다."을 참조 하세요.

- 정기적으로 상태는 로깅이 NPSs 프로그램에서 사용 되는지 확인 합니다.
