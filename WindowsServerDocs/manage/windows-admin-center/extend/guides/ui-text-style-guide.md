# <a name="windows-admin-center-ui-text-and-design-style-guide"></a>Windows Admin Center UI 텍스트와 디자인 스타일 가이드

>적용 대상: Windows Admin Center

이 항목에서는 Windows Admin Center용 사용자 인터페이스(UI) 텍스트를 작성하는 일반적인 방법 및 일부 특정 규칙과 접근 방법에 대해 설명합니다.

환경이 쉽게 사용할 수 있고 친숙하도록 Windows Admin Center 및 확장은 [Microsoft의 음성 원칙](https://docs.microsoft.com/style-guide/brand-voice-above-all-simple-human)을 따라야 합니다. 이 스타일 가이드는 이러한 음성 원칙과 [Microsoft 쓰기 스타일 가이드](https://docs.microsoft.com/style-guide/welcome/)를 기반으로 하므로, [접근성](https://docs.microsoft.com/style-guide/accessibility/accessibility-guidelines-requirements), [머리글자어](https://docs.microsoft.com/style-guide/acronyms) 및 [청유](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/p/please) 및 [사과](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/s/sorry)와 같은 [단어 선택](https://docs.microsoft.com/style-guide/word-choice/) 등의 정보에 대한 리소스를 모두 확인하도록 하십시오.

## <a name="buttons"></a>단추

- 버튼은 가능한 경우, 특히 도구를 지역화할 계획이 있는 경우 항상 한 단어여야 합니다. 두세 확인 되었지만 더 이상 사용 하지 마세요. 네 개의 단어 있는 경우 더 좋을 수도 링크 컨트롤을 사용 하도록 합니다.
- 버튼 레이블은 간결하고 구체적이며 쉽게 이해할 수 있어야 합니다. 일반적인 "제출" 버튼 대신 "만들기", "삭제", "추가", "포맷" 등 사용자 작업에 해당하는 동사를 사용합니다.
- 질문에 버튼이 따르는 경우 해당 레이블은 질문(일반적으로 "예" 또는 "아니요")에 명확하게 일치해야 합니다.

## <a name="capitalization"></a>대문자 표시

Microsoft의 [대문자 표시](https://docs.microsoft.com/style-guide/capitalization) 스타일을 따릅니다. 거의 모든 작업에 문장형 대문자 표시를 사용합니다.

| UI 요소              |대문자 표시|주석|
|-------------------------|--------------|--------|
|배지(예: 미리 보기) |모두 대문자      ||
|모든 기타          |문장 스타일|그러나 제어 범위 밖에 있는 WMI 또는 PowerShell의 개체 속성을 표시하는 몇 가지 예외가 있습니다.|

## <a name="colons"></a>콜론

콜론을 사용 하 여 목록 소개 합니다. 예를 들어 다음과 같은 가치를 제공해야 합니다.

    Choose one of the following:
    Cats
    Dogs
    Quokkas

줄에서 가장이 레이블 또는 레이블을 지정 명확히 구분 된 레이블 및 가장이 있으면 다른 레이블이 되 면 UI 텍스트에 콜론을 사용 하지 마세요.

레이블이 고 두 요소를 함께 실행 되지 않도록 하려면 레이블 텍스트와 동일한 줄 켜져 있을 때 UI 텍스트에 콜론을 사용 합니다.

## <a name="confirmation-messages"></a>확인 메시지

확인 대화 상자는 계속 있을 경우에 유용 데이터 손실 등의 예기치 않은 결과입니다. 특히 바꿀 수 없는 이벤트에 대 한 명확한 결과 사용 하 여 검색할 수 있게 되었습니다, 유용한 정보를 포함 해야 합니다. 

- 확인 메시지가 필요한 지 확인 합니다. 제품에 새 정보가 없는 경우 (예를 들어, "하 시겠습니까?") 확인 메시지를 필요한 수 있습니다.  
- 고객 작업을 진행 하기 원한다는 것을 확인 합니다.
- 기본 명령 (제목) 및 설명 텍스트 (본문)를 중복 되지 있는지 확인 합니다.
- 머리글에서 질문 또는 다음 발생 하는 결과 대 한 가능한 결과 정의 합니다. 예를 들어, "Erase이 드라이브의 모든 데이터가 있습니까? 또는 "하려는 모든 데이터 지우기"입니다.
- 본문에 세부 정보를 추가 합니다. 변경에 대 하는 항목의 이름과 같은 변수를 여기서 포함 됩니다.
- 간단한 질문 (또는 헤더의 본문에서)는 두 개의 작업 단추 간에 명확한 선택을 프레임을 포함 합니다.
- 선택 하는 복잡 한 사용 예/아니요 단추 신중 하 게 읽는 것이 좋습니다. 와 같은 간단한 선택 하는 작업에 관련 된 단추 사용에 대 한 모든를 삭제 하거나 취소 합니다.

## <a name="first-run-experiences"></a>첫 번째 실행 환경

사용자가 페이지를 처음으로 방문하면 도구를 사용해 볼 수 있는 기회가 있습니다. 다음과 같을 수 있습니다.

- 시작하는 방법을 간단하게 설명하는 빈 페이지의 텍스트 스트링. 예: "'추가'를 선택하여 앱을 선택합니다."
- 사용자가 시작할 수 있는 컨트롤에 대한 링크. 예: "앱을 추가하여 시작합니다."
- 사용자에게 시작하는 방법을 보여주는 소규모 및 간단한 애니메이션 또는 비디오

다음은 Windows 스타일 가이드의 유용한 팁입니다.

### <a name="1-be-helpful"></a>1. 유용성

- 마케팅 스타일 및 언어를 피합니다.
- 무엇을 설명하거나 제안하는 경우 최종 결과가 명확하도록 합니다. 단지 고객에게 특정 작업을 수행하는 방법을 보여주는 것은 그 이유를 모르는 경우 유효하지 않습니다.
- 고객이 이를 필요로 하지 않는 경우 팁을 제시하지 마십시오.

### <a name="2-show-dont-tell"></a>2. 설명 하지

텍스트를 최대한 단순하게 유지합니다(작은 애니메이션 또는 비디오).

### <a name="3-dont-overwhelm"></a>3. 과부하가 안 함

- 시스템 알림 및 셸 알림을 포함하여 팝업 및 팁을 사용 세션당 4개로 제한합니다.
- 팝업의 타이밍이 도움이 되는지 확인합니다.
- 고객의 작업 수행을 방해하지 마십시오.
- 팝업을 쉽게 닫을 수 있는지 확인하십시오.

### <a name="4-keep-it-contextual"></a>4. 컨텍스트 유지

- 고객을 가르치는 것은 적절한 시기에 제공하는 경우 효과적입니다.
- 자습서 또는 슬라이드쇼를 만드는 경우 정보를 구체적으로 만듭니다.
- 마케팅적인 "포장"보다 구체적인 팁과 유용한 정보에 중점을 둡니다.
- 관련된 경우 고객이 나중에 자습서로 돌아올 수 있는 경로를 제공합니다. (사람들은 종종 처음에는 정보를 기억하지 않지만 설정 지침이 관련이 있을 수 있습니다.)
- 빈 상태 메시징은 학습 및/또는 즐거움에 자연스러운 위치입니다. 메시지를 단순하고 유용하게 만듭니다.

### <a name="5-minimize-painful-setup"></a>5. 번거로운 설치를 최소화 합니다.

고객이 전체 가치를 경험하게 하기 위해 다른 작업을 수행해야 하는 경우(온라인 서비스에 로그인 등), 가능한 한 번거롭지 않게 만듭니다.

- 메시지는 짧고 직접적이어야 합니다.
- 논점을 흐리지 마십시오. 가능하면 현재 위치에서 연결할 수 있는 수단을 제공합니다.
- 가능하면 나중에 작업을 수행하는 옵션을 허용하고 나중에 수행하도록 알립니다.
- 고객의 환경에서 벗어나게 하고자 하는 경우 빠르고 쉽게 돌아가는 방법을 제공합니다.

## <a name="help-links"></a>도움말 링크

다음은 Windows 스타일 가이드의 유용한 팁입니다.

### <a name="when-should-we-provide-a-help-link"></a>도움말 링크를 제공 해야 하는 경우는에서는?

거의 하지 않습니다. 도움말 링크를 제공 합니다. 경우에만:

- 분명 하 고 중요 한 질문에 고객에 게는 답 UI 작업에 성공 하는 데 도움이 됩니다 UI에 있는 동안 있을 수 있습니다. 
- 사용자가 UI 작업에 성공 하는 데 필요한 정보를 제공 하기 위해 UI에 충분 한 공간이 없습니다. 

### <a name="where-should-help-links-appear"></a>여기서 표시 하는 링크는 데 도움이 됩니다. 

- 링크 텍스트는 가능한 도움말에 전송 되는 UI 요소에 가깝게 표시 됩니다. 
- 전체 UI 화면에 적용 되는 텍스트 링크를 제공 해야 하는 경우 화면의 왼쪽 아래에 배치 합니다. 
- 도움말 단추 (?)를 통해 링크를 제공 하는 경우 도구 설명 "Help" 여야 합니다.

### <a name="what-url-should-we-use"></a>어떤 URL에서는 사용 해야 합니까?

웹 주소에 직접 연결 되지 않습니다-대신 리디렉션 서비스를 사용 합니다.

Microsoft 개발자에 게 사용자를 수동으로 입력을 사용 하는 경우는 aka.ms 링크 (으로 URL의 대상이 자동으로 Docs.microsoft.com 등의 브라우저 로캘을 인식 하는 웹 사이트) 있을 수 있음을 도움말 링크를 하는 경우는 FWLink를 사용 해야

### <a name="text-guidelines"></a>텍스트 지침 

- 전체 문장을 사용 합니다.
- 물음표를 제외 하 고 문장 부호를 포함 하지 않습니다. 
- 작업 제목;으로 동일한 텍스트를 사용 하지 않아도 됩니다. UI의 컨텍스트에서 적합 한 텍스트를 사용 하지만 둘 간의 논리적 연결 인지 확인 합니다. 예를 들어 다음과 같은 가치를 제공해야 합니다. 
- 도움말 링크: 예외 허용의 문제점은 무엇 인가요? 
- 도움말 항목 제목: "Windows 방화벽을 통해 통신 하는 프로그램을 수 있습니다."
- 도움말 항목의 내용에 대 한 최대한으로 한정 됩니다. 
    - 이 스타일
        - Windows 방화벽 내 컴퓨터를 보호 정보
        - 강조 표시를 그림 향상 시킬 수 있습니다 이유
    - 이 스타일 없습니다
        - Windows 방화벽에 대 한 자세한 정보
        - 색 관리에 자세히 알아보기
        - 자세한 내용
- 링크 텍스트, 키 단어 뿐만 아니라 전체 문장을 사용 합니다. 
    - 이 스타일 
        - [예외 허용의 문제점은 무엇 인가요?]()
    - 이 스타일 없습니다
        - 이란 합니다 [예외를 허용 하는 위험이]()? 

## <a name="error-messages"></a>오류 메시지

다음은 Windows 스타일 가이드에서 조정 하는 몇 가지 지침이입니다.

충분 한 설명을 제공 하지만 하지 과도 하 게 기술; 간의 균형은 적절 한 메시지를 작성 합니다. 간에 편안 하 고 멋진 있지만 성가신 또는 불쾌감을 주는 합니다.

### <a name="general-guidelines"></a>일반 지침

오류 사례 당 메시지 한 개를 사용 합니다.

#### <a name="headings"></a>머리글

- 간단 하 게 하 고 문제가 무엇 인지 간결 하 게 설명 하거나 **수행할 작업에 대해 이상적으로**입니다. <br>일부 UI 화면에는 너무 긴 경우 래핑하는 대신 잘라낼, 따라서 관심을 가지십시오 이러한 머리글 있을 수 있습니다.
- 간단한 단계는 머리글에 솔루션을 사용 합니다.
- 판독기에서 본문 텍스트를 무시 하는 경우 머리글 단추에 직접 연결 해야 합니다.
- 다른 선택 되지 않은 경우 "는 문제가" 머리글에서 사용 하지 마세요. 문제에 대 한 보다 구체적인 수 있습니다.
- 머리글의 변수 (예: 파일, 폴더 및 앱 이름)을 사용 하 여를 방지 합니다. 본문에 배치 합니다.

#### <a name="body"></a>본문

- 제목 문제 또는 솔루션에 충분히 설명, 본문 텍스트가 필요가 없습니다.
- 약간 다른 단어를 사용 하 여 메시지의 제목을 반복 합니다.
- 명확 하 게 통신 하 고 간결 하 게 솔루션 이란 키를 누릅니다.
- 팩트를 먼저 제공에 집중 합니다.
- 오류에 대 한 사용자를 탓 하지는 마십시오.
- 없는 경우 오류 코드는 오류와 관련 된 하 고는 오류 코드를 포함 하는 데 도움이 고객 또는 Microsoft 지원에서 문제를 조사 하는 것이 생각 되 면 본문 텍스트를 바로 아래 포함 다음과 같이 작성:

    오류 코드: # # #

    고객 코드 없이 오류를 해결 하는 데 필요한 모든 정보에 포함할 필요가 없습니다.

#### <a name="buttons"></a>단추

- 기본 명령에 대 한 특정 응답 되도록 단추 텍스트를 작성 합니다. 가능 하지 않은 경우에 "닫기" 사건의 단추 텍스트 ("무방" 또는 "완료") 대신 사용 합니다.
- 단추를 여러 개 있으면 맨 왼쪽 단추가 되려면이 작업을 확인 합니다. 맨 오른쪽 단추를 "취소 합니다."와 같은 더 일반적인 작업을 확인

#### <a name="help-links"></a>도움말 링크

구체적이 고 실행 가능한 할 수 없습니다. 오류 메시지에 대 한 도움말 링크를 고려해 야 합니다.

## <a name="null-state-text"></a>Null 상태 텍스트

다음은 Windows 스타일 가이드를 활용 합니다.

고객 데이터 또는 콘텐츠가 없는 경우 null 상태 앱 또는 기능에서 필요할 때 또는 검색 한 후 결과가 반환 될 때 정보가 없는 경우는 트랜잭션에 대 한 정보를 청구와 같은 양식에서.

### <a name="guidelines"></a>지침

- 가능한 경우 기회로 null 상태 상황을 사용 하 여 기능 (예를 들어, 추가 하는 방법, 음악 등 그림을 찾 수 있는 위치입니다.)을 사용 하는 방법에 대 한 사용자 교육  
  - 제목을 UI에 있는 경우 null 상태 (예를 들어, "일부 음악" 추가) "수정" 수행할 동작을 설명 
  - 텍스트와 함께 사용 합니다. 이 공간에 있으므로 해당 않을 표시 됩니다 여러 번 기쁨을 제공할 수 있습니다 수 있습니다. 
  - "이 비어 있습니다."를 방지 합니다. Sad 이며에 저하 되었습니다. 
  - "프린터를 연결 하지 않은?"와 같은 질문을 방지 합니다. 한 번 사용 하 여 확인 하지만이 형식을 가져올 저하 경향이 하 고 질문 추가 부담/부담에 고객. 또한 condescending 느낄 수 것. 
  - Null 상태 텍스트에 다양 한 좋은 것입니다.

|알림 세부 정보|완전한 문장, 개체를 표시하는 UI에 대한 링크가 있는 것이 이상적|

## <a name="tooltips"></a>도구 설명

적절 한 도구 설명을 간단 하 게 레이블이 지정 되지 않은 컨트롤에 설명 또는이 유용한 경우 레이블이 지정 된 컨트롤에 대 한 약간의 추가 정보를 제공 합니다. 추가 제공 하 여 UI를 이동 하는 고객은 유용할 수 있습니다-반복 안 됨-등, 링크, 아이콘, 레이블 컨트롤에 대 한 정보입니다.

도구 설명 꼭 필요할 때만 나타나거나 전혀 나타나지 사용 되어야 합니다. 단순히 레이블을 반복 하거나 명백한 상태는 도구 설명이 포함 되지 않습니다 있도록 고객에 게는 중단 될 수 있습니다. 항상 중요 한 정보를 추가 해야 합니다.

|    컨텍스트                                 |    도구 설명을 작성 하는 방법    |
|    -----------------------                 |    -------------------------    |
|컨트롤 또는 UI 요소가 없는 경우 레이블이 지정 된 하는 중...|간단 하 고 설명이 포함 된 명사구를 사용 합니다. 예를 들어 다음과 같은 가치를 제공해야 합니다.<br> 펜을 강조 표시 |
|경우에 UI 요소를 레이블이 지정 되지만 용도 대 한 설명 해야 하는 중...|<ul><li>이 UI 요소를 사용 하 여 수행할 수 있는 작업을 간략하게 설명 합니다. </li><li>명령형 동사 형식을 사용 합니다. 예를 들어, "텍스트이 파일에서 찾을 이" (없습니다 "텍스트를 찾아이 파일에").</li><li>않는 여러 완전 한 문장 끝 문장 부호를 포함 하지 마십시오.</li> </ul>|
|경우에 텍스트 레이블을 잘린 또는 일부 언어에서 자를 가능성이 있는...|<ul><li>도구 설명에 잘리지 않은 레이블을 제공 합니다.</li><li>선택 사항: 필요한 경우 다른 줄에 명확히 설명만을 제공 합니다.</li><li>잘리지 않은 정보 페이지나 흐름에 다른 곳에서 제공 된 경우에 도구 설명을 제공 하지 마십시오.</li></ul>|
|바로 가기 키를 사용할 수 있는 경우...|<ul><li>선택 사항: 예: 레이블 또는 설명이 포함 된 구 뒤의 괄호에 바로 가기 키를 제공 합니다. "인쇄 (Ctrl + P)" 또는 "(Ctrl + F)이이 파일에서 텍스트 찾기"</li><li>명확히 도구 설명에 유용한 바로 가기 키를 추가 하지만 바로 가기 키를 표시할 도구 설명을 추가 하지 않도록 괜찮습니다. </li></ul>|
