# Windows Admin Center UI 텍스트와 디자인 스타일 가이드

>적용 대상: Windows Admin Center

이 항목에서는 Windows Admin Center용 사용자 인터페이스(UI) 텍스트를 작성하는 일반적인 방법 및 일부 특정 규칙과 접근 방법에 대해 설명합니다.

환경이 쉽게 사용할 수 있고 친숙하도록 Windows Admin Center 및 확장은 [Microsoft의 음성 원칙](https://docs.microsoft.com/style-guide/brand-voice-above-all-simple-human)을 따라야 합니다. 이 스타일 가이드는 이러한 음성 원칙과 [Microsoft 쓰기 스타일 가이드](https://docs.microsoft.com/style-guide/welcome/)를 기반으로 하므로, [접근성](https://docs.microsoft.com/style-guide/accessibility/accessibility-guidelines-requirements), [머리글자어](https://docs.microsoft.com/style-guide/acronyms) 및 [청유](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/p/please) 및 [사과](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/s/sorry)와 같은 [단어 선택](https://docs.microsoft.com/style-guide/word-choice/) 등의 정보에 대한 리소스를 모두 확인하도록 하십시오.

## 버튼

- 버튼은 가능한 경우, 특히 도구를 지역화할 계획이 있는 경우 항상 한 단어여야 합니다. 두세 확인 이지만 더 이상 하지 마십시오. 4 개의 단어를가지고 있거나 더 오래 더 좋을 링크 컨트롤을 사용 합니다.
- 버튼 레이블은 간결하고 구체적이며 쉽게 이해할 수 있어야 합니다. 일반적인 "제출" 버튼 대신 "만들기", "삭제", "추가", "포맷" 등 사용자 작업에 해당하는 동사를 사용합니다.
- 질문에 버튼이 따르는 경우 해당 레이블은 질문(일반적으로 "예" 또는 "아니요")에 명확하게 일치해야 합니다.

## 대문자 표시

Microsoft의 [대문자 표시](https://docs.microsoft.com/style-guide/capitalization) 스타일을 따릅니다. 거의 모든 작업에 문장형 대문자 표시를 사용합니다.

| UI 요소              |대문자 표시|설명|
|-------------------------|--------------|--------|
|배지(예: 미리 보기) |모두 대문자      ||
|모든 기타          |문장 스타일|그러나 제어 범위 밖에 있는 WMI 또는 PowerShell의 개체 속성을 표시하는 몇 가지 예외가 있습니다.|

## 콜론

콜론을 사용 하 여 목록 소개 합니다. 예를 들면 다음과 같습니다.

    Choose one of the following:
    Cats
    Dogs
    Quokkas

레이블을 켜진 다른 정렬 것은에서 레이블 또는 레이블 지정 레이블과 것은 간에 명확한 차이가 있는 경우 UI 텍스트에서 콜론을 사용 하지 마세요.

레이블이 함께 실행에서 두 개의 요소를 유지 해야 하는 레이블 텍스트와 같은 줄 켜져 있을 때 UI 텍스트에서 콜론을 사용 합니다.

## 확인 메시지

확인 대화 상자는 계속 있을 경우에 유용 데이터 손실 등의 예기치 않은 결과입니다. 특히 되돌릴 수 없습니다 이벤트에 대 한 명확한 결과 사용 하 여 스캔이, 유용한 정보를 포함 해야 합니다. 

- 확인이 필요한 있는지 확인 합니다. 제공할 새로운 정보가 없는 경우 (예를 들어 "확실 합니까?") 다음 확인 메시지를 생략할 수도 있습니다.  
- 고객이 계속 작업 하는 것을 확인 합니다.
- 기본 명령 (머리글) 및 설명 텍스트 (본문) 중복 되지 있는지 확인 합니다.
- 머리글에서 질문 또는 발생 하는 다음에 대 한 문 가능한 결과 정의 합니다. 예를 들어 "지우기이 드라이브의 모든 데이터? 또는 "하려는 모든 데이터 지우기".
- 본문에 세부 정보를 추가 합니다. 있으면 다음 변수를 변경 하는 방법에 대 한 중인 항목의 이름과 같은 포함 다음과 같습니다.
- 간단한 질문 (또는 헤더에서 본문)을 둘러싸는 두 작업 단추 간에 명확한 선택을 포함 됩니다.
- 복잡 한 선택의 경우 사용 예/아니요 버튼을 신중 하 게 읽는 것을 권장 합니다. 간단한 선택, 특정 한 작업을 사용 하 여 단추에 대 한 모든 삭제와 같은 또는 취소 합니다.

## 첫 실행 경험

사용자가 페이지를 처음으로 방문하면 도구를 사용해 볼 수 있는 기회가 있습니다. 다음과 같을 수 있습니다.

- 시작하는 방법을 간단하게 설명하는 빈 페이지의 텍스트 스트링. 예: "'추가'를 선택하여 앱을 선택합니다."
- 사용자가 시작할 수 있는 컨트롤에 대한 링크. 예: "앱을 추가하여 시작합니다."
- 사용자에게 시작하는 방법을 보여주는 소규모 및 간단한 애니메이션 또는 비디오

다음은 Windows 스타일 가이드의 유용한 팁입니다.

### 1. 도움 되기

- 마케팅 스타일 및 언어를 피합니다.
- 무엇을 설명하거나 제안하는 경우 최종 결과가 명확하도록 합니다. 단지 고객에게 특정 작업을 수행하는 방법을 보여주는 것은 그 이유를 모르는 경우 유효하지 않습니다.
- 고객이 이를 필요로 하지 않는 경우 팁을 제시하지 마십시오.

### 2. 말하는 대신 보여주기

텍스트를 최대한 단순하게 유지합니다(작은 애니메이션 또는 비디오).

### 3. 부담 주지 않기

- 시스템 알림 및 셸 알림을 포함하여 팝업 및 팁을 사용 세션당 4개로 제한합니다.
- 팝업의 타이밍이 도움이 되는지 확인합니다.
- 고객의 작업 수행을 방해하지 마십시오.
- 팝업을 쉽게 닫을 수 있는지 확인하십시오.

### 4. 상황에 맞는 설명

- 고객을 가르치는 것은 적절한 시기에 제공하는 경우 효과적입니다.
- 자습서 또는 슬라이드쇼를 만드는 경우 정보를 구체적으로 만듭니다.
- 마케팅적인 "포장"보다 구체적인 팁과 유용한 정보에 중점을 둡니다.
- 관련된 경우 고객이 나중에 자습서로 돌아올 수 있는 경로를 제공합니다. (사람들은 종종 처음에는 정보를 기억하지 않지만 설정 지침이 관련이 있을 수 있습니다.)
- 빈 상태 메시징은 학습 및/또는 즐거움에 자연스러운 위치입니다. 메시지를 단순하고 유용하게 만듭니다.

### 5. 번거로운 설정 최소화

고객이 전체 가치를 경험하게 하기 위해 다른 작업을 수행해야 하는 경우(온라인 서비스에 로그인 등), 가능한 한 번거롭지 않게 만듭니다.

- 메시지는 짧고 직접적이어야 합니다.
- 논점을 흐리지 마십시오. 가능하면 현재 위치에서 연결할 수 있는 수단을 제공합니다.
- 가능하면 나중에 작업을 수행하는 옵션을 허용하고 나중에 수행하도록 알립니다.
- 고객의 환경에서 벗어나게 하고자 하는 경우 빠르고 쉽게 돌아가는 방법을 제공합니다.

## 도움말 링크

다음은 Windows 스타일 가이드의 유용한 팁입니다.

### 도움말 링크 때 제공 해야 하나요?

거의 없습니다. 도움말 링크를 제공 경우에만:

- 고객에 게는에 대 한 대답 UI 작업에 성공 하는 데 도움이 됩니다 UI에 있는 동안 되어 있을 가능성이 있는 명확 하 고 중요 한 질문이 있습니다. 
- 양을 사용자가 UI 작업에 성공 하는 데 필요한 정보를 제공 하기 위해 UI에 충분 한 공간이 없습니다. 

### 여기서 링크 표시 되는 데 도움이 됩니다. 

- 텍스트 링크 도움말 가능한 대상으로 하는 UI 요소 깝 나타나야 합니다. 
- UI 화면을 전체에 적용 되는 텍스트 링크를 제공 해야 하는 경우 화면의 왼쪽 아래에 배치 합니다. 
- 도움말 보기 버튼 (?)을 통해 링크를 제공한 경우 tooltip "도움말." 이어야 합니다.

### 어떤 URL을 사용 해야 하나요?

Microsoft 개발자는 FWLink 사용자 수를 수동으로 입력을 사용 하는 경우 aka.ms 링크 (으로 URL의 대상이 Docs.microsoft.com의 브라우저 로캘이 자동으로 인식 하는 웹 사이트)가 도움말 링크는 경우에만 사용 해야

## 오류 메시지

Windows 스타일 가이드에서 적용 하는 몇 가지 지침은 다음과 같습니다.

좋은 메시지를 작성 하는 것은 충분 한 설명을 제공 하지만 없는 technical 지나치게; 사이 균형 간에 적인 멋진 하지만 남았습니다 나 아니라 불쾌 될 예정입니다.

### 일반 지침

오류 사례 당 하나의 메시지를 사용 합니다.

#### 머리글

- 간단 하 게 하 고 문제가 무엇 인지 또는 **이상적으로 무엇을**간결 하 게 설명 합니다. <br>일부 UI 화면 너무 긴 경우 배치 하는 대신 잘라내기, 따라서 잘 찾아봐야가는 머리글 있을 수 있습니다.
- 간단한 단계를 수행 하는 경우 제목에서 솔루션을 사용 합니다.
- 리더에서 본문 텍스트를 무시 하는 경우 제목 단추에 직접 연결 되었는지 확인 합니다.
- 다른 선택 되지 않았으면 "문제가 발생"에서 머리글을 사용 하지 마세요. 문제에 대 한 더욱 구체적인 수 있습니다.
- 머리글에서 변수 (예: 파일, 폴더 및 앱 이름)를 사용 하지 마세요. 본문에 배치 합니다.

#### Body

- 머리글 충분히 문제 또는 솔루션에서 설명 하는 경우 본문 텍스트가 필요는 없습니다.
- 표현만 조금 메시지에서 제목을 반복 하지 마십시오.
- 명확 하 게 통신 하 고 간결 하 게 솔루션 이란.
- 먼저 정보 제공에 집중 합니다.
- 오류에 대 한 사용자 탓 하지는 마십시오.
- 있으면 오류 코드 오류와 관련 된 및 고객 또는 Microsoft 지원 하 여 문제를 조사 오류 코드를 포함 하 여 도움이 본문 텍스트 바로 아래 포함 하 고 다음과 같이 씁니다.

    오류 코드: # # #

    고객이 있는 경우 코드 없이 오류를 해결 하는 데 필요한 모든 정보를 포함할 필요가 없습니다.

#### 단추

- 기본 명령에 대 한 특정 응답 되도록 단추 텍스트를 작성 합니다. 가능 하지 않은 경우 "닫기" 철회 단추 텍스트 ("그것도" 또는 "수행") 하는 대신 사용 합니다.
- 둘 이상의 단추를 맨 왼쪽 단추로 작업을 확인 해야 할 사용자는 것이 좋습니다. "취소 합니다."와 같은 더 보수적 작업의 오른쪽에 있는 단추 만들기

#### 도움말 링크

특정 하 고 실행 가능한 걸 수 없는 오류 메시지에 대 한 도움말 링크 고려 합니다.

## Null 상태 텍스트

다음은 Windows 스타일 가이드에서 일부 도움말이입니다.

고객 데이터 또는 콘텐츠가 없을 때 발생 하는 null 상태에서 앱 또는 기능을 검색 한 후 또는 필요할 때 결과가 반환 되는 경우 정보는 결제는 거래에 대 한 정보 등, 양식에서 누락 합니다.

### 지침

 - 가능한 경우 null 상태 상황 (예를 들어 추가 하는 방법을 음악, 위치 찾기 사진 등.) 기능을 사용 하는 방법에 대 한 사용자를 교육 하는 기회 사용  
- 제목을 UI에 있는 경우 "수정" "(예를 들어," 추가 일부 음악) null 상태를 수행할 작업을 설명 
- 텍스트를 사용 하 여 즐기세요. 이 공간을이 아닐 표시 여러 번 이후 재미 있게를 제공할 수 될 수 있습니다. 
- "는 유일한 여기에서" 방지 슬픈 이며가 지나치게 사용 되었습니다. 
- "프린터를 연결 하지 않은?" 같은 질문을 방지 한 번 사용 하 여 확인이 형식은 가져오기 과도 하 게 사용할 경향이 있고 질문 고객에 추가 해야 하는 부담/압력을 배치 합니다. 또한 condescending 느끼게 하면 수 있습니다. 
- Null 상태 텍스트에서 다양 한 것입니다. 

### 예

- "를 즐겨찾기로 추가 및 여기 표시 됩니다."

## 문장 부호

- 머리글이나 불완전한 문장에 끝 문장 부호(마침표, 물음표) 없음. 예외는 제목이 질문을 표시하는 확인 대화 상자에 있음
- [기간](https://docs.microsoft.com/style-guide/punctuation/periods) 및 [물음표](https://docs.microsoft.com/style-guide/punctuation/question-marks)에 대한 Microsoft 스타일 가이드의 지침을 사용합니다.

## 상태 메시지

상태 메시지는 팝업(알림) 메시지와 알림으로 구성됩니다.

|문자열 형식         | 메모                               |
|------------        |-------------------------------------|
|알림               |마침표가 있는 첫 글자를 대문자로 - 사용자가 대상에서 벗어난 경우 메시지가 적용되는 대상을 이해할 수 있도록 개체 변수를 사용하는 것이 이상적|
|알림 제목|마침표가 있든 없든 첫 글자를 대문자로(제목) - 개체 변수를 사용하는 것이 이상적|
|알림 세부 정보|완전한 문장, 개체를 표시하는 UI에 대한 링크가 있는 것이 이상적|

알림 메시지에 대한 일부 자세한 권장 사항은 다음과 같습니다.

|문자열 형식         | 메모                               |
|------------        |-------------------------------------|
|시작             |가능한 경우 생략 - 일반적으로 주의를 분산시키는 요소를 최소화하기 위해 진행 중인 메시지를 생략할 수 있습니다.|
|진행 중         |수행 중인 작업의 동사로 시작하고 타원을 진행 중인 작업을 나타내기 위해 줄임표로 끝냅니다. 예를 들면 다음과 같습니다.<br> *"고객 데이터" 볼륨을 만드는 중...*|
|성공             |"성공적으로"로 시작하고 소프트웨어가 성공한 작업으로 끝냅니다. 예를 들면 다음과 같습니다.<br> *성공적으로 "고객 데이터" 볼륨 키를 만들었습니다.*|
|실패             |소프트웨어가 수행하지 못한 작업으로 시작하고 "~할 수 없습니다"로 끝냅니다. 예를 들면 다음과 같습니다.<br> *볼륨 "고객 데이터"를 만들 수 없습니다.*|

## 도구 설명

좋은 도구 설명을 간단 하 게 레이블이 지정 되지 않은 컨트롤에 설명 하거나 유용 때 레이블이 있는 컨트롤에 대 한 약간의 추가 정보를 제공 합니다. 추가 제공 하 여 UI를 탐색 하는 고객도 확인해보시기-안됨-컨트롤 레이블, 아이콘, 링크에 대 한 정보.

가급적 또는 전혀 도구 설명은 사용 해야 합니다. 단순히 반복 레이블 또는 상태 분명 하는 도구 설명이 포함 하지 않는 고객에 게 중단 될 수 있습니다. 항상 중요 한 정보를 추가 해야 합니다.

|    컨텍스트                                 |    도구 설명을 작성 하는 방법    |
|    -----------------------                 |    -------------------------    |
|컨트롤 또는 UI 요소가 없는 경우 레이블이 지정 된...|간단 하 고 설명 명사구를 사용 합니다. 예를 들면 다음과 같습니다.<br> 펜을 강조 표시 |
|UI 요소는 레이블이 지정 하지만 용도 다듬어야 경우...|<ul><li>이 UI 요소를 사용 하 여 수행할 수 있는 작업을 간단히 설명 합니다. </li><li>명령적 동사 양식을 사용 합니다. 예를 들어 "텍스트이 파일에서 찾을 이" (하지 "텍스트를 찾아이 파일에").</li><li>않는 여러 완전 한 문장에 끝 문장 부호를 포함 하지 마세요.</li> </ul>|
|때 텍스트 레이블이 잘리거나 일부 언어로 자를 가능성이...|<ul><li>도구 설명에 untruncated 레이블을 제공 합니다.</li><li>옵션: 다른 선에 명료 설명을 제공만 필요한 경우.</li><li>Untruncated 정보 페이지 또는 흐름에 다른 곳에서 제공 하는 경우 도구 설명을 제공 하지 마십시오.</li></ul>|
|바로 가기 키를 사용할 수 있는 경우...|<ul><li>선택 사항: 괄호로 레이블 또는 설명 구에 따라 "인쇄 (Ctrl + P)" 또는 "(Ctrl + F)이이 파일에서 텍스트를 찾기" 예: 바로 가기 키를 제공 합니다.</li><li>것이 확인 명료 tooltip, 유용한 바로 가기 키를 추가 바로 가기 키를 표시 하도록 도구 설명을 추가 하지 마세요. </li></ul>|
