## <a name="ADATM_BM"></a>Active Directory 관리 계층 모델
이 계층 모델 환경 (계층 0)의 모든 권한 및 공격자 자주 손상 높은 위험 워크스테이션 자산 간에 버퍼가 영역 집합을 사용 하 여 신원을 시스템 보호 하는 것입니다.

![세 가지 계층 계층 모델을 보여 주는 다이어그램](../media/securing-privileged-access-reference-material/PAW_RM_Fig1.JPG)

계층 모델의 세 수준 중 구성 되며 하지 표준 사용자 계정을 관리자 계정 포함 됩니다.

-   **계층 0** -엔터프라이즈 id 환경에서 직접 제어 합니다. 계층 0 계정, 그룹 하 고 직접 관리 제어 Active Directory 숲, 도메인 또는 도메인 컨트롤러의 기타 자산 및 모든 자산에 포함 됩니다. 서로 제어 효과적으로 모두은 보안 감도 모든 계층 0 자산 매각과 같습니다.

-   **1 단계** -응용 프로그램 및 엔터프라이즈 서버를 제어 합니다. 1 단계 자산 서버 운영 체제, 클라우드 서비스 및 엔터프라이즈 응용 프로그램에 포함 됩니다. 계층 1 관리자 계정에서 이러한 자산 상당한 양의 호스트 비즈니스 가치를 관리 제어할 수 있습니다. 일반적인 예 역할은이 운영 체제 모든 엔터프라이즈 서비스에 영향을 미칠 수 있는 기능을 유지 하는 서버 관리자입니다.

-   **2 계층** -사용자 워크스테이션 및 장치를 제어 합니다. 계층 2 관리자 계정 있는 사용자 워크스테이션 및 디바이스에서 호스트 하는 비즈니스 값 상당한의 관리 제어 합니다. 예를 들면 기술 지원 서비스 및 컴퓨터 지원 관리자 거의 모든 사용자 데이터의 무결성 영향을 줄 수 있으므로 보세요.

> [!NOTE]
> 계층 관리 자산 보호 하기 위한 기본 우선 순위 지정 메커니즘 역할도 있지만 언제 든 계층 모든 자산 제어 공격자 대부분 나 모든 비즈니스 자산에 액세스할 수 있는지 고려 하 중요 합니다. 기본 우선 순위 지정 메커니즘 유용 이유 공격자 어려움/비용입니다. 모든 id (계층 0) 또는 서버의 모든 권한을 함께 작동 하도록 공격자 쉽게 이며 것 보다 클라우드 서비스 (계층 1) 각 개별 워크스테이션 또는 사용자 장치 조직의 데이터 (계층 2)에 액세스 해야 하는 경우 수 있습니다.

### <a name="containment-and-security-zones"></a>제약 및 보안 영역
계층 특정 보안 영역 기준 으로합니다. 하는 많은 이름 고대의 보안 영역 서로 네트워크 계층 격리를 통해 보안 위협에 포함 되를 제공 하는 정립 접근 됩니다. 격리 된 포함 악성 네트워크 격리 적용 되지 않는 보안 영역 내에서 제공 하 여 보완 하는 계층 모델 합니다. 보안 영역 두 온-프레미스도 확장 하 고 도메인 컨트롤러 및 도메인 회원 같은 도메인에 있는 호스팅된 온-프레미스 예제와 Azure 인프라와 같은 클라우드 수 있습니다.

### <a name="primary-responsibilities-and-critical-restrictions"></a>기본 책임 및 중요 한 제한 사항을
**계층 0 관리자** -id 저장소 및 적은 수의, 효과적인 제어 하는 시스템의 관리 하 고 있습니다.

-   관리 하 고 필요에 따라 모든 수준 자산 제어할 수 있습니다

-   만 대화형 로그온 또는 자산 계층 0 수준 신뢰할 수 있는 액세스할 수 있는

**1 단계 관리자** -엔터프라이즈 서버, 서비스 및 응용 프로그램을 관리 하 고 있습니다.

-   관리 하 고 수 자산 계층 1 또는 2 계층 수준 제어

-   만 액세스 자산을 (네트워크 로그온 형식)를 통해 계층 1 또는 계층 0 수준 신뢰할 수 있습니다.

-   대화형만 자산 계층 1 수준 신뢰할 수 있는에 로그온 할 수 있습니다.

**2 계층 관리자** -엔터프라이즈 데스크톱, 노트북, 프린터 및 다른 사용자 장치를 관리 하 고 있습니다.

-   만 관리 하 고 2 계층 수준 자산 제어할 수 있습니다.

-   필요에 따라 모든 수준 자산 (네트워크 로그온 형식)를 통해 액세스할 수 있습니다.

-   2 계층 수준 신뢰할 수 있는 자산에 로그온 할 수만 대화형

## <a name="CSP_BM"></a>새로 소스 원칙
새로 소스 원칙을 개체로 보안을 유지 되 고 신뢰할 수 있는 모든 보안 종속성 필요 합니다.

![개체를 제어 제목을 개체의 보안 종속성은 방법을 보여 주는 다이어그램](../media/securing-privileged-access-reference-material/PAW_RM_Fig5.JPG)

모든 주제 개체를 제어 개체의 보안 종속성입니다. 악의적인 사용자 대상 개체 제어 효과적으로 표시 된 것을 제어할 수를 해당 대상 개체를 제어할 수 있습니다. 이 때문에 대 한 모든 보안 종속성 보장 개체 자체의 원하는 보안 수준 이상 인지 확인 해야 합니다.

관심 (개체) 자산 제어 관계를 이해 하 고 모든 보안 종속성 (Subject(s))를 검색 려의 종속성을 분석 필요이 적용 하는 동안 원칙에서 간단 하 고.

제어 전이 이기 때문에이 원칙에서 반복된 반복적으로 있어야 합니다. 들어 B 및 B 컨트롤 C 컨트롤에 대 한 다음 A도 직접 컨트롤 하지 C.

![B 및 B 컨트롤 C를 제어 하는 경우 다음 A도 직접 제어 하는 방법을 하지 C을 보여 주는 다이어그램](../media/securing-privileged-access-reference-material/PAW_RM_Fig6.JPG)

모든 B 및 컨트롤 (B 포함), 타협점에 가져옵니다 모든 항목에 액세스 공격자 (C) 포함 제어할 수 있습니다. 보안 종속성 언어를 사용 하 여 같은 예의를 B 및 A c 보안 종속성 되며 해당 보증 수준으로 C 하기 위해에서 C 보증 원하는 수준에서 보안 됩니다.

IT 인프라와 id 시스템에 대 한이 원칙 시스템 설치 된 하드웨어를 포함 하 여 컨트롤, 시스템, 아키텍처와 시스템과 일상적인 작업 구성 용 설치 미디어 하는 가장 일반적인 수단에 적용 해야 합니다.

#### <a name="software-acquisition"></a>소프트웨어 인수
소스 소프트웨어의 다음 방법 중 하나를 통해 유효성 검사를 해야 합니다.

-   소프트웨어는 제조업체 또는 신뢰할 수 있는 원본 공급 업체에서 제공 하는 일반적으로 제조 미디어에서 제공 하는 물리적 미디어에서 가져옵니다.

-   소프트웨어는 인터넷에서 가져온와 공급 업체에서 제공 파일 해시 유효성을 검사 합니다.

-   소프트웨어는 인터넷에서 가져온 및 다운로드 하 여 독립 복사본을 2 개 비교 하 여 유효성을 검사:

    -   보안 관계 없이 두 호스트로 다운로드 (동일한 도메인에 동일한 도구에 의해 하지 관리), 별도 인터넷 연결에서 것이 좋습니다.

    -   Certutil 같은 유틸리티를 사용 하 여 다운로드 한 파일 비교 합니다.

        `certutil -hashfile <filename>`

When possible, all application software, such as application installers and tools should be digitally signed and verified using Windows Authenticode with the [Windows Sysinternal](https://www.microsoft.com/sysinternals)s tool, *sigcheck.exe*, with revocation checking. 일부 소프트웨어 필요할 수는 공급 업체에서 디지털 서명 이러한 유형의 제공 하지 않을 수 있습니다.

#### <a name="software-storage-and-transfer"></a>소프트웨어 저장 및 전송이
소프트웨어를 가져온 후 호스트 인터넷에 연결 된 또는 소프트웨어 또는 운영 체제 설치 수 있는 시스템 보다 하위 수준에서 신뢰할 수 있는 직원으로 특히 수정 로부터 보호 되는 위치에 저장 합니다. 이 저장소 실제 미디어 또는 전자 안전한 위치 될 수 있습니다.

#### <a name="software-usage"></a>소프트웨어 사용
것이 좋습니다 소프트웨어를 사용 하는 경우 수동으로 설치, 포장 구성 관리 도구에 대 한 인지 구성 관리 도구로 가져올 등 당시 유효성 검사 해야 합니다.

### <a name="clean-source-for-architecture-and-design"></a>새로 소스 아키텍처와 디자인에 대 한
시스템 아키텍처에 새로 소스 원칙이 적용 시스템 아래 보안 시스템에 따라 달라 있는지 확인 해야 합니다. 시스템에 따라 달라질 수 있습니다 낮은 보안 표준을 낮은 보안 시스템에는 없지만 높은 신뢰 시스템에서 합니다.

예를 들어, 그에 적합 한 Active Directory 하지만 표준 사용자의 바탕 화면을 제어 하는 Active Directory를 제어 하는 표준 사용자 데스크톱에 대 한 권한 위험의 중요 한 확대 합니다.

![시스템 높은 보안 시스템에 따라 달라 수 있는 방법을 보여 주는 다이어그램 낮은 보안 표준 낮은 신뢰 시스템](../media/securing-privileged-access-reference-material/PAW_RM_Fig09.JPG)

보안 액세스 제어 (Acl 목록)를 포함 하 여 파일을 컴퓨터 또는 체제로 (임의 코드와 스크립트 실행 하는 기능)를 실행 하는 컴퓨터에 설치 된 상담원에 로컬 관리자가 그룹의 회원 같은 물체에서 많은 수단을 통해 제어 관계를 제공할 수 있습니다.

자주 간과 예제는 노출 통해 로그온, 다른 시스템 시스템 관리자 자격 증명을 제공 하 여 제어 관계를 만듭니다. 원인을 공격 해시 전달 같은 자격 증명 도용의 매우 강력 이유입니다. 관리자 표준 사용자 데스크톱 계층 0 자격 증명으로 로그인 할 때 해당 자격 증명을의 광고 컨트롤에 삽입 하 고 ad 권한 경로의 에스컬레이션 만드는 데스크톱에 노출 있습니다. 이러한 공격에 대 한 자세한 내용은 참조 [이 페이지](https://technet.microsoft.com/en-us/security/dn785092)합니다.

많은 수의 Active Directory 등의 신원 시스템에 따라 달라 자산,으로 인해 시스템에 따라 달라 고 Active Directory 수를 최소화 해야 합니다.

![시스템 수가 Active Directory 최소화 해야 및 도메인 컨트롤러에 따라 달라을 보여 주는 다이어그램](../media/securing-privileged-access-reference-material/PAW_RM_Fig010.JPG)

###### <a name="administrative-personnel-standards"></a>관리 담당자 표준
지정 된 관리 담당자 신뢰할 수 있는 되 고 관리자 권한이 필요 되도록 검사 해야 합니다.

-   관리자 권한이 지정 전에 담당자 배경 검사를 수행 합니다.

-   관리자 권한이 각 분기 검토 하는 담당자는 여전히 적법 한 업무를 확인 하려면 관리자 권한이 필요 합니다.

###### <a name="administrative-security-briefing-and-accountability"></a>관리 보안 브리핑 및 책임
관리자 정보를 제공 하 고 조직의 역할 위험을 관리 하는 위험에 대 한 책임 있어야 합니다. 관리자는에 매년 교육 해야 합니다.

-   일반 위협 환경

    -   결정된 악성

    -   도난 자격 증명을 해시를 통과 등을 포함 기술을 공격합니다

-   조직 관련 위협과 문제

-   관리자 역할 공격 으로부터 보호 합니다.

    -   자격 증명 노출 계층 모델 관리

    -   관리 워크스테이션 사용

    -   원격 데스크톱 프로토콜 RestrictedAdmin 모드 사용

-   조직 관련 관리 방법

    -   이 표준 모든 작업 지침이 검토

    -   다음 주요 규칙을 구현 합니다.

        -   사용 하지 마십시오 관리자 계정 관리 워크스테이션 제외

        -   사용 하지 않도록 설정 하지 않거나 계획인 워크스테이션 (예: 로그온 제한을 또는 스마트 카드에 필요한 속성) 또는 계정에 보안 컨트롤

        -   문제 보고 또는 비정상적인 활동

책임을 제공 하려면 관리자 계정 사용 하 여 모든 직원 조직 관련 관리 정책 사례에 따라 하려고 한다는 관리자 권한 준수 문서에 서명 해야 합니다.

###### <a name="provisioning-and-deprovisioning-processes-for-administrative-accounts"></a>관리자 계정에 대 한 프로세스 발급 및 프로 비전
수명 주기 요구 사항을 충족한 다음 표준은 충족 해야 합니다.

-   다음 표에 설명 된 승인 기관이 모든 관리자 계정 승인을 받아야 합니다.

    -   승인만 받아야는 담당자가 합법적 하는 경우 비즈니스 관리자 권한이 필요 합니다.

    -   관리자 권한이 대 한 승인 6 개월 초과 하지 않아야 합니다.

-   관리자 권한이에 액세스 해야 즉시 때 링고에서:

    -   담당자는 위치를 변경 합니다.

    -   담당자 조직을 둡니다.

-   계정 조직 두면 인사 뒤 즉시 비활성화할 수 있어야 합니다.

-   6 개월 이내 비활성화 된 계정을 삭제 해야 하 고 변경 승인 보드 기록을 삭제 하기 기록을 입력 해야 합니다.

-   모든 권한이 있는 계정 구성원 매월 무단된 사용 권한이 부여 된 되도록 검토 합니다. 이 변경에 게 알려 주는 자동화 된 도구 교체 합니다.

|계정 권한 수준|승인 기관|멤버십 검토 주파수|
|--------------|------------|----------------|
|계층 0 관리자|승인 보드 변경|매월 나 자동화 된|
|1 단계 관리자|계층 0 관리자 또는 보안|매월 나 자동화 된|
|2 계층 관리자|계층 0 관리자 또는 보안|매월 나 자동화 된|

##### <a name="operationalize-least-privilege"></a>최소 권한 operationalize
이러한 표준 최소 권한 관리자 역할의 수와 권한을 수 있는 시간을 줄이고 달성 하는 데 도움이 됩니다.

> [!NOTE]
> 조직에서 최소 권한을 달성 조직 역할, 요구 사항 및 디자인 메커니즘 해당 최소 권한을 사용 하 여 작업을 수행할 수 있는지 확인 하려면 이해 필요 합니다. 최소 권한 관리 모델에서의 상태를 자주 달성 여러 가지 사용을 해야 합니다.
>
> -   관리자의 수 나 권한이 있는 그룹의 회원 제한
> -   계정에 더 적은 권한을 위임합니다
> -   필요에 따라 시간 범위 권한을 제공합니다
> -   다른 담당자 (호텔 방법) 작업을 수행 하는 기능을 제공 합니다.
> -   긴급 액세스 및 사용 희귀 시나리오에 대 한 프로세스를 제공 합니다.

###### <a name="limit-count-of-administrators"></a>관리자 개수 제한
두 명의 자격 있는 직원의 최소 비즈니스 연속성 되도록 관리 역할을 할당 되어야 합니다.

역할에 할당 직원의 수를 초과 두, 변경 승인 팀에서 원래 2 등 각 개별 구성원에 게 권한이 지정 하는 방법에 대 한 구체적인 이유 승인 해야 합니다. 승인 하는 이유를 포함 해야 합니다.

-   관리자 권한이 필요 관리자가 기술 어떤 작업을 수행

-   작업을 수행 하는 빈도

-   대신 다른 관리자가 작업을 수행할 수 없습니다 이유 특정 이유

-   문서 모든 다른 알려진된 대안을 부여 하 여 권한 및 각 사용할 수 없는 이유

###### <a name="dynamically-assign-privileges"></a>동적으로 권한이 지정
관리자는 작업을 수행 언제 든 지 사용할 수 있는 권한을 "에서 적시" 얻으려면 해야 합니다. 관리자 계정에 사용 권한이 영구적으로 지정 됩니다.

> [!NOTE]
> 관리자 권한이 영구적으로 지정 된 관리 담당자에 빠르게 액세스할 수 있는 문제 인지 운영 가용성 유지 하기 위해 사용 권한 필요 하기 때문에 자연스럽 "대부분 권한" 전략을 만듭니다. 적시에 사용 권한을 하는 기능을 제공합니다.
>
> -   사용 권한을 할당 세부적으로, 최소 권한 수 있 합니다.
> -   권한 노출 시간 줄이기
> -   추적 권한을 남용 또는 공격 검색을 사용 합니다.

##### <a name="manage-risk-of-credential-exposure"></a>자격 증명 노출 위험을 관리
다음과 같은 방법을 사용 하 여 적절 한 노출 자격 증명 위험을 관리할 수 있습니다.

###### <a name="separate-administrative-accounts"></a>별도 관리자 계정
관리자 권한이 보유 하 고을 실시 권 자는 모든 담당자 사용자 계정에서 독특한 관리 기능에 대 한 별도 계정이 있어야 합니다.

-   **표준 사용자 계정을** -표준 사용자에 게 권한이 부여 표준 사용자 작업 메일 등, 검색 및 온라인 비즈니스 응용 프로그램을 사용 하 여 웹를 합니다. 이러한 계정은 관리자 권한이 부여 하지 해야 합니다.

-   **관리자 계정** -계정이 관리 적절 한 권한이 할당 된 인사에 대해 생성 합니다. 각 계층에서 자산 관리 하는 데 필요한 하는 관리자 각 계층 별도 계정이 있어야 합니다. 이러한 계정에 메일 또는 공개 인터넷에 액세스할 수 없으며 있어야 합니다.

###### <a name="administrator-logon-practices"></a>관리자 로그온 관행
관리자에 로그온 할 수는 호스트 대화형 로컬로 (표준 RDP, RunAs을 사용 하 여 또는 virtualization 본체를 사용 하 여)을 통해, 되기 전에 해당 호스트를 충족 하거나 초과 표준 계층 관리자 계정에 대 한 (또는 이상 계층) 해야 합니다.

관리자는 관리자 워크스테이션 관리 자신의 계정으로 로그인 할 수만 있습니다. 관리자만 로그온 관리 리소스에는 다음 섹션에 설명 된 승인된 지원 기술을 사용 하 여.

> [!NOTE]
> 이 해당 호스트로 돌아갑니다 자격 증명 제어 권한을 부여 대화형 호스트에 로그온 할 필요 합니다.
>
> 참조는 [관리 도구와 로그온 유형이](http://aka.ms/admintoolsecurity) 로그온 유형, 일반 관리 도구 및 자격 증명 노출 대 한 자세한 내용은 합니다.

###### <a name="use-of-approved-support-technology-and-methods"></a>승인 된 지원 기술과 방법 사용
관리자에 게 지원 원격 시스템 및 사용자가 원격 컴퓨터를 제어 악의적인 사용자 도용을 관리 자격 증명을 방지 하려면 다음이 지침을 따라야 합니다.

-   사용할 수 있는 경우 기본 지원 옵션을 사용 해야 합니다.

## <a name="ESAE_BM"></a>ESAE 관리 숲 디자인 방법
이 섹션에는 관리자 숲 cybersecurity 공격 하는 고객을 보호 하기 위해 Microsoft의 cybersecurity 전문 서비스 팀에서 배포 강화 된 보안 관리 환경 (ESAE) 참조 아키텍처에 따라에 대 한 접근 포함 되어 있습니다.

전용된 관리 숲 생산 환경 보다 더 강력한 보안 컨트롤이 있는 환경에서 조직 호스트 관리자 계정, 워크스테이션 및 그룹을 허용 합니다.

이 아키텍처 가능한 또는 쉽게 구성 된 권한 있는 액세스 워크스테이션 (발)를 사용 하 여 관리할 하나 단일 숲 아키텍처에서 되지 않는 보안 컨트롤 수가 있습니다. 이 방법은 표준 권한이 없는 사용자로 관리 숲 속의 생산 환경에서 매우 특권 되는 통제의 큰 기술 적용 되도록 계정을 프로비저닝 수 있게 합니다. 이 아키텍처 해줍니다 신뢰 선택 인증 기능을 제한 하는 수단으로 사용 권한이 있는 호스트 로그온 (및 자격 증명 노출)에 합니다. 비용과 전체 재구축하기 복잡성 높인 보증 큰 수준을 프로덕션 숲에 대해 원하는 경우, 관리 숲 보증 수준의 생산 환경 향상 된 환경을 제공할 수 있습니다.

이 방법을 사용을 숲 Active Directory 환경에서는 추가 고정 된 디자인을 작은 하드웨어/소프트웨어 발자국 적은 수의 사용자 비용과 복잡성 제한 됩니다.

> [!NOTE]
> 많은 응용 프로그램이 계정을 하 여 표준 신뢰를 사용 하는 외장형 숲 속의 관리 되 고과 호환 되지 하지만이 방법을 Active Directory를 관리 하는 데 적합 합니다.

다음이 그림에서는 계층 0 자산과 Microsoft Id 관리자 권한이 있는 액세스 관리 기능을 사용 하도록 구성 개인 숲 속의 관리를 위해 사용 하는 ESAE 숲을 보여 줍니다. 에 대 한 자세한 내용은 MIM PAM 인스턴스 배포, [Active Directory (AD DS 도메인 서비스)에 대 한 권한 있는 Id 관리](https://technet.microsoft.com/en-us/library/mt150258.aspx) 문서 합니다.

![표시 된 ESAE 숲 계층 0 자산과 Microsoft Id 관리자 권한이 있는 액세스 관리 기능을 사용 하도록 구성 개인 숲 속의 관리를 위해 사용 되는 그림](../media/securing-privileged-access-reference-material/PAW_RM_Fig14.JPG)

전용된 관리 숲은 표준 단일 도메인 Active Directory 관리 기능을 전용된 Active Directory 숲입니다. 관리 숲 및 도메인 프로덕션 숲 제한으로 인해 사례를 사용 하는 것 보다 엄격 강화 수 있습니다.

관리 숲 디자인 다음 사항을 포함 됩니다.

-   **범위에 제한이** -관리자 숲 기본 값은 높은 수준의 보안 assurance 및 감소 공격 남은 위험 감소 발생 합니다. 숲 쉽습니다 추가 관리 기능과 응용 프로그램을 사용할 수 있지만 범위 내에서 각 증가 숲 및 리소스에 대 한 공격 높아집니다. 목표 각 범위 증가 신중 하 게 고려해 야 하므로, 최소한의 공격을 유지 하려면 내부 숲 및 관리자 사용자의 기능을 제한 하는 것입니다.

-   **구성 신뢰** -관리 숲 (s) 로부터 신뢰 구성 또는 도메인이를 관리 숲

    -   단방향 보안은 생산 환경에서 관리자 숲 합니다. 이 숲 신뢰 나 도메인 신뢰 수 있습니다. 관리자 숲 도메인 양방향 보안 관계, 보안 유효성 검사 응용 프로그램을 추가로 필요할 수 있지만 Active Directory를 관리 하는 도메인/숲 관리를 신뢰 하지 않아도 하 고 테스트 합니다.

    -   선택 인증 관리자 숲 속의 계정에만 해당 환경 호스트에 로그온 제한 하는 것 같습니다. 도메인 컨트롤러를 유지 하 고 권한 Active Directory에 위임, 일반적으로 필요 "로그온 하도록 허용" 부여 관리자 숲 속의 지정된 계층 0 관리자 계정에 도메인 컨트롤러에 대 한 바로 합니다. 선택적 인증 설정을 구성에 대 한 자세한 내용은 참조 하십시오.

-   **도메인 보안 강화 및 권한** -Active Directory 관리에 대 한 요구 사항에 따라 최소 권한 관리 숲 구성 해야 합니다.

    -   도메인 컨트롤러를 관리할 수 있는 권한을 위임 하는 권한을 부여 하려면 관리자 숲 계정을 BUILTIN\Administrators 도메인 로컬 그룹에 추가 해야 합니다. 즉, 도메인 관리자 글로벌 그룹 구성원 외부 도메인에서 사용할 수 없습니다.

    -   이 그룹을 사용 하 여 권한을 부여 하려는 대 한 개 주의 없습니다 액세스할 수 있는 관리 하려면 새 그룹 정책 개체 기본적으로입니다. This can be changed by following the procedure in [this knowledge base article](https://support.microsoft.com/kb/321476) to change the schema default permissions.

    -   프로덕션 환경을 관리 하기 위해 사용 되는 관리자 숲 속의 계정을 도메인 또는에 워크스테이션 관리자 숲 하려면 관리자 권한이 부여 하지 해야 합니다.

    -   관리자 숲 통해 관리자 권한이 공격자가 또는 악성 참가자에 대 한 감사 로그 지울 기회를 줄이기 위해 오프 라인 프로세스를 통해 밀접 하 게 제어 해야 합니다. 이렇게 하면 프로덕션 관리자 계정 사용 하 여 직원 긴장 자신의 계정에 대 한 제한 하 고 조직에 대 한 위험이 증가 수 없습니다.

    -   The administrative forest should follow the Microsoft Security Compliance Baseline (SCB) configurations for the domain, including strong configurations for authentication protocols.

    -   보안 업데이트로 모든 관리자 숲 호스트 자동으로 업데이트 해야 합니다. 도메인 컨트롤러 유지 관리 작업 중단 위험을 만들 수이 동안 보안 위험이 있이 패치되지 취약성 중요 한 mitigation을 제공 합니다.

        > [!NOTE]
        > 전용된 Windows Server Update Services 인스턴스는 자동으로 업데이트를 승인 하기 구성할 수 있습니다. 자세한 내용은 승인 업데이트가 자동으로 승인 업데이트에 대 한 "설치를" 섹션을 참조 합니다.

-   **워크스테이션 강화** -빌드를 사용 하 여 관리 워크스테이션은 [권한 있는 액세스 워크스테이션](../securing-privileged-access/privileged-access-workstations.md) (3 단계)를 통해 하지만 도메인 멤버 자격으로 변경 생산 환경 대신 관리 숲입니다.

-   **서버와 DC 강화** 모든 도메인 컨트롤러 및 서버 관리 숲 속의 예:

    -   모든 미디어에 대 한 지침을 사용 하 여 유효성을 검사는 확인 [설치 미디어를 새로 소스](http://aka.ms/cleansource)

    -   이것이 생산에서 불가능 하는 경우에 관리자 숲 서버에서 최신 운영 체제 설치 해야 하는지 확인 합니다.

    -   관리자 숲 호스트 보안 업데이트를 자동으로 업데이트 해야 합니다.

        > [!NOTE]
        > Windows Server Update Services 자동으로 업데이트를 승인 하기 구성할 수 있습니다. 자세한 내용은 승인 업데이트가 자동으로 승인 업데이트에 대 한 "설치를" 섹션을 참조 합니다.

    -   보안 초기 구성 시작으로 사용 해야 합니다.

        > [!NOTE]
        > Customers can use the Microsoft Security Compliance Toolkit (SCT) for configuring the baselines on the administrative hosts.

    -   보안 부팅을 공격자 또는 부팅이에 서명 되지 않은 코드 로드 하는 동안 맬웨어가 줄이기 위해 합니다.

        > [!NOTE]
        > Windows 8의 UEFI Unified Extensible Firmware Interface () 활용할 수 있도록이 기능이 도입 되었습니다.

    -   완전 한 볼륨 암호화 컴퓨터 관리 노트북 원격으로 사용 되는 등의 물리적 손상에 대 한 줄일 수 있습니다.

        > [!NOTE]
        > 참조 [BitLocker](https://technet.microsoft.com/en-us/library/dn641993.aspx) 에 대 한 자세한 내용은 합니다.

    -   실제 감염 경로 으로부터 보호 하기 위해 USB 제한 됩니다.

        > [!NOTE]
        > 참조 [제어 읽기 또는 이동식 디바이스 또는 미디어에 대 한 쓰기](https://technet.microsoft.com/en-us/library/cc730808(v=ws.10).aspx) 에 대 한 자세한 내용은 합니다.

    -   실수로 관리자 작업 및 네트워크 공격 으로부터 보호 하기 위해 네트워크 격리 합니다. 호스트 방화벽 꼭 필요한 제외한 모든 들어오는 연결 차단 하 고 모든 아웃 바운드 인터넷 액세스 차단 해야 합니다.

    -   알려진된 위협와 악성 소프트웨어 로부터 보호 하기 위해 악성 코드 방지 합니다.

    -   Windows의 새로운 공격 도입 새 소프트웨어의 설치 도중 방지 하기 위해 공격 surface 분석 합니다.

        > [!NOTE]
        > 와 같은 도구를 사용 하는 [공격 Surface 분석기 (ASA)](https://www.microsoft.com/en-us/download/details.aspx?id=24487) 구성 설정을 호스트에 평가 하 고 공격 도입 소프트웨어 또는 구성을 변경 하 여 사용자의 신원을 확인 하는 데 도움이 됩니다.

-   강화 계정

    -   다중 요소 인증 계정이 두 개를 제외 하 고 관리자 숲 속의 모든 계정에 대해 구성 합니다. 암호 액세스할 수 있도록 기반 다단계 인증 나누기 처리 하는 경우 작동 하는지 관리자 계정이 하나 이상 이어야 합니다. 이 계정 엄격한 물리적 제어 프로세스를 통해 보호 해야 합니다.

    -   계정 다단계 인증을 사용 하도록 설정 하는 새로운 NTLM 해시 계정에 정기적으로 구성 합니다. 비활성화 하 여이 작업을 수행할 수 있습니다 이며 활성화 계정 특성 스마트 카드 로그온 해야 합니다.

        > [!NOTE]
        > 이 관리자 없습니다 되지 사용 하는 경우 해당 계정에와 같은 또는 주말 밤에이 프로세스 시작 해야 하므로이 계정을 사용 하는 진행 중인 작업을 중단할 수 없습니다.

-   검사 컨트롤

    -   검사 컨트롤 관리 숲에 관리자 숲 속의 예외에 경고 디자인 해야 합니다. 제한 된 수 공인된 시나리오와의 활동 이러한 컨트롤을 생산 환경 보다 더 정확 하 게 조정할 수 있습니다.

참여 디자인 및 귀하의 환경에 대 한 ESAE 배포 Microsoft 서비스에 대 한 자세한 내용은 참조 [이 페이지](https://www.microsoft.com/services)합니다.

## <a name="T0E_BM"></a>계층 0 같지
대부분의 조직 도메인 관리자 관리자, Enterprise 관리자와 같은 강력한 계층 0 Active Directory 그룹 구성원을 제어합니다.  많은 조직 간과 효과적으로에 해당 하는 일반적인 active directory 환경에 대 한 권한 다른 그룹 위험 합니다. These groups offer a relatively easy escalation path for an attacker to the same explicit Tier 0 privileges using various different attack methods.

예를 들어 서버 통신사 수 도메인 컨트롤러의 미디어를 백업에 액세스 하 고 모든 자격 증명 미디어 파일의 풀고 권한을 확대 하는 데 사용 합니다.

조직을 제어 하 고 모니터링 모든 포함 하 여 계층 0 그룹 (중첩 된 회원 포함)의 회원:

-   엔터프라이즈 관리

-   도메인 관리

-   스키마 관리자

-   BUILTIN\Administrators

-   계정에서

-   백업 관리자

-   연산자 인쇄

-   서버 연산자

-   도메인 컨트롤러

-   읽기 도메인 컨트롤러

-   그룹 정책 Creators 소유자

-   암호화 연산자

-   분산된 COM 사용자

-   그룹 위임 다른

    > [!NOTE]
    > "다른 위임된 그룹" 조직 디렉터리 작업 효과적인 계층 0 액세스할 수 있는을 관리 하 여 만들 수 있는 그룹 가리킵니다.
