# <a name="mapping-design-requirements-to-forest-design-models"></a>포리스트 디자인 모델을 디자인 요구 사항 매핑

>적용 대상: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

조직에서 대부분의 그룹에는 단일 조직 포리스트에 단일 정보 기술 (IT) 그룹에서 관리 되는 사용자 계정 및 모든 포리스트를 공유 하는 그룹에 대 한 리소스를 포함 하 고 공유할 수 있습니다. 초기 조직 포리스트에 라는 공유이 포리스트에 조직에 대 한 포리스트 디자인 모델의 기반으로 합니다.  

초기 조직 포리스트는 조직에 있는 여러 그룹을 호스팅할 수, 있으므로 포리스트 소유자 당사자 모두가 예상 되는 결과 이해할 수 있도록 각 그룹을 사용 하 여 서비스 수준 계약을 설정 해야 합니다. 합의 서비스 기대치를 설정 하 여 개별 그룹과 포리스트 소유자를 보호 합니다.  

그렇지 않은 경우 조직에서 그룹의 모든 단일 조직 포리스트를 공유할 수 있습니다, 그리고 다른 그룹의 요구를 수용 하도록 포리스트 설계를 확장 해야 합니다. 여기에 자율성 및 격리 및 여부는 네트워크가 제한 된 연결을 위해 필요에 따라 그룹에 적용 되는 디자인 요구 사항을 식별 한 다음 해당를 수용 하기 위해 사용할 수 있는 포리스트 모델 요구 사항입니다. 다음 표에서 자율성, 격리 및 연결 요인에 따라 포리스트 디자인 모델 시나리오를 나열 합니다. 요구 사항에 가장 잘 맞는 포리스트 디자인 시나리오를 확인 한 후에 디자인 사양에 맞게 추가 결정 해야 하는 경우를 결정 합니다.

## <a name="BKMK_1"></a>시나리오 1: 데이터 자치를 위해 기존 포리스트  

기존 조직 포리스트에 조직 구성 단위 (Ou) 그룹을 호스트 하면 데이터 자율성에 대 한 요구 사항이 충족 시킬 수 있습니다. 데이터 자율성을 달성 하기 위해 해당 그룹에서 데이터 관리자에 게는 Ou에 대 한 제어를 위임 합니다. Ou를 사용 하 여 제어를 위임 하는 방법에 대 한 자세한 내용은 참조 하세요. [는 조직 구성 단위 디자인 만들기](../../ad-ds/plan/Creating-an-Organizational-Unit-Design.md)합니다.  
  
## <a name="BKMK_2"></a>시나리오 2: 조직 포리스트 또는 도메인 서비스 자율성을 위한 사용  

조직에서 그룹 요구 사항으로 서비스 자율성을 식별 하는 경우에이 요구 사항에 다시 먼저 고려할 것이 좋습니다. 서비스 자율성을 달성 하기 더 많은 관리 오버 헤드가 및 비용이 조직에 만듭니다. 서비스 자율성을 위한 요구 사항 편의 위해 간단히 아니며이 요구 사항을 충족 하는 데 필요한 비용을 정당화할 수 있는지 확인 합니다.  
  
다음 중 하나를 수행 하 여 서비스 자율성에 대 한 요구를 충족할 수 있습니다.  

- 조직 포리스트를 만드는 중입니다. 사용자, 그룹 및 컴퓨터를 별도 조직 포리스트에 서비스 자율성을 필요한 그룹에 배치 합니다. 포리스트 소유자 해당 그룹에서 개별을 할당 합니다. 그룹을 조직에서 다른 포리스트를 사용 하 여 리소스를 액세스 또는 공유 하는 경우 해당 조직 포리스트와 다른 포리스트 간에 트러스트를 설정할 수 있습니다.  

- 조직 도메인을 사용 합니다. 기존 조직 포리스트에 있는 별도 도메인에서 사용자, 그룹 및 컴퓨터를 배치 합니다. 이 모델에는 전체 서비스 자율성을 대해서가 아니라 서비스 격리 또는 데이터 격리 및 도메인 수준 서비스 자율성만을 위한 제공 합니다.  

조직 도메인을 사용 하는 방법에 대 한 자세한 내용은 참조 하세요. [조직 도메인 포리스트 모델을 사용 하 여](../../ad-ds/plan/../../ad-ds/plan/Using-the-Organizational-Domain-Forest-Model.md)입니다.  

## <a name="BKMK_3"></a>시나리오 3: 조직 포리스트 또는 포리스트를 사용 하 여 서비스 격리  

다음 중 하나를 수행 하 여 서비스 격리에 대 한 요구를 충족할 수 있습니다.  

- 조직 포리스트에 사용합니다. 사용자, 그룹 및 컴퓨터를 별도 조직 포리스트에 서비스 격리 해야 하는 그룹에 배치 합니다. 포리스트 소유자 해당 그룹에서 개별을 할당 합니다. 그룹을 조직에서 다른 포리스트를 사용 하 여 리소스를 액세스 또는 공유 하는 경우 해당 조직 포리스트와 다른 포리스트 간에 트러스트를 설정할 수 있습니다. 그러나 권장 하지는 않습니다이 이렇게 하므로 유니버설 그룹을 통해 리소스 액세스 포리스트 트러스트 시나리오에서 크게 제한 됩니다.  

- 리소스 포리스트를 사용합니다. 기존 조직 포리스트에 사용자 계정을 유지 하는 별도 리소스 포리스트로 리소스 및 서비스 계정을 배치 합니다. 필요한 경우 대체 계정 리소스 포리스트 조직 포리스트에서 사용할 수 없는 경우 리소스 포리스트의 리소스에 액세스 하에서 만들 수 있습니다. 대체 계정 리소스 포리스트로 로그온 하 고 조직 포리스트 다시 온라인 상태가 될 때까지 리소스 제어를 유지 관리 하는 데 필요한 권한이 있어야 합니다.  

   사용자가 일반 사용자 계정을 사용 하 여 포리스트의 리소스에 액세스할 수 있도록 리소스 및 조직 포리스트 간에 트러스트를 설정 합니다. 이 구성에는 사용자가 조직 포리스트에 사용할 수 없는 경우 리소스 포리스트의 대체 계정으로 대체 하도록 허용 하는 동안 중앙 집중화 된 사용자 계정 관리할 수 있습니다.  

서비스 격리에 대 한 고려 사항은 다음과 같습니다.

- 서비스 격리를 위해 만들어진 포리스트 도메인에서 다른 포리스트를 신뢰할 수 있지만 해당 서비스 관리자 그룹의 다른 포리스트의 사용자를에서 포함할 수 없습니다. 다른 포리스트의 사용자는 분리 된 포리스트에서 관리 그룹에 포함 된, 분리 된 포리스트의 보안을 잠재적으로 손상 될 수 있습니다 포리스트의에서 서비스 관리자에 독점적인 제어권 없기 때문입니다.  

- 도메인 컨트롤러를 네트워크에 액세스할 수 있다면 이러한 공격을 받을 수 (예: 서비스 거부 공격) 해당 네트워크에서 악성 소프트웨어 로부터 합니다. 공격의 가능성을 방지 하려면 다음을 수행할 수 있습니다.  

   - 안전한 것으로 간주 되는 네트워크에만 도메인 컨트롤러를 호스트 합니다.  

   - 네트워크 또는 도메인 컨트롤러를 호스팅하는 네트워크에 대 한 액세스를 제한 합니다.  

- 서비스 격리 추가 포리스트의 만들어야합니다. 추가 포리스트를 지원 하도록 인프라를 유지 관리 비용 손실로 인해 사용할 수 없게 하는 조직 포리스트는 리소스에 대 한 액세스를 사용 하 여 관련 비용을 능가 하는지 여부를 평가 합니다.  

## <a name="BKMK_4"></a>시나리오 4: 조직 포리스트 또는 제한 된 액세스 포리스트 데이터를 격리 하기 위해 사용  

다음 중 하나를 수행 하 여 데이터 격리를 얻을 수 있습니다.  

- 조직 포리스트에 사용합니다. 사용자, 그룹 및 컴퓨터를 별도 조직 포리스트에 데이터를 격리 하려면 그룹에 배치 합니다. 포리스트 소유자 해당 그룹에서 개별을 할당 합니다. 그룹을 조직에서 다른 포리스트를 사용 하 여 리소스를 액세스 또는 공유 하는 경우 조직 포리스트와 다른 포리스트 간에 트러스트를 설정 합니다. 새 조직 포리스트에 분류 된 정보에 액세스 해야 하는 사용자만 존재 합니다. 사용자 계정이 트러스트 관계를 사용 하 여 해당 포리스트의 액세스 모두 분류 데이터와 다른 포리스트의 분류 되지 않은 데이터를 사용 하는 경우  

- 제한 된 액세스 포리스트를 사용합니다. 제한 된 데이터 및 해당 데이터에 액세스 하는 데 사용 되는 사용자 계정을 포함 하는 별도 포리스트입니다. 별도 사용자 계정은 네트워크에서 제한 없는 리소스에 액세스 하는 데 사용 되는 기존 조직 포리스트에 유지 됩니다. 제한 된 액세스 포리스트 및 기업에서 다른 포리스트 간에 트러스트 없음 만들어집니다. 다른 포리스트에 연결할 수 없습니다 있도록 별도 실제 네트워크에 포리스트를 배포 하 여 포리스트의 추가로 제한할 수 있습니다. 사용자가 두 워크스테이션 있어야 다른 네트워크에 포리스트를 배포 하는 경우: 제한 된 포리스트에 하 고 네트워크의 nonrestricted 영역에 액세스 하기 위한 하나에 액세스 하기 위한 하나입니다.  

데이터 격리를 위해 포리스트 만들기에 대 한 고려 사항은 다음과 같습니다.  

- 데이터 격리를 위해 만들어진 조직 포리스트 포리스트의 다른 도메인을에서 신뢰할 수 있지만 다른 포리스트의 사용자는 다음 중 하나에 포함 하지 않아야 합니다.  

   - 서비스 관리 또는 서비스 관리자 그룹의 멤버 자격을 관리할 수 있는 그룹을 담당 하는 그룹  

   - 보호 된 데이터를 저장 하는 컴퓨터에 대 한 관리 제어 권한이 있는 그룹  

   - 보호 된 데이터에 액세스할 수 있는 그룹 또는 사용자 개체에 액세스할 수 있는 그룹 개체의 관리를 담당 하는 그룹에 보호 된 데이터  

   이러한 그룹 중 하나에서 다른 포리스트의 사용자는 포함 하는 경우 보호 된 데이터의 유출 하는 분리 된 포리스트 손상으로 다른 포리스트의 손상이 발생할 수 있습니다.  

- 다른 포리스트의 격리 된 포리스트에서 사용자가 다른 포리스트의 리소스에 액세스할 수 있도록 데이터 격리를 위해 만들어진 조직 포리스트를 신뢰 하도록 구성할 수 있습니다. 그러나 격리 된 포리스트에서 사용자를에서 대화형으로 사용 되지에 로그온 해야 트러스팅 포리스트에 있는 워크스테이션. 트러스팅 포리스트의 컴퓨터 악성 소프트웨어에 의해 잠재적으로 손상 될 수 있습니다 및 사용자의 로그온 자격 증명을 캡처할 수 있습니다.  

   > [!NOTE]
   > 트러스팅 포리스트의 서버에서 격리 된 포리스트에서 사용자를 가장 하 고 격리 된 포리스트의 리소스에 액세스를 방지 하려면 포리스트 소유자는 위임 된 인증을 사용 하지 않도록 설정 하거나 제한 된 위임 기능을 사용할 수 있습니다. 위임 된 인증 및 제한 위임에 대 한 자세한 내용은 참조 하세요. [위임 인증](https://go.microsoft.com/fwlink/?LinkId=106614)합니다.  

- 해당 포리스트 외부의 정보에 대 한 사용자 액세스를 제한 하기 위해 조직에서 조직 포리스트와 다른 포리스트 간에 방화벽을 설정 해야 합니다.  

- 별도 포리스트를 만드는 사용 하도록 설정할 데이터 격리를 도메인 컨트롤러에서 분리 된 포리스트 및 컴퓨터에 보호 된 호스트 정보는 네트워크에 액세스할 수 있지만 해당 네트워크의 컴퓨터에서 시작 하는 공격을 받을 않습니다. 공격 위험을 너무 높게 되었거나 지는 공격 또는 보안 위반의 결과가 너무 판단 하는 조직은 네트워크 또는 도메인 컨트롤러를 호스트 하는 네트워크에 대 한 액세스를 제한 해야 하 고 보호 된 데이터를 호스팅하는 컴퓨터 . 액세스를 제한 하는 방화벽 및 인터넷 프로토콜 보안 (IPsec) 같은 기술을 사용 하 여 수행할 수 있습니다. 극단적인 경우에 조직에서는 조직에서 다른 네트워크에 없는 실제 연결을 독립 네트워크에 보호 된 데이터를 유지 하기 위해 선택할 수 있습니다.  

   > [!NOTE]  
   > 제한 된 액세스 포리스트 및 다른 네트워크 사이의 네트워크 연결이 있는 경우 데이터는 다른 네트워크 전송 하도록 제한 된 영역에 대 한 가능성이 있습니다.  

## <a name="BKMK_5"></a>시나리오 5: 조직 포리스트를 사용할지 또는 제한 된 연결에 대 한 방화벽을 다시 구성  

제한 된 연결 요구 사항에 맞게, 다음 중 하나를 수행할 수 있습니다.  

- 기존 조직 포리스트에 사용자 배치 하 고 충분히 Active Directory 트래픽의 통과 허용 하도록 방화벽을 엽니다.  

- 조직 포리스트에 사용 합니다. 별도 조직 포리스트에 사용자, 그룹 및 연결 제한 되는 그룹에 대 한 컴퓨터를 배치 합니다. 포리스트 소유자 해당 그룹에서 개별을 할당 합니다. 조직 포리스트 방화벽의 다른 쪽에서 별도 환경을 제공합니다. 포리스트의는 사용자의 일상 업무를 위해 방화벽을 통해 이동 하지 않아도 되도록 사용자 계정 및 포리스트 내의 관리 되는 리소스를 포함 합니다. 특정 사용자 또는 응용 프로그램이 다른 포리스트에 연결할 방화벽을 통과 하는 기능을 필요로 하는 특별 한 요구 사항이 있을 수 있습니다. 필요한 함수에 대 한 모든 트러스트에 대 한 포함 하 여 방화벽에서 적절 한 인터페이스를 열어 이러한 요구를 개별적으로 해결할 수 있습니다.  

Active Directory Domain Services (AD DS)를 사용 하 여 사용에 대 한 방화벽을 구성 하는 방법에 대 한 자세한 내용은 참조 하세요. [방화벽으로 분리 된 네트워크에서 Active Directory](https://go.microsoft.com/fwlink/?LinkId=37928)합니다.  

## <a name="BKMK_6"></a>시나리오 6: 조직 포리스트 또는 도메인을 사용 하 고 제한 된 연결을 사용 하 여 서비스 자율성에 대 한 방화벽을 다시 구성  

조직에서 그룹 요구 사항으로 서비스 자율성을 식별 하는 경우에이 요구 사항에 다시 먼저 고려할 것이 좋습니다. 서비스 자율성을 달성 하기 더 많은 관리 오버 헤드가 및 비용이 조직에 만듭니다. 서비스 자율성을 위한 요구 사항 편의 위해 간단히 아니며이 요구 사항을 충족 하는 데 필요한 비용을 정당화할 수 있는지 확인 합니다.  

제한 된 연결 문제가 발생 하 고 서비스 자율성을 위한 요구 사항이 있는 경우 다음 중 하나를 수행할 수 있습니다.  

- 조직 포리스트에 사용 합니다. 사용자, 그룹 및 컴퓨터를 별도 조직 포리스트에 서비스 자율성을 필요한 그룹에 배치 합니다. 포리스트 소유자 해당 그룹에서 개별을 할당 합니다. 조직 포리스트 방화벽의 다른 쪽에서 별도 환경을 제공합니다. 포리스트의는 사용자의 일상 업무를 위해 방화벽을 통해 이동 하지 않아도 되도록 사용자 계정 및 포리스트 내의 관리 되는 리소스를 포함 합니다. 특정 사용자 또는 응용 프로그램이 다른 포리스트에 연결할 방화벽을 통과 하는 기능을 필요로 하는 특별 한 요구 사항이 있을 수 있습니다. 필요한 함수에 대 한 모든 트러스트에 대 한 포함 하 여 방화벽에서 적절 한 인터페이스를 열어 이러한 요구를 개별적으로 해결할 수 있습니다.  

- 기존 조직 포리스트에 있는 별도 도메인에서 사용자, 그룹 및 컴퓨터를 배치 합니다. 이 모델에는 전체 서비스 자율성을 대해서가 아니라 서비스 격리 또는 데이터 격리 및 도메인 수준 서비스 자율성만을 위한 제공 합니다. 포리스트에 있는 다른 그룹 포리스트 소유자를 신뢰 하는지 동일한 수준으로 새 도메인의 서비스 관리자를 신뢰 해야 합니다. 이러한 이유로 권장 하지는 않습니다이 방법을 사용 합니다. 조직 도메인을 사용 하는 방법에 대 한 자세한 내용은 참조 하세요. [조직 도메인 포리스트 모델을 사용 하 여](../../ad-ds/plan/../../ad-ds/plan/Using-the-Organizational-Domain-Forest-Model.md)입니다.  

Active Directory 트래픽을 전달할 수 있도록 충분히 방화벽 열기 해야 합니다. AD DS와 함께 사용 하기 위해 방화벽을 구성 하는 방법에 대 한 자세한 내용은 참조 하세요. [방화벽으로 분리 된 네트워크에서 Active Directory](https://go.microsoft.com/fwlink/?LinkId=37928)합니다.  

## <a name="BKMK_7"></a>시나리오 7: 리소스 포리스트를 사용 하 고 제한 된 연결을 사용 하 여 서비스 격리에 대 한 방화벽을 다시 구성  

제한 된 연결 문제가 발생 하 고 서비스 격리에 대 한 요구 사항이 있는 경우 다음 중 하나를 수행할 수 있습니다.  

- 조직 포리스트에 사용 합니다. 사용자, 그룹 및 컴퓨터를 별도 조직 포리스트에 서비스 격리 해야 하는 그룹에 배치 합니다. 포리스트 소유자 해당 그룹에서 개별을 할당 합니다. 조직 포리스트 방화벽의 다른 쪽에서 별도 환경을 제공합니다. 포리스트의는 사용자의 일상 업무를 위해 방화벽을 통해 이동 하지 않아도 되도록 사용자 계정 및 포리스트 내의 관리 되는 리소스를 포함 합니다. 특정 사용자 또는 응용 프로그램이 다른 포리스트에 연결할 방화벽을 통과 하는 기능을 필요로 하는 특별 한 요구 사항이 있을 수 있습니다. 필요한 함수에 대 한 모든 트러스트에 대 한 포함 하 여 방화벽에서 적절 한 인터페이스를 열어 이러한 요구를 개별적으로 해결할 수 있습니다.  

- 리소스 포리스트를 사용 합니다. 기존 조직 포리스트에 사용자 계정을 유지 하는 별도 리소스 포리스트로 리소스 및 서비스 계정을 배치 합니다. 조직 포리스트를 사용할 수 없게 하는 경우 리소스 포리스트에 대 한 액세스를 유지 하기 위해 리소스 포리스트에서 일부 대체 사용자 계정을 만들어야 할 수 있습니다. 대체 계정 리소스 포리스트로 로그온 하 고 조직 포리스트 다시 온라인 상태가 될 때까지 리소스 제어를 유지 관리 하는 데 필요한 권한이 있어야 합니다.  

   사용자가 일반 사용자 계정을 사용 하 여 포리스트의 리소스에 액세스할 수 있도록 리소스 및 조직 포리스트 간에 트러스트를 설정 합니다. 이 구성에는 사용자가 조직 포리스트에 사용할 수 없는 경우 리소스 포리스트의 대체 계정으로 대체 하도록 허용 하는 동안 중앙 집중화 된 사용자 계정 관리할 수 있습니다.  

서비스 격리에 대 한 고려 사항은 다음과 같습니다.  

- 서비스 격리를 위해 만들어진 포리스트 도메인에서 다른 포리스트를 신뢰할 수 있지만 해당 서비스 관리자 그룹의 다른 포리스트의 사용자를에서 포함할 수 없습니다. 다른 포리스트의 사용자는 분리 된 포리스트에서 관리 그룹에 포함 된, 분리 된 포리스트의 보안을 잠재적으로 손상 될 수 있습니다 포리스트의에서 서비스 관리자에 독점적인 제어권 없기 때문입니다.  

- 도메인 컨트롤러를 네트워크에 액세스할 수 있다면 해당 네트워크에서 컴퓨터에서 (예: 서비스 거부 공격) 공격을 받이 됩니다. 공격의 가능성을 방지 하려면 다음을 수행할 수 있습니다.  

   - 안전한 것으로 간주 되는 네트워크에만 도메인 컨트롤러를 호스트 합니다.  

   - 네트워크 또는 도메인 컨트롤러를 호스팅하는 네트워크에 대 한 액세스를 제한 합니다.  

- 서비스 격리 추가 포리스트의 만들어야합니다. 추가 포리스트를 지원 하도록 인프라를 유지 관리 비용 손실로 인해 사용할 수 없게 하는 조직 포리스트는 리소스에 대 한 액세스를 사용 하 여 관련 비용을 능가 하는지 여부를 평가 합니다.  

   특정 사용자 또는 응용 프로그램이 다른 포리스트에 연결할 방화벽을 통과 하는 기능을 필요로 하는 특별 한 요구 사항이 있을 수 있습니다. 필요한 함수에 대 한 모든 트러스트에 대 한 포함 하 여 방화벽에서 적절 한 인터페이스를 열어 이러한 요구를 개별적으로 해결할 수 있습니다.  

AD DS와 함께 사용 하기 위해 방화벽을 구성 하는 방법에 대 한 자세한 내용은 참조 하세요. [방화벽으로 분리 된 네트워크에서 Active Directory](https://go.microsoft.com/fwlink/?LinkId=37928)합니다.
