# <a name="audit-policy-recommendations"></a>감사 정책 권장 사항

>적용 대상: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012, Windows 10, Windows 8.1, Windows 7

이 섹션에서는 Windows 기본 감사 정책 설정, 권장 되는 감사 정책 설정 및 더 적극적으로 권장 사항을 Microsoft에서 워크스테이션 및 서버 제품에 대 한 합니다.  

손상이 감지 하는 데 좋습니다 설정과 함께 여기에 표시 된 SCM 기본 권장 사항은 관리자에 대 한 시작 초기 계획 가이드로만 사용 됩니다. 각 조직 과연 위협가 허용 가능한 위험 허용 오차 및 사용 해야는 감사 정책 범주 또는 하위 범주에 대 한 고유한 결정을 내려야 합니다. 위협에 대 한 자세한 내용은 참조는 [위협 및 대책 가이드](https://technet.microsoft.com/library/hh125921(v=ws.10).aspx)합니다. 신중 하 게 감사 정책이 적용 하지 않고 관리자는 여기서 권장 설정으로 시작 하는 것이 좋습니다 및 수정 하 고 프로덕션 환경에서 구현 하기 전에 테스트 합니다.  

권장 사항을 Microsoft 평균 보안 요구 사항이 높은 수준의 작동 기능을 필요로 하는 컴퓨터를 정의 하는 엔터프라이즈급의 컴퓨터에 적용 됩니다. 요구 사항을 더 적극적으로 고려해 야 하는 더 높은 보안을 필요로 하는 엔터티 정책을 감사 합니다.  

> [!NOTE]  
> Microsoft Windows 기본값은 및에서 기본 권장 사항을 가져왔습니다 합니다 [도구 Microsoft Security Compliance Manager](https://technet.microsoft.com/library/cc677002.aspx)합니다.  

확인된 된 악의적 사용자 또는 맬웨어가에서 활성, 성공적인 공격을 받고에 알려지지 않은 일반적인 보안 컴퓨터에 대 한 다음 기본 감사 정책 설정 하는 것이 좋습니다.

## <a name="set-audit-policy-on-workstations-and-servers"></a>워크스테이션 및 서버 감사 정책 설정  
모든 이벤트 로그 관리 계획에는 워크스테이션 및 서버 모니터링 해야 합니다. 일반적인 실수만 서버나 도메인 컨트롤러를 모니터링 하는 것입니다. 워크스테이션에서 악의적인 해킹을 초기에 자주 발생 하기 때문에 워크스테이션을 모니터링 하지를 가장 하 고 가장 오래 된 정보의 소스를 무시 합니다.  

관리자 신중 하 게 검토 하 고 프로덕션 환경에서 구현 하기 전에 모든 감사 정책을 테스트 해야 합니다.  

## <a name="events-to-monitor"></a>모니터링할 이벤트  
보안 경고를 생성 하기 위한 완벽 한 이벤트 ID는 다음과 같은 특성이 포함 되어야 합니다.  

-   높은 가능성 발생 권한 없는 활동 일  

-   적은 수의 거짓 긍정  

-   조사/법정 응답에서 발생 해야 결과  

두 가지 유형의 이벤트를 모니터링 및 경고를 표시 해야 합니다.  

1.  이러한 이벤트는 한 번 발생 하더라도 권한 없는 활동 일  

2.  예상 및 허용 기준을 초과하는 이벤트 누적  

첫 번째 이벤트의 예는 다음과 같습니다.  

Domain Admins DAs ()는 도메인 컨트롤러가 아닌 컴퓨터에 로그온 할 사용할 수 없음, DA 멤버는 최종 사용자 워크스테이션에 로그온 한 번만 경고를 생성 해야 하 고 조사 합니다. 이 유형의 경고 4964 특수 로그온 감사 이벤트를 사용 하 여 생성할 쉽습니다 (특수 한 그룹에 할당 된 새 로그온). 다른 단일 인스턴스 경고의 예입니다.  

-   서버 A Server B에서 서로 연결할 때의 경고에 연결 되지 해야 합니다.  

-   중요 한 보안 그룹에 일반적인 최종 사용자 계정 추가 예기치 않게 되 면 경고 합니다.  

-   경우에 factory 위치는 직원 밤에 경고를 사용자가 로그온 할 때 자정에 작동 하지 않습니다.  

-   권한이 없는 서비스를 도메인 컨트롤러에 설치 된 경우 경고 합니다.  

-   일반 최종 사용자는 이렇게 하는 것에 대 한 명확한 이유가 없는 가지는 SQL Server에 직접 로그온 하려고 시도 하는 경우를 조사 합니다.  

-   DA 그룹에 멤버가 있고 누군가 자체가 있는 경우 확인이 즉시 합니다.  

두 번째 이벤트의 예는 다음과 같습니다.  

실패 한 로그온 수를 어 베 암호 추측 공격을 나타낼 수 있습니다. 예외적으로 너무 많은 실패 한 로그온에 대 한 경고를 제공 하는 엔터프라이즈에 대 한 악의적인 보안 이벤트를 하기 전에 해당 환경 내에서 실패 한 로그온 정상 수준 먼저 이해 해야 있습니다.  

손상 징후에 대 한 모니터링 하는 경우를 포함 해야 하는 이벤트의 포괄적인 목록은 참조 하십시오 [부록 l: 모니터링할 이벤트](../../../ad-ds/plan/Appendix-L--Events-to-Monitor.md)합니다.  

## <a name="active-directory-objects-and-attributes-to-monitor"></a>Active Directory 개체 및 모니터링 하는 특성  
다음은 계정, 그룹 및를 Active Directory Domain Services 설치를 손상 시키려는 시도 감지할 수 있도록 모니터링 해야 하는 특성입니다.  

-   시스템을 사용 하지 않도록 설정 하거나 (자동으로 다시 보호 수동으로 비활성화 된 경우) 하는 바이러스 백신 및 맬웨어 방지 프로그램 소프트웨어 제거  

-   무단된 변경에 대 한 관리자 계정  

-   (자동으로 제거 계정이 의심 스러운 활동을 완료 하거나 시간을 할당 하는 경우이 만료 된) 권한 있는 계정을 사용 하 여 수행 되는 활동  

-   권한 및 AD DS에 VIP 계정입니다. 변경, 특히 특성 (예: cn, 이름, sAMAccountName, userPrincipalName, 또는 userAccountControl) 계정 탭에서 모니터링 합니다. 계정을 모니터링 하는 것 외에도 작은으로 계정을 관리자에 게 최대한의 집합을 수정할 수 있는 사람을 제한 합니다.  

참조 [부록 l: 모니터링할 이벤트](../../../ad-ds/plan/Appendix-L--Events-to-Monitor.md) 모니터, 해당 중요도 등급 및 이벤트 메시지 요약을 권장 되는 이벤트의 목록은 합니다.  

-   가장 근접 하 게 모니터링 되 고 가장 엄격 하 게 구성 해야 하는 서버를 신속 하 게 식별할 수 있도록 하는 워크 로드를 분류 하 여 그룹 서버  

-   속성 및 다음 AD DS 그룹의 멤버 자격 변경: Enterprise Admins (EA), 도메인 관리자 (DA), 관리자 (BA) 및 스키마 관리자 (SA)  

-   계정을 사용 하도록 설정 하는 것에 대 한 비활성화 된 권한 있는 계정 (예: 기본 제공 관리자 계정에 Active Directory에 구성원 시스템)  

-   계정에 대 한 모든 쓰기를 기록할 관리 계정  

-   기본 제공 보안 구성 마법사 서비스, 레지스트리, 감사 및 서버 공격 표면을 줄이기 위해 방화벽 설정을 구성 합니다. 관리 호스트 전략의 일환으로 점프 서버를 구현 하는 경우이 마법사를 사용 합니다.

## <a name="general-list-of-security-event-id-recommendation-criticalities"></a>보안 이벤트 ID 권장 한 중요도 따라 일반 목록  
이벤트 ID의 모든 권장 사항은 같이 등급을 중요도 함께 제공 됩니다.  

**고가용성:** 높은 중요도 등급을 사용 하 여 이벤트 Id는 항상 즉시 수 경고 및 조사 합니다.  

**보통:** 중간 중요도 등급을 사용 하 여 이벤트 ID는 악의적인 활동을 나타낼 수 있지만 다른 비정상적인 종료 수반 되어야 합니다 (예를 들어 비정상적인 수를 특정 기간 동안, 예기치 않은 발생 또는 발생 한 컴퓨터에서 발생 하는 일반적으로 되지 않을 이벤트를 기록 합니다.). 중간 중요도 이벤트 수도 r를 기준으로 수집 하 고 시간이 지남에 따라 비교할 수 있습니다.  

**낮음:** 및 낮은 중요도 이벤트와 이벤트 ID 주의 차지 하거나 중간 또는 높은 중요도 이벤트와 상관 하지 않으면 경고를 유발 하지 해야 합니다.  

이러한 권장 사항은 관리자에 대 한 초기 지침을 제공 하기 것입니다. 모든 권장 사항은 프로덕션 환경에서 구현 하기 전에 철저 하 게 검토 해야 합니다.  

참조 [부록 l: 모니터링할 이벤트](../../../ad-ds/plan/Appendix-L--Events-to-Monitor.md) 모니터, 해당 중요도 등급 및 이벤트 메시지 요약을 권장 되는 이벤트의 목록에 대 한 합니다.
