# 손상에 대 한 계획

>적용 대상: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

*법률 숫자 1: 잘못 된 수 변화가 없는, 수행 될 때까지 아무도 생각 합니다.* - [10 변경할 수 없는 법률이 보안 관리](https://technet.microsoft.com/library/cc722488.aspx)  
  
대부분의 조직에서 장애 복구 계획 국가별 장애 또는 서비스 컴퓨팅 놓칠 실패에서 복구에 집중 합니다. 그러나 손상 된 고객과 함께 작업 하는 경우 자주 찾을 의도적 손상에서 복구가 없는 경우 해당 장애 복구 계획에 있습니다. 손상의 지적 재산 또는 실제 경계 아닌 (와 같은 논리적 경계 (예: 모든 Active Directory 도메인 또는 모든 서버 폐기)을 활용 하는 의도적 폐기 도용을 때 특히 유용 데이터 센터 폐기)입니다. 조직 손상 발견 될 때 실행할 초기 작업을 정의 하는 문제 대응 계획 수 있지만, 이러한 계획 전체 컴퓨팅 인프라에 영향을 주는 손상에서 복구 하는 단계를 자주 생략 합니다.  
  
Active Directory 사용자, 서버, 워크스테이션 및 응용 프로그램에 대 한 다양 한 id 및 액세스 관리 기능을 제공, 하므로 침입자가 아닌 대상 따라 거의 예외는 합니다. 침입자 도메인 컨트롤러 또는 Active Directory 도메인에 대 한 액세스 권한이 높은 획득을 하는 경우 액세스 하는 액세스 제어를 또는 전체 Active Directory 포리스트도 삭제를 활용할 수 있습니다.  
  
이 문서에는 설명 창과 Active Directory에 대 한 가장 일반적인 공격 중 일부 이며 대책 공격, 하지만 Active Directory의 전체 손상 발생할 경우 복구할 수만 있는지 방법은 줄이기 위해 구현할 수 있습니다. 수 발생 하기 전에 피해를 준비 합니다. 이 섹션에서는 기술 구현 세부 사항이이 문서의 이전 구역 보다 작으면 한 개를 만드는 데 사용할 수 있는 고급 권장 사항을 전체적으로 포괄적인 접근 보호 하 고 조직 관리의 중요 한 비즈니스 및 IT 자산 합니다.  
  
Resisted가 있는지 여부를 인프라 공격 적이 없는는 것을 침해를 시도 하거나 공격 올 며 된는 하면 공격을 받게 됩니다 다시 불가피 현실에 대 한 계획 해야 손상 완벽 하 게 됩니다. 공격을 방지 하기 위해 수도 있지만 중요 한 침해 또는 대규모로 손상을 방지 하려면 작업 맞는지 수 있습니다. 밀접 하 게 조직 마다 해당 기존 위험 관리 프로그램을 평가 하 고 방지, 검색, 포함, 및 복구에서 고르게 분배 된 투자 하 여 자신의 전반적인 수준의 취약점을 단축할 수 있도록 필요 조정 해야 합니다.  
  
효과적인 방어 여전히 사용자 및 회사가 인프라 및 응용 프로그램에 종속 된 서비스를 제공 하는 동안를 만들려면 될 수 있습니다를 방지 하기 위해 독창적인 방법을 고려해 야 할 검색, 및 손상 해당 환경에 포함을에서 다음 복구 손상 됩니다. 방법과이 문서에서 권장 사항을 도움이 되지 않을 수 있습니다 손상 된 Active Directory 설치를 복구 하지만 다음 하나를 보호 하는 데 도움이 됩니다.  
  
Active Directory 포리스트 복구 하기 위한 권장 사항에 표시 되는 [Windows Server 2012: Active Directory 포리스트 복구 계획](https://www.microsoft.com/download/details.aspx?id=16506)합니다. 새 환경 완전히 손상 되지 않도록 방지 하기 위해 수 있지만 복구 하 고 다시 환경 제어 도구를 사용할 수 없는 경우에 갖습니다.

일반적으로 잘 허용 되는 시스템, 관리자, 루트 또는 운영 체제에 상관 없이 컴퓨터에 해당 하는 액세스 침입자 부여 되는 경우 해당 컴퓨터 더 이상으로 간주 될 수 얼마나 많은 노력 "정리"에 대 한 시간에 관계 없이 신뢰할 수 있는 시스템이 지원 됩니다. 그러나 Active Directory 도메인 모든 도메인 컨트롤러 호스트 동일한 AD DS 데이터베이스의 복제를 사용 합니다. 단일 도메인 컨트롤러 손상 되는 AD DS 데이터베이스 수정 하는 경우 해당 수정 마다 다른 도메인 컨트롤러의 도메인에서 한는 수정이 완료 되, 포리스트 파티션을 따라 복제 합니다. 모든 도메인 컨트롤러 포리스트에 다시 설치 하는 경우에 단순히 다시 설치 하는 AD DS 데이터베이스 상주 하는 호스트 합니다. Active Directory에 부정적인 영향 새로 설치 된 도메인 컨트롤러는 년간 실행 된 도메인 컨트롤러를 복제 쉽게 복제 됩니다.

모든 패싯 인프라를 지켜야 수도 있지만 침입자 하기만 충분 한 결함 원하는 목표를 달성에 액세스 하기 위한 조직의 방어에서 찾을 수 있습니다. 환경이 비교적 간단 하 고 기본적인, 과거 잘 관리, 다음이 문서 앞부분에 제공 되는 권장 구현 경우 직접적인 제안을 수도 있습니다.  
  
그러나가 발견 하는 이전, 크고 더 복잡 한이 문서에 있는 권장 사항을 부적합 또는 불가능해 구현 될 것은 환경, 가능성이 높습니다. 것이 사후 인프라 보안 시작 신선 하 고 방지 공격 및 손상 되는 환경을 생성 하는 것 보다 훨씬 더 어렵습니다. 하지만 설명한 대로 전체 Active Directory 포리스트 작성 해야 없는 작은 시작 됩니다. 이러한 이유로 보다 집중, Active Directory 포리스트 보안 대상된 접근 하는 것이 좋습니다.  
  
중점 "끊어진"에 대해 모든 문제를 해결 하 려 하는 대신는 우선 순위를 지정할 방법 비즈니스에 및 인프라에 가장 중요 한 기반 하는 것이 좋습니다. 오래 된, 잘못 구성 된 시스템 및 응용 프로그램으로 채워진 환경을 개선 하는 대신 새 작은, 보안 환경을 올 수 안전 하 게 포트 사용자, 시스템 및 가장 중요 한 정보를 만드는 것이 좋습니다 사용자 비즈니스 합니다.  
  
이 섹션에서는 "수명 보트" 또는 "안전한 셀" 코어 비즈니스 인프라에 대 한 다른 이름으로 사용 되는 순수 AD DS 포리스트를 만들 수 있는 방법에 설명 합니다. 순수 포리스트는 일반적으로 크기 및 범위를 제한 되어 있고 [에 설명 된 원칙 알림과 현재 운영 체제, 응용 프로그램을 사용 하 여 기본 제공 되는 단순히 새로 설치 된 Active Directory 포리스트 Active Directory 공격 줄이기 표면형](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md)합니다.  
  
권장 되는 구성 설정을 새로 작성 포리스트에서를 구현 하 여 보안 설정 및 테스트 방법, 기초부터 기본 제공 되는 AD DS 설치 만들 수 있으며 지원 레거시 시스템 함께 표시 하는 문제를 줄일 수 및 응용 프로그램입니다. 몇 가지 일반적인 원칙 및 셀을 만들 수 "안전한" 위치에 유지할 수 있습니다에 가장 중요 한 지침에 따라 해야 디자인 및 구현 초기 AD DS 설치 중에 대 한 자세한 내용은이 문서의 범위 밖에, 자산 합니다. 이 지침은 다음과 같습니다.  
  
1.  분리 하 고 중요 한 자산 보호에 대 한 원칙을 식별 합니다.  
  
2.  제한 된, 위험 기반 마이그레이션 계획을 정의 합니다.  
  
3.  필요한 경우 "nonmigratory" 마이그레이션을 활용 합니다.  
  
4.  구현 "창의적인 폐기."  
  
5.  레거시 시스템과 응용 프로그램을 파악 합니다.  
  
6.  최종 사용자에 대 한 보안을 간소화할 수 있습니다.  
  
### 분리 하 고 중요 한 자산 보호에 대 한 원칙 식별  

집 중요 한 자산을 만드는 기본적인 환경의 특성 크게 달라질 수 있습니다. 예를 들어 VIP 사용자 및 해당 사용자가 액세스할 수 있는 중요 한 데이터 마이그레이션에 기본적인 포리스트를 만들 수 있습니다. 뿐만 아니라 VIP 마이그레이션할 순수 포리스트를 만들 수 있습니다 사용자에 게 하지만 보안 관리 계정을 만들고 [Active Directory 공격 영역을 줄일](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md) 수 있는 설명 원칙 구현 관리 포리스트도 구현 하는 순수 포리스트에서 레거시 포리스트 관리에 사용할 수 있는 호스트 하 고 있습니다. VIP 계정, 권한이 있는 계정 및 시스템 보안 수준이 낮은에서 분리의 유일한 목표와 Active Directory 인증서 서비스 (AD CS)를 실행 하는 서버와 같은 추가 보안을 요구 하는 "회사" 포리스트 구현할 수 있습니다. 포리스트 합니다. 마지막으로, 결국 소모전 () 통해 레거시 포리스트를 해제할 수 있도록 시스템, 응용 프로그램 및 데이터를 새로운 모든 사용자에 대해 사실상 위치 되는 순수 포리스트를 구현할 수 있습니다.  
  
에 관계 없이 계획에서 이러한 원칙에 따라 해야 여부 순수 포리스트 포함 안 되는 사용자와의 연결이 시스템 또는 것 보다 적극적으로 마이그레이션 위한 기본 양식,:  
  
1.  레거시 포리스트 손상 되었을 것으로 가정 합니다.  
  
2.  순수 포리스트 신뢰 레거시 환경 구성할 수는 있지만 레거시 포리스트 신뢰 초기 환경을 구성 하지 마십시오.  
  
3.  마이그레이션 안 함 사용자 계정이 나 그룹 레거시 포리스트 로부터 초기 환경에는 계정 그룹 구성원, SID 기록 또는 기타 특성이 수 악의적으로 수정한 가능성이 있는 경우. 대신 "nonmigratory" 방법을 사용 하 여 순수 포리스트 채울 수 있습니다. (Nonmigratory 방법이 설명 되어 나중에이 섹션입니다.)  
  
4.  순수 포리스트를 레거시 포리스트에서 컴퓨터를 마이그레이션할을 실행 합니다. 순수 포리스트에서 새로 설치 된 서버를 구현, 새로 설치 된 서버에서 응용 프로그램을 설치 하 고 새로 설치한 시스템에 응용 프로그램 데이터를 마이그레이션할 합니다. 파일 서버에 대 한 데이터를 새로 설치 된 서버를 복사, 새 포리스트의 사용자 및 그룹을 사용 하 여 Acl을 설정 하 고 유사한 방식으로 인쇄 서버를 만듭니다.  
  
5.  이전 운영 체제 또는 순수 포리스트에서 응용 프로그램의 설치를 허용 하지 않습니다. 응용 프로그램 업데이트 하 고 새로 설치한 수 없는 경우, 레거시 포리스트에 두고 창의적인 폐기 응용 프로그램의 기능을 대체 하는 것이 좋습니다.  
  
### 제한 된, 위험 기반 마이그레이션 계획 정의  
제한 된 만들기, 위험 기반 마이그레이션 계획 것을 의미 하는 경우 조직 노출 되는 위험 노출 된 정도에 따라 마이그레이션 대상 나타내야 사용자, 응용 프로그램 및 데이터에 순수 포리스트로 마이그레이션할 결정할 때의 사용자 또는 시스템 손상 됩니다. 순수 포리스트에 계정이 있는 침입자가 대상이 될 가능성이 VIP 사용자를 배치 해야 합니다. 새로 작성 된 서버 순수 포리스트에서 중요 한 비즈니스 기능을 설치 해야 하며 매우 중요 한 데이터를 이동 해야 하는 응용 프로그램 서버에서 순수 포리스트 보호 합니다.  
  
가장 중요 한 비즈니스 사용자의 명확한 그림 아직 없는 경우 비즈니스 단위를 식별 하 시스템, 응용 프로그램 및 Active Directory 환경에서 데이터 작업 합니다. 중요 한 응용 프로그램을 실행 하는 또는 중요 한 데이터는 저장 된 서버 해야 처럼 작동 하는 비즈니스에 필요한 모든 응용 프로그램 식별 해야 합니다. 사용자와 계속 작동 하 여 조직에 필요한 리소스를 식별 하 여 자신의 노력을 자산의 자연스럽 게 우선 순위가 지정 된 모음을 만듭니다.  
  
### "Nonmigratory" 마이그레이션 활용  
손상 된 또는 레거시 Active Directory 설치에서 레거시 데이터 및 개체를 새 컴퓨터로 마이그레이션를 선호 하는 의심 기술적 실행 마이그레이션 방법 고려 환경 손상 되었음을 알 수 있는지 여부 "마이그레이션" 개체입니다.  
  
### 사용자 계정  
기존 Active Directory 마이그레이션의 포리스트에서 다른, 사용자 개체에 SIDHistory (SID 기록) 특성 사용자의 SID 및 사용자가 레거시 포리스트에 소속 된 그룹의 Sid 저장할 사용 됩니다. 새 포리스트로 마이그레이션된 사용자 계정 레거시 포리스트의 자원에 액세스 하는 경우 SID 기록에서 Sid는 전의 액세스 계정이 마이그레이션된 리소스에 액세스할 수 있도록 액세스 토큰을 만드는 데 사용 됩니다.  
  
하지만 SID 기록가 검증 일부 환경에서 문제를 일으키는 유지 현재 및 이전 Sid와 사용자의 액세스 토큰을 채우는 때문에 토큰 크기 증가에서 발생할 수 있습니다. 토큰 크기 증가 Sid 사용자의 액세스 토큰에 저장 해야 하는 번호를 사용 하는 문제가 발생 하거나 토큰에서 사용 가능한 공간의 크기를 초과 합니다.  
  
토큰 크기를 제한 된 범위를 늘릴 수 있습니다를 있지만 토큰 크기 증가 하는 최고의 솔루션 그룹 구성원을 SID 기록 또는 둘 다 제거 방법 미리 정규화가 있는지 여부를 사용자 계정과 관련 된 Sid 개수를 줄이는 것입니다. 토큰 크기 증가 하는 방법에 대 한 자세한 내용은 [MaxTokenSize 및 Kerberos 토큰 볼록](http://blogs.technet.com/b/shanecothran/archive/2010/07/16/maxtokensize-and-kerberos-token-bloat.aspx)를 참조 하십시오.  
  
레거시 환경에서 사용자를 마이그레이션 대신 (특히 하나는 그룹 멤버 자격 및 SID 기록을 생길 수) SID 기록을 사용 하 여 "사용자를 마이그레이션할", SID 기록을 수행 하지 않고 메타 응용 프로그램을 활용 하 여 것이 좋습니다. 에 새 포리스트 합니다. 새 포리스트에서 사용자 계정을 만들 때 계정을 레거시 포리스트에서 해당 계정에 매핑할 메타 응용 프로그램을 사용할 수 있습니다.  
  
레거시 포리스트의 리소스에 새 사용자 계정 액세스를 제공 하려면 메타 도구를 사용 하 여 자원 그룹에 있는 사용자의 레거시 계정 액세스를 부여 된 식별 한 다음 해당 그룹에 사용자의 새 계정 추가 수 있습니다. 레거시 포리스트에서 그룹 전략을 따라 리소스 액세스에 대 한 도메인 로컬 그룹을 만들거나 기존 그룹 리소스 그룹에 추가 하려면 새 계정을 포함 하도록 도메인 로컬 그룹으로 변환 해야 할 수 있습니다. 먼저 가장 중요 한 응용 프로그램 및 데이터에 초점을 하 고 (있거나 없는 SID 기록을) 새 환경으로 마이그레이션 레거시 환경에서 소요 작업량을 제한할 수 있습니다.  
  

  
### 워크스테이션과 서버  
하나의 Active Directory에서 기존 마이그레이션의 포리스트 다른, 마이그레이션 컴퓨터에는 자주 비교적 간단한 마이그레이션은 사용자, 그룹 및 응용 프로그램에 비해입니다. 컴퓨터 역할에 따라 새 포리스트로 마이그레이션할 수 처럼 간단한 항목 일 이전 도메인 가입을 취소 하 고 새 참가 합니다. 그러나 계정을 마이그레이션하는 컴퓨터 그대로 순수 포리스트 비교에 새 환경 만들기의 용도입니다. (마이그레이션 잠재적으로 손상, 잘못 구성 되었습니다, 또는 오래 된) 컴퓨터를 새 포리스트 계정, 대신 새로 설치 해야 워크스테이션과 서버 새 환경에 있습니다. 마이그레이션할 수 있습니다 데이터 순수 포리스트 시스템과 레거시 포리스트에서 시스템 하지만 아닌 시스템에서 해당 하우스 데이터입니다.  
  
### Applications  

응용 프로그램을 다른 포리스트에서 모든 마이그레이션에서 가장 큰 과제 프레젠테이션할 수 있지만 "nonmigratory" 마이그레이션의 경우 적용 해야 하는 가장 기본적인 원칙 중 하나에서 순수 포리스트 응용 프로그램을 현재 되어야 함을 지원 하 고 새로 설치 합니다. 가능한 경우 오래 된 포리스트에서 응용 프로그램 인스턴스에서 데이터를 마이그레이션할 수 있습니다. 응용 프로그램 "다시 만들 수 없습니다" 순수 포리스트의 경우에는 다음 섹션에 설명 된 대로 창의적인 삭제 또는 레거시 응용 프로그램의 격리 등 접근을 고려해 야 합니다.  
  
### 창의적인 폐기 구현  
창의적인 폐기 이전 수주일 되어도 만든 economic 개발에 설명 하는 경제학 용어입니다. 최근 몇 년 동안 용어 정보 기술에 적용 되었습니다. 일반적으로 참조 하는 메커니즘으로는 이전 인프라 제거 하 여, 업그레이드,가 아닌 하지만 완전히 새로운 것으로 바꿈으로써 합니다. Cio 및 선임 IT 임원 2011 [Gartner 심포지움 ITXPO](http://www.gartner.com/technology/symposium/orlando/) 비용 절감 및 증가 주요 테마 중 하나로 창의적인 폐기 효율성에 나와 있습니다. 향상 된 보안 기능 프로세스의 자연 outgrowth로 만들 수 있습니다.  

예를 들어 조직에 다양 한 수준의 modernity 및 타사 공급 업체 지원와 유사한 기능을 수행 하는 다른 응용 프로그램을 사용 하는 여러 비즈니스 단위가 구성 될 수 있습니다. 지금까지 IT 각 사업부 응용 프로그램을 개별적으로 유지 관리를 담당 수 있으며 통합 노력으로 구성 됩니다 응용 프로그램을 제공 하는 가장 좋은 기능 파악 시도 하 고 다음에 데이터를 마이그레이션 다른 응용 프로그램입니다.  
  
창의적인 폐기를 유지 하지 않고 오래 되었거나 중복 응용 프로그램을에서 이전 바꾸기, 새로운 응용 프로그램에 데이터를 마이그레이션할 완전히 새로운 응용 프로그램을 구현 한 이전 응용 프로그램 및 실행 하는 시스템을 해제 합니다. 경우에 따라 자신의 구조에 새 응용 프로그램을 배포 하 여 창의적인 폐기 레거시 응용 프로그램을 구현할 수 있지만 클라우드 기반 솔루션에 응용 프로그램을 대신 이식 고려해 야 가능한 경우 합니다.  
  
레거시 사내 응용 프로그램을 클라우드 기반 응용 프로그램을 배포 하 여 유지 관리 노력 하 고 비용, 뿐만 아니라 줄일 있지만 레거시 시스템 및 보안 문제를 제공 하는 응용 프로그램을 제거 하 여 조직의 공격 줄이기 활용할 수 있도록 침입자에 대 한 이 방법은 레거시 대상 인프라에서 동시에 제거 하는 동안 원하는 기능을 구하는 조직에 대 한 보다 빠르게를 제공 합니다. 창의적인 폐기 원칙 모든 IT 자산에 적용 되지 않음, 수도 있지만 동시에 강력한, 보안, 클라우드 기반 응용 프로그램을 배포 하는 동안 레거시 시스템과 응용 프로그램을 제거 하는 자주 사용 가능한 옵션이 제공 합니다.  
  
### 레거시 시스템 및 응용 프로그램 격리  
업무용 사용자와 시스템 기본적인, 보안 환경으로 마이그레이션하는 자연 outgrowth 덜 중요 한 정보 및 시스템 레거시 포리스트 포함 되어 있음을입니다. 레거시 시스템 및 보안 수준 낮음 환경에 남아 있는 응용 프로그램이 프레젠테이션 손상 높은 위험 수도 있지만 손상 감소 심각도 나타내도록 합니다. 위치 이동 하 고 중요 한 비즈니스 자산 현대화를 효과적인 관리 배포 및 레거시 설정 및 프로토콜에 맞게 필요 없이 하는 동안 모니터링에 집중할 수 있습니다.  
  
순수 포리스트 하 여 중요 한 데이터를 기반으로 한 옵션을 평가할 수 있습니다 추가 레거시 시스템과 "주" AD DS 포리스트에서 응용 프로그램을 차단 합니다. 하나의 응용 프로그램 및 실행 중인 서버를 바꾸려면 창의적인 폐기를 구현 수 있지만 다른 경우에 보안이 가장 취약 시스템과 응용 프로그램의 추가 격리를 고려 될 수 있습니다. 대체 옵션 없음 이미 없는 시스템을 지원 하도록 만들 소수의 사용자가 사용 되는 작은 도메인으로 마이그레이션할 수 LAN Manager 해시와 같은 레거시 자격 증명 필요 하지만 응용 프로그램 예를 들어 있습니다.  
  
이러한 시스템 레거시 설정 구현을 강제로 도메인을 제거 하 여 현재 운영 체제 및 응용 프로그램을 지원 하도록 구성 하 여 도메인의 보안 이후에 늘릴 수 있습니다. 하지만 레거시 시스템과 가능할 때마다 응용 프로그램을 제거 하는 것이 좋습니다. 폐기 없는 경우 단순히 레거시 모집단의 작은 세그먼트에 대 한 가능한 경우, 레거시 설치의 나머지에서를 향상 시키는 수행할 수 있습니다 별도 도메인 (또는 포리스트)로 분리 합니다.  
  
### 최종 사용자에 대 한 보안 단순화  
대부분의 조직에서 자주 조직 외부의 해당 역할의 성격으로 인해 가장 중요 한 정보에 액세스할 수 있는 사용자는 최소 양의 시간 학습 복잡 한 액세스 제한 및 컨트롤을 사용할 수 있습니다. 보안을 투명 하 게 하 고 간소화 원리는 사용자가 있는 컨트롤에 구현 하 여 사용 가능한 단순 하에 집중 해야 조직의 모든 사용자에 대 한 포괄적인 보안 교육 프로그램 해야 하지만 준수 합니다.  
  
예를 들어 있는 임원 및 기타 Vip 하는 데 필요한 보안 워크스테이션 중요 한 데이터 및 시스템, 액세스를 사용 하 여 다른 장치를 사용 하 여 덜 중요 한 데이터에 액세스할 수 있도록 정책을 정의할 수 있습니다. 사용자가 기억할 수 있도록 하는 간단한 원칙 개이지만 다양 한 방법을 적용할 수 있도록 백 엔드 컨트롤을 구현할 수 있습니다.
