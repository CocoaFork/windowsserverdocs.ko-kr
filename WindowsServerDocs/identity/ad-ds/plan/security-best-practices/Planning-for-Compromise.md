# <a name="planning-for-compromise"></a>손상 계획

>적용 대상: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

*법률 숫자 1: 잘못 된 항목에 발생할 수 있습니다, 완료 될 때까지 아무도 한다고 생각 합니다.* - [10 immutable Laws of Security 관리](https://technet.microsoft.com/library/cc722488.aspx)  
  
많은 조직에 재해 복구 계획에서 컴퓨팅 서비스의 손실 될 수 있는 오류 또는 지역 재해 복구에 집중 합니다. 그러나 손상 된 고객에서 작업할 때 종종 알게 의도적인 손상 으로부터 복구 absent는 재해 복구 계획에 해당 합니다. 손상 도난 지적 재산 또는 의도적으로 소멸 (예: 물리적 경계 대신 논리적 경계 (예: 소멸 모든 Active Directory 도메인의 모든 서버)를 활용 하는 경우이 특히 데이터 센터 소멸)입니다. 조직 손상이 발견 될 때 적용할 초기 작업을 정의 하는 인시던트 응답 계획을 가질 수 있지만 이러한 요금제는 종종 전체 컴퓨팅 인프라에 영향을 주는 손상 으로부터 복구 하는 단계를 생략 합니다.  
  
Active Directory 사용자, 서버, 워크스테이션 및 응용 프로그램에 대 한 다양 한 id 및 액세스 관리 기능을 제공 하므로 공격자에 의해 하지 대상 가변 됩니다. 공격자는 Active Directory 도메인 또는 도메인 컨트롤러에 대 한 높은 액세스에 액세스 하는 액세스, 컨트롤 또는 전체 Active Directory 포리스트를 파괴도 활용할 수 있습니다.  
  
이 문서는 가장 일반적인 공격 Windows 및 Active Directory의 일부에 대해 설명 했습니다 고 대책 하면 공격 노출 되지만 확실 한 방법은 Active Directory의 전체 손상이 발생 한 경우 복구를 위해 구현할 수 있습니다를 발생 하기 전에 손상에 대 한 준비 합니다. 이 섹션에서는 기술 구현 세부 사항이이 문서의 이전 섹션 보다 작은 중점을 둡니다 및 더 높은 수준의 권장 사항을 만드는 데 사용할 수 있는 보안 및 조직을 관리 하는 전체적으로 포괄적인 접근 방법을 중요 한 비즈니스 및 IT 자산입니다.  
  
Resisted에 있는지 여부를 인프라 되지 공격을 침해 시도 공격 올 있으며 또는 되었습니다는 하면가 로부터 공격 대상이 될 다시 필연적인 현실에 대비해 야 완벽 하 게 손상 되 면 합니다. 공격을 방지할 수 있지만 실제로 중요 한 위반 또는 도매 손상 방지 못할 것입니다. 밀접 하 게 모든 조직이 자신의 기존 위험 관리 프로그램을 평가 하 고 방지, 검색, 포함 및 복구에 분산 된 투자 하 여 해당 전체 수준의 취약점을 줄이는 데 필요한 대로 조정 해야 합니다.  
  
서비스 사용자와 인프라 및 응용 프로그램에 의존 하는 기업에 제공 하면서 효율적인 방어를 만들려면 수 있습니다를 방지 하는 새로운 방법을 고려 검색 및 사용자 환경에서 손상 포함 및 하에서 복구 손상 됩니다. 방법 및이 문서의 권장 사항이 도움이 되지 않을 수 있습니다 공격에 노출 된 Active Directory 설치를 복구 하지만 다음 번째 앱을 보호 하는 데 도움이 됩니다.  
  
Active Directory 포리스트를 복구 하는 것에 대 한 권장 사항은 [Windows Server 2012: Active Directory 포리스트 복구 계획](https://www.microsoft.com/download/details.aspx?id=16506)합니다. 새로운 환경을 완전히 손상 되지 않도록 방지 하는 일을 할 수 있지만 도구를 복구 하 고 다시 환경 제어할 수 없는 경우에 해야 합니다.  
  
## <a name="rethinking-the-approach"></a>방법을 다시 살펴보기  
*법률 숫자 8: 네트워크 방어의 어려움은 해당 복잡성에 정비례 합니다.* - [10 immutable Laws of Security 관리](https://technet.microsoft.com/library/cc722488.aspx)  
  
일반적으로 잘 허용 되는 공격자가 시스템, 관리자, 루트 또는 운영 체제에 관계 없이 컴퓨터에 동일한 액세스 권한을 획득 하는 경우 해당 컴퓨터 이상 간주할 수 "정리"에 대 한 얼마나 많은 작업에 관계 없이 신뢰할 수 있는 것은 시스템입니다. Active Directory 다르지 않습니다. 공격자가 도메인 컨트롤러나 Active Directory에서 높은 권한이 있는 계정에 대 한 권한 있는 액세스 공격자에 게 사용 하면 모든 수정 하거나 양호한 백업 레코드 경우가 아니라면를 가져온 경우 디렉터리를 되지 복원할 수 있습니다를 완전히 신뢰할 수 있는 상태입니다.  
  
구성원 서버 또는 워크스테이션은 손상 하 고 공격자에 의해 변경 하는 경우 컴퓨터가 더 이상 신뢰할 수 있는, 인접 한 손상 되지 않은 서버 및 워크스테이션 수 수 있지만 신뢰할 수 있는 합니다. 한 컴퓨터의 모든 컴퓨터가 손상 된 것을 의미 하지 않습니다.  
  
그러나 Active Directory 도메인에서 시에 모든 도메인 컨트롤러에서 동일한 AD DS 데이터베이스의 복제본을 호스트 하 합니다. 단일 도메인 컨트롤러가 손상 되는 공격자가 AD DS 데이터베이스를 수정 하는 경우 이러한 수정 사항은 다른 모든 도메인 컨트롤러를 도메인에는 수정 사항이 포리스트의 파티션에 따라에 복제 합니다. 포리스트의 모든 도메인 컨트롤러를 다시 설치 하는 경우에 단순히 다시 설치 하는 AD DS 데이터베이스 상주 하는 호스트입니다. Active Directory에 부정적인 영향 오랫동안 실행 된 도메인 컨트롤러에 복제는 쉽게 새로 설치 된 도메인 컨트롤러에 복제 됩니다.  
  
손상 된 환경 평가 일반적으로 알게는 첫 번째 위반 "이벤트" 한 것으로 알려져 된 어떤 실제로 트리거된 주 후 월 또는 년도 공격자가 환경을 손상 처음 후 합니다. 공격자가 일반적으로 긴 높은 권한이 있는 계정의 자격 증명 위반을 발견 되었습니다. 해당 계정이 디렉터리, 도메인 컨트롤러, 구성원 서버, 워크스테이션을 손상 시킬를 활용 하는 이러한 전에 가져와 비-Windows 연결 시스템입니다.  
  
이러한 결과 Verizon의 2012 데이터 위반 조사 하는 보고서는 상태에서 여러 결과와 일치 합니다.  
  
-   외부 에이전트에서 형태소가 분석 데이터 침해 98%  
  
-   데이터 침해의 85%를 검색 하는 데 몇 주 이상 걸렸습니다.  
  
-   92%의 인시던트를 제 3 자에 의해 검색 되었습니다 및  
  
-   위반의 97% 단순 하지만 피할 수 되었거나 중간 컨트롤입니다.  
  
앞에서 설명한 정도 손상 효과적으로 가능 없는 및 "평면화 하 고 다시 작성" 표준 통지 손상 된 모든 시스템 아닙니다 단순히 가능 또는 가능한 Active Directory 손상 되었거나 제거 된 경우. 알려진 정상 상태로 복원도 환경을 처음부터 손상 될 수 있는 결함 제거 하지 않습니다.  
  
인프라의 모든 측면을 반드시 지켜야 합니다, 있지만 공격자가 원하는 목표를 달성 하려면 약해집니다에서 충분 한 결함을 찾을 수만 필요 합니다. 환경은 비교적 간단 하 고 인위적인, 지금까지 잘 관리 되는 다음이 문서의 앞부분에서 제공한 권장 사항을 구현 하는 경우에 이해 하기 제안 수 있습니다.  
  
그러나 개는 오래 된, 크고 더 복잡 한이 문서의 권장 사항이 되도록 알아내기는 어렵거나도 구현 하는 환경, 가능성이 높아집니다. 훨씬 새로 시작 하 고 공격 및 손상에 사용 되는 환경을 생성 하는 것 보다 사후 인프라를 보호 하는 것이 어렵습니다. 이지만 설명한 대로 없는 작은 착수할 전체 Active Directory 포리스트를 다시 작성 해야 합니다. 이러한 이유로 중점을 둔, 대상된 방법은 Active Directory 포리스트를 보호 하는 것이 좋습니다.  
  
에 집중 하 고 "중단" 되는 작업을 모두 수정 하려고 하는 대신 우선는 접근 방식을 인프라 및 비즈니스에 가장 중요 한 사항에 기반 하는 것이 좋습니다. 오래 된, 잘못 구성 된 시스템 및 응용 프로그램으로 채워진 환경 재구성 하는 대신에 이식할 수 있습니다 안전 하 게 사용자, 시스템 및 정보에 가장 중요 한 새 소규모의 보안 환경을 만드는 것이 좋습니다. 하 비즈니스입니다.  
  
이 섹션에서는 "수명 재벌" 또는 "보안 셀" 핵심 비즈니스 인프라에 사용 되는 기본적인 AD DS 포리스트를 만들 수 있습니다 하는 방법을 설명 합니다. 순수 포리스트는 일반적으로 크기와 범위를 제한 된 및 현재 운영 체제, 응용 프로그램을 사용 하 여 및에 설명 된 원칙을 사용 하 여 빌드되는 단순히 새로 설치 된 Active Directory 포리스트 [활성 감소 Directory 공격](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md)합니다.  
  
권장 되는 구성 설정을 새로 빌드된 포리스트에 구현 하 여 처음부터 보안 설정 및 사례를 사용 하 여 기본 제공 되는 AD DS 설치를 만들 수 있습니다 및 레거시 시스템을 지 원하는 수반 되는 문제를 줄일 수 있습니다 및 응용 프로그램입니다. 몇 가지 일반적인 원칙 및 셀을 만들어서 "보안"를 유지할 수 있습니다에 가장 중요 한 지침 따라야 디자인 및 구현 기본적인 AD DS 설치에 대 한 자세한 지침은이 문서의 범위를 벗어나는, 하는 동안 자산입니다. 이러한 지침은 아래와 같습니다.  
  
1.  분리 및 중요 한 자산 보호에 대 한 원칙을 식별 합니다.  
  
2.  제한, 위험 기반 마이그레이션 계획을 정의 합니다.  
  
3.  필요한 경우 "nonmigratory" 마이그레이션을 활용 합니다.  
  
4.  "Creative 소멸 합니다."를 구현 합니다.  
  
5.  레거시 시스템 및 응용 프로그램을 격리 합니다.  
  
6.  최종 사용자에 대 한 보안을 간소화 합니다.  
  
### <a name="identifying-principles-for-segregating-and-securing-critical-assets"></a>분리 및 중요 한 자산 보호에 대 한 원칙을 식별 합니다.  

사용자가 만든 집 중요 한 자산에 초기 환경의 특징은 광범위 하 게 달라질 수 있습니다. 예를 들어, VIP 사용자 및 해당 사용자만 액세스할 수 있는 중요 한 데이터 마이그레이션에 순수 포리스트를 만들 수도 있습니다. 마이그레이션한 하지는 순수 포리스트를 만들 수 있습니다만 VIP 사용자 하지만 관리 포리스트로 구현 하는에 설명 된 원칙 구현 [Active Directory 공격 가능성을 줄일](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md) 보안 만들려면 관리자 계정 및 순수 포리스트에서 레거시 포리스트를 관리할 수 있는 호스트 합니다. VIP 계정, 권한 있는 계정 및 보안 수준이 낮은에서 분리 합니다의 유일한 목표를 사용 하 여 Active Directory 인증서 서비스 (AD CS)를 실행 하는 서버와 같은 추가 보안을 요구 하는 시스템을 포함 하는 "특화 된" 포리스트를 구현할 수 있습니다. 포리스트입니다. 마지막으로, 결국 없앨 수를 통해 기존 포리스트를 해제할 수 있는 사실상 모든 사용자, 시스템, 응용 프로그램 및 데이터를 새 위치는 순수 포리스트를 구현할 수 있습니다.  
  
에 관계 없이 계획에서 이러한 원칙을 따르는 해야 있는지 여부를 순수 포리스트 소수의 사용자의 시스템 이거나 더 적극적으로 마이그레이션 위한 기본을 형성 합니다.  
  
1.  기존 포리스트 손상 된 것을 가정 합니다.  
  
2.  순수 포리스트를 신뢰 하는 레거시 환경을 구성할 수 있지만 기존 포리스트를 신뢰 하는 초기 환경을 구성 하지 마세요.  
  
3.  사용자 계정 또는 그룹 레거시 포리스트에서 마이그레이션하지 초기 환경 하는 계정의 그룹 멤버 자격, SID 기록 또는 기타 특성 악의적으로 수정 된 가능성이 있는 경우. 대신 "nonmigratory" 접근 방식을 사용 하는 순수 포리스트를 채웁니다. (Nonmigratory 방법은 나중에이 섹션에서 설명 합니다.)  
  
4.  순수 포리스트에 기존 포리스트에서 컴퓨터를 마이그레이션하지 않습니다. 순수 포리스트에 새로 설치 된 서버를 구현 하 고 새로 설치 된 서버에서 응용 프로그램을 설치 새로 설치 된 시스템에 응용 프로그램 데이터를 마이그레이션. 파일 서버에 대 한 새로 설치 된 서버에 데이터를 복사, 사용자 및 그룹에 새 포리스트를 사용 하 여 Acl을 설정 하 한 다음 비슷한 방식으로 인쇄 서버를 만듭니다.  
  
5.  레거시 운영 체제 또는 순수 포리스트의 응용 프로그램의 설치를 허용 하지 않습니다. 응용 프로그램 업데이트 되 고 새로 설치 될 수 없습니다, 하는 경우 레거시 포리스트에 두고 creative 파기 응용 프로그램의 기능을 대체 하는 것이 좋습니다.  
  
### <a name="defining-a-limited-risk-based-migration-plan"></a>제한, 위험 기반 마이그레이션 계획을 정의합니다.  
제한 된 만들기, 위험 기반 마이그레이션 계획 것을 의미 하는 경우 조직의 노출 되는 위험 수준을 기반으로 하는 마이그레이션 대상을 식별 해야는 사용자, 응용 프로그램 및 데이터에 순수 포리스트로 마이그레이션할 수를 결정할 때의 사용자 또는 시스템 손상 되었습니다. VIP 사용자 계정을 가진 공격자의 목표가 될 가능성이 높습니다. 순수 포리스트에 배치 해야 합니다. 중요 한 비즈니스 함수 중 완전히 깨끗 포리스트에 새로 빌드된 서버에 설치 해야 하며 매우 중요 한 데이터를 이동 해야 제공 하는 응용 프로그램 인위적인 포리스트에 서버를 보호 합니다.  
  
가장 중요 한 비즈니스 사용자를 분명히 아직 없는 경우 시스템, 응용 프로그램 및 Active Directory 환경에서 데이터 식별할 비즈니스 단위를 사용 하 여 작동 합니다. 모든 서버는 중요 한 응용 프로그램 실행 또는 중요 한 데이터가 저장 되는 대로 작동 하는 비즈니스에 필요한 모든 응용 프로그램 식별 되어야 합니다. 사용자와 계속 작동 하기 위해 조직에 필요한 리소스를 식별 하 여 최선을 다 할 자산의 자연스럽 게 우선 순위가 지정 된 컬렉션을 만듭니다.  
  
### <a name="leveraging-nonmigratory-migrations"></a>"Nonmigratory" 마이그레이션을 활용  
가 손상 된 경우 또는 레거시 Active Directory 설치에서 레거시 데이터 및 개체를 새 대시보드를로 마이그레이션하지을 선호 하는 주의 대상 기술적으로 하지 않는 마이그레이션 방법을 고려해 야 환경의 손상 된 것을 알고 있는지 여부를 "마이그레이션" 개체입니다.  
  
### <a name="user-accounts"></a>사용자 계정  
기존 Active Directory 마이그레이션 이상의 포리스트에서 다른에서는 사용자의 SID와 사용자가 기존 포리스트의 멤버인 그룹의 Sid를 저장할 사용자 개체에 대 한 SIDHistory (SID 기록) 특성이 됩니다. 사용자 계정이 새 포리스트를 마이그레이션됩니다 레거시 포리스트의 리소스에 액세스 하는 경우 SID 기록의 Sid는 전의 액세스 계정이 마이그레이션된 리소스에 액세스할 수 있도록 액세스 토큰을 만드는 데 사용 됩니다.  
  
하지만 SID 기록, 입증 된 일부 환경에서는 문제가 있는 유지 관리 현재 및 과거 Sid 사용 하 여 사용자의 액세스 토큰을 채우지 토큰 블 로트가 발생에서 발생할 수 있습니다. 토큰 블 로트가 발생 사용자의 액세스 토큰에 저장 해야 하는 Sid 수가 사용 하는 문제 또는 토큰에 사용 가능한 공간 크기를 초과 합니다.  
  
토큰 크기는 제한 된 범위의 증가할 수 있지만 토큰 블 로트가 발생 최종적인 솔루션 합리화 그룹 멤버 자격 제거 SID 기록 또는 둘의 조합으로 사용자 계정과 연결 된 Sid의 수를 줄이는 것입니다. 토큰 블 로트가 발생 하는 방법에 대 한 자세한 내용은 참조 하십시오 [MaxTokenSize 및 Kerberos 토큰 블 로트](http://blogs.technet.com/b/shanecothran/archive/2010/07/16/maxtokensize-and-kerberos-token-bloat.aspx)합니다.  
  
레거시 환경에서 사용자를 마이그레이션 대신 (특히 하나는 그룹 멤버 자격에 SID 기록을 손상 될 수 있습니다) SID 기록을 사용 하 여 사용자를 SID 기록을 수행 하지 않고 "마이그레이션" 메타 디렉터리 응용 프로그램을 활용 하 여 것이 좋습니다. 새 포리스트로 합니다. 새 포리스트의 사용자 계정 생성 되 면 계정 레거시 포리스트의 해당 계정에 매핑할 메타 디렉터리 응용 프로그램을 사용할 수 있습니다.  
  
기존 포리스트의 리소스에 새 사용자 계정 액세스를 제공 하려면 사용자의 기존 계정에 액세스를 부여 된 리소스 그룹을 식별할 메타 디렉터리 도구를 사용할 수 있으며 추가한 다음 이러한 그룹에 새 사용자 계정 키를 누릅니다. 기존 포리스트의 사용자 그룹 전략에 따라 리소스 액세스에 대 한 도메인 로컬 그룹을 만들거나 기존 그룹을 새 계정 리소스 그룹에 추가할 수 있도록 도메인 로컬 그룹으로 변환 해야 합니다. 가장 중요 한 응용 프로그램 및 데이터에 먼저 집중적으로 처리 하 고 (사용 하 여 또는 SID 기록이 없는) 새 환경으로 마이그레이션하여를 레거시 환경에서 사용 된 작업의 양을 제한할 수 있습니다.  
  

  
### <a name="servers-and-workstations"></a>서버 및 워크스테이션  
하나의 Active Directory에서 기존의 마이그레이션에서 포리스트 마이그레이션, 다른 컴퓨터에는 종종 비교적 간단한 마이그레이션 사용자, 그룹 및 응용 프로그램에 비해입니다. 컴퓨터 역할에 따라 새 포리스트를 마이그레이션 간단할 수 있습니다는 이전 도메인 가입을 취소와 새 조인 합니다. 그러나 컴퓨터 계정을 그대로로 마이그레이션하는 순수 포리스트 비교 새 환경을 만드는 목적은 합니다. (마이그레이션 잠재적으로 손상, 잘못 구성 된, 또는 오래 된) 컴퓨터 계정을 새 포리스트를 대신 새로 설치 해야 서버 및 워크스테이션 새 환경에서. 마이그레이션할 수 있습니다 데이터 중 완전히 깨끗 포리스트의 시스템에 레거시 포리스트의 시스템 있지만 시스템이 아닌 데이터를 수용 된 합니다.  
  
### <a name="applications"></a>애플리케이션  

응용 프로그램에서 다른 하나의 포리스트에서 모든 마이그레이션에서 가장 중요 한 과제를 제공할 수 이지만 "nonmigratory" 마이그레이션 경우 적용 해야 하는 가장 기본적인 원칙 중 인위적인 포리스트의 응용 프로그램 현재 되도록 지원 되 고 새로 설치 합니다. 가능한 경우 기존 포리스트의 응용 프로그램 인스턴스에서 데이터를 마이그레이션할 수 있습니다. 응용 프로그램 "다시 만들 수 없습니다" 순수 포리스트에 없는 경우에서 다음 섹션에 설명 된 대로 creative 소멸 또는 레거시 응용 프로그램의 격리와 같은 방식을 고려해 야 합니다.  
  
### <a name="implementing-creative-destruction"></a>Creative 소멸 구현  
Creative 소멸은 경제 용어 소멸의 순서를 이전에 만든 경제 개발에 설명입니다. 최근 몇 년 간에서 정보 기술 용어 적용 되었습니다. 일반적으로 참조 메커니즘에서 어떤 기존 인프라를 제거 하는, 업그레이드, 아니라 완전히 새로운 작업으로 대체 합니다. 2011 [Gartner Symposium ITXPO](http://www.gartner.com/technology/symposium/orlando/) Cio 및 선임 IT 임원 나오는 비용 절감 및 증가 대 한 해당 키 테마 중 하나로 creative 소멸 효율성에 대 한 합니다. 향상 된 보안 기능으로 프로세스의 자연 스러운 outgrowth 있을 수 있습니다.  

예를 들어, 조직의 다양 한 수준의 modernity 및 공급 업체 지원 사용 하 여 유사한 기능을 수행 하는 다른 응용 프로그램을 사용 하는 여러 비즈니스 단위가 구성 될 수 있습니다. 지금까지 IT 각 사업부의 응용 프로그램을 별도로 유지 관리 해야 할 수 있습니다 및 통합 노력으로 구성 됩니다 최상의 기능을 제공 하는 응용 프로그램을 파악 하는 동안 및 후에 데이터를 마이그레이션 다른 응용 프로그램입니다.  
  
Creative 소멸을 하지 않고 오래 되었거나 중복 응용 프로그램 유지 관리에서 이전 대체, 새 응용 프로그램에 데이터를 마이그레이션에 완전히 새로운 응용 프로그램을 구현 하 고 이전의 응용 프로그램 및 실행 되는 시스템을 역할 해제 합니다. 경우에 따라 고유한 인프라에 새 응용 프로그램을 배포 하 여 레거시 응용 프로그램의 창의적인 소멸을 구현할 수 있습니다 있지만 가능할 대신 클라우드 기반 솔루션에 응용 프로그램을 이식 고려해 야 합니다.  
  
레거시 사내 응용 프로그램의 이름을 바꾸려면 클라우드 기반 응용 프로그램을 배포 하 여 유지 관리 노력 및 비용 뿐만 아니라 줄일 있지만 레거시 시스템 및 보안 문제를 제공 하는 응용 프로그램을 제거 하 여 조직의 공격 노출 영역을 줄이기 공격자가 활용 하 합니다. 이 방법은 동시에 인프라에서 레거시 대상을 제거 하는 동안 원하는 기능을 가져오려면 조직에 대 한 더 빠른 방법이 제공 합니다. Creative 소멸 원칙 IT는 모든 자산에 적용 되지 않습니다, 하지만 동시에 강력한, 보안, 클라우드 기반 응용 프로그램을 배포 하는 동안 레거시 시스템 및 응용 프로그램을 제거 하는 종종 실행 가능한 옵션이 제공 합니다.  
  
### <a name="isolating-legacy-systems-and-applications"></a>레거시 시스템 및 응용 프로그램 격리  
비즈니스에 중요 한 사용자 및 시스템 중 완전히 깨끗 하 고 안전한 환경으로 마이그레이션하는 과정의 자연 스러운 outgrowth 덜 중요 한 정보 및 시스템 레거시 포리스트에 포함 되어 있음을입니다. 레거시 시스템 및 보안 수준이 낮은 환경에 남아 있는 응용 프로그램 수를 제공 하지만 상승 된 위험을 손상, 또한 손상의 축소 된 심각도를 나타냅니다. 위치 이동 하 고 중요 한 비즈니스 자산 현대화를 효과적으로 관리 배포 및 레거시 설정 및 프로토콜을 수용 하기 위해 필요 하지 않은 하는 동안 모니터링에 집중할 수 있습니다.  
  
옵션을 평가할 수는 순수 포리스트를 중요 한 데이터를 기반으로 있는 경우 추가 "main" AD DS 포리스트의 응용 프로그램과 레거시 시스템을 격리 합니다. 하나의 응용 프로그램 및 실행 되는 서버를 교체할 수 creative 소멸을 구현할 수 있습니다, 있지만 다른 경우에 해야 할 최소 보안 시스템 및 응용 프로그램의 추가적인 격리가. 예를 들어, 소수의 사용자에 게 사용 되는 작은 도메인 LAN Manager 해시와 같은 레거시 자격 증명을 마이그레이션할 수 있습니다 필요 하지만 응용 프로그램 하기 위해 만든 없는 대체 옵션을 사용할 수 없는 시스템을 지원 합니다.  
  
레거시 설정 구현의 강제 적용 하는 도메인에서 이러한 시스템을 제거 하 여 현재 운영 체제 및 응용 프로그램을 지원 하도록 구성 하 여 도메인의 보안을 이후에 늘릴 수 있습니다. 하지만 가능 하면 응용 프로그램과 레거시 시스템을 해제 하는 것이 좋습니다. 서비스 해제 작업이 불가능 한 경우 단순히 레거시 구성원의 작은 세그먼트를 레거시 설치의 나머지 부분에서 증분 향상 된 기능을 수행할 수 있습니다 별도 도메인 (또는 포리스트)를 분리 합니다.  
  
### <a name="simplifying-security-for-end-users"></a>최종 사용자에 대 한 보안 간소화  
대부분의 조직에서는 종종 조직에서 해당 역할의 특성으로 인해 가장 중요 한 정보에 대 한 액세스 권한이 있는 사용자에는 복잡 한 액세스 제한 사항 및 컨트롤을 학습에 사용할 최소 기간입니다. 조직의 모든 사용자에 대 한 포괄적인 보안 교육 프로그램 해야 하지만 또한 중점을 두어야를 최대한 투명 하 고 간소화 하기 위한 원칙을 어떤 사용자가 있는 컨트롤을 구현 하 여 사용 하는 보안 준수 합니다.  
  
예를 들어 있는 임원 및 다른 Vip는 하는 데 필요한 보안 워크스테이션을 사용 하 여 중요 한 데이터 및 시스템 액세스 덜 중요 한 데이터에 액세스 하는 다른 장치를 사용 하도록 허용 하는 정책을 정의할 수 있습니다. 사용자가 기억 하기에 대 한 간단한 원칙 이지만 다양 한 방법을 적용 하는 데 백 엔드 컨트롤을 구현할 수 있습니다.  

사용할 수 있습니다 [인증 메커니즘 보증](https://technet.microsoft.com/library/dd391847(v=WS.10).aspx) 사용자 권한 제어에 대 한 제한 하면 스마트 카드를 사용 하 여 해당 보안 시스템에 로그온 하 고 IPsec을 사용할 수 있고 경우에 중요 한 데이터에 액세스 하도록 허용 하는 중요 한 데이터 저장소에 연결할 수 있는 시스템. 사용할 수는 [Microsoft 데이터 분류 도구 키트](https://www.microsoft.com/download/details.aspx?id=27123) 구현할 수 있습니다 및 강력한 파일 분류 인프라를 빌드하는 데 [동적 Access Control](http://blogs.technet.com/b/windowsserver/archive/2012/05/22/introduction-to-windows-server-2012-dynamic-access-control.aspx) 기반으로 데이터에 대 한 액세스를 제한 하려면 비즈니스 규칙 기술적 컨트롤 변환 액세스 시도, 특성입니다.  
  
사용자의 관점에서는 작동, 보안된 시스템에서 중요 한 데이터 액세스 및 보안 되지 않은 시스템에서 작업을 수행 하는 동안 "만 하지 않습니다." 그러나 모니터링 및 관리 환경의의 관점에서 도움을 요청 사용자가 액세스 방식을 중요 한 데이터 및 시스템을 쉽게 비정상적인 액세스 시도 감지할 수에서 식별할 수 있는 패턴을 만들 수 있습니다.  
  
사용자는 길고 복잡 한 암호가에 대 한 저항 했습니다 부족 하 여 암호 정책, 특히 VIP 사용자에 대 한 환경에서 스마트 카드를 통한 인증을 하는 다른 방법을 고려 (다양 한 폼 팩터에서에서 제공 되는 및 손가락 살짝 밀기 판독기나에서 보호 되는 인증 데이터 등의 생체 인식 컨트롤 추가 기능 인증을 강화 하기 위해)를 사용 하 여 사용자의 컴퓨터에서 TPM (플랫폼 모듈) 칩 신뢰할 수 있습니다. 컴퓨터가 이미 손상 되 면 사용자에 게 사용 하기 쉬운 인증 제어를 제공 하 여 다단계 인증 자격 증명 도난 공격 하지 않는 있지만를 할당할 수 있습니다 더 강력한 암호 사용자의 계정에 대 한 기존 사용자 이름 및 암호 컨트롤은 비 져 서 없습니다.
