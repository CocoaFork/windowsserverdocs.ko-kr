# <a name="implementing-secure-administrative-hosts"></a>보안 관리 호스트 구현

>적용 대상: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

보안 관리 호스트 되는 권한 있는 계정을 Active Directory 또는 도메인 컨트롤러에서 관리 작업을 수행할 수는 안전한 플랫폼을 만들기 위해 특별히 구성 된 서버나 워크스테이션 도메인에 가입 된 시스템 및 응용 프로그램 도메인에 가입 된 시스템을 실행 합니다. 이 경우 "권한 있는 계정" Active Directory에서 가장 권한이 있는 그룹의 구성원 인 계정 뿐 아니라 모든 위임 된 권한 된 계정 및 관리 작업을 수행할 수 있는 권한을 나타냅니다.  
  
이러한 계정에는 대부분의 DNS 레코드 및 영역을 관리 하는 데 사용 되는 계정 또는 구성 관리에 사용 되는 계정 도메인에서 사용자에 대 한 암호를 재설정할 수 있는 지원 센터 계정 수 있습니다. 보안 관리 호스트를 전담 하 관리 기능 및 Microsoft Office와 같은 전자 메일 응용 프로그램, 웹 브라우저 또는 생산성 소프트웨어와 같은 소프트웨어 실행 하지 마십시오.  
  
"가장 권한이 있는" 계정 및 그룹을 적절 하 게 해야 가장 엄격 하 게 보호 되지만 모든 계정 및 그룹 부여 된 계정 보다 높은 수준의 표준 사용자는 권한으로 보호 해야를 제거 하지 않습니다.  
  
보안 관리 호스트 관리 작업을 원격 데스크톱 게이트웨이 서버 역할을 실행 하는 멤버 서버에만 사용 되는 전용된 워크스테이션이 수 있고 IT 사용자를 관리 대상 호스트 또는 실행 하는 서버 연결 Hyper-v 역할 및 해당 관리 작업에 사용할 각 IT 사용자의 고유한 가상 컴퓨터를 제공 합니다. 많은 환경에서는 세 방법 모두의 조합은 구현 될 수 있습니다.  
  
보안 관리 호스트 구현 계획 및 조직의 크기, 관리 사례, 위험 성향 및 예산을 사용 하 여 일치 하는 구성에 필요 합니다. 고려 사항 및 보안 관리 호스트 구현 하기 위한 옵션 제공 됩니다 여기 조직에 적합 한 관리 전략 개발에서 사용할 수 있습니다.  
  
## <a name="principles-for-creating-secure-administrative-hosts"></a>보안 관리 호스트를 만들기 위한 원칙  
공격 으로부터 시스템을 효과적으로 보호 하려면 몇 가지 일반적인 원칙 염두에서 유지 해야 합니다.  
  
1.  되지 관리 하는 것 (즉, 도메인 컨트롤러와 같은 보안 서버)를 신뢰할 수 있는 시스템에서 신뢰할 수 없는 호스트 (즉, 관리 하는 시스템과 동일한 수준으로 보호 되지 않은 워크스테이션).  
  
2.  권한 있는 작업을 수행 하는 경우 단일 인증 요소에 의존 하지 않아야 즉, 사용자 이름 및 암호 조합은 간주 되지 않아야 적합 한 인증 (알고)는 단일 요소에만 표시 되기 때문에 있습니다. 자격 증명 생성 및 캐시 되거나 관리 시나리오에서 저장 위치를 고려해 야 합니다.  
  
3.  현재 위협 상황에서 대부분의 공격은 맬웨어 및 악의적인 해킹을 활용 하지만 물리적 보안 설계 및 보안 관리 호스트를 구현 하는 경우를 생략 하지 마십시오.  
  
### <a name="account-configuration"></a>계정 구성  
조직의 현재 스마트 카드를 사용 하지 않는 경우에 권한 있는 계정 및 보안 관리 호스트에 대 한 스크립트를 구현 하는 것이 좋습니다. 관리 호스트는 관리 호스트를 포함 하는 Ou에 연결 된 GPO에 다음 설정을 수정 하 여 모든 계정에 대 한 스마트 카드 로그온이 필요 하도록 구성 되어야 합니다.  
  
**컴퓨터 구성 \ 정책 \ 로컬 정책 \ 옵션 \ 대화형 로그온: 스마트 카드 필요**  
  
이 설정은 개별 계정 Active Directory에서 구성에 관계 없이 스마트 카드를 사용 하는 모든 대화형 로그온을 해야 합니다.  
  
보안 관리 호스트에 구성 될 수 있는 권한 있는 계정에 의해서만 로그온을 허용 하도록 구성 해야 합니다.  
  
**컴퓨터 구성 \ 정책 \ 로컬 정책 \ 보안 \ 로컬 정책 \ 사용자 권한 할당**  
  
이 대화형 부여 (및 필요한 경우 원격 데스크톱 서비스) 보안 관리 호스트의 권한이 있는 사용자만 로그온 권한이 있습니다.  
  
### <a name="physical-security"></a>물리적 보안  
신뢰할 수 있는 것으로 간주 하 여 관리 호스트에 대 한 해당 구성 하 고 자신이 관리 하는 시스템과 동일한 수준으로 보호 합니다. 에 제공 된 권장 사항의 대부분 [공격에 대 한 도메인 컨트롤러 보안](../../../ad-ds/plan/security-best-practices/Securing-Domain-Controllers-Against-Attack.md) 도메인 컨트롤러 및 AD DS 데이터베이스를 관리 하는 데 사용 되는 호스트에 해당 됩니다. 대부분의 환경에서 보안 관리 시스템을 구현 하는 문제 중 하나는 물리적 보안 서버와 같은 데이터 센터에서 호스트 되는 것 만큼 안전 하지 않은 영역에서 이러한 컴퓨터는 종종 하므로 구현 하기가 더 어렵습니다 수 관리 사용자의 바탕 화면입니다.  
  
물리적 보안 관리 호스트에 대 한 물리적 액세스를 제어 하는 포함 되어 있습니다. 소규모 조직에서는 office 또는 사용 중이 아니면 센터 서랍에서 유지 되는 전용된 관리 워크스테이션 잠금 유지 관리 하는이 의미할 수 있습니다. 또는 Active Directory 또는 도메인 컨트롤러의 관리를 수행 해야 할 때 로그온 한 도메인 컨트롤러에 직접 감소할 수 있습니다.  
  
중간 규모 조직에서는 구현을 고려할 수 보안 관리 "점프 서버"는 사무실의 안전한 위치에 있고 Active Directory 또는 도메인 컨트롤러의 관리를 필요할 때 사용 됩니다. 또한 사용 점프 서버 유무에 없는 경우 안전한 위치에 잠겨 있는 관리 워크스테이션을 구현할 수 있습니다.  
  
대규모 조직에서는 Active directory; 엄격 하 게 제어 된 액세스를 제공 하는 데이터 센터 있는 점프 서버를 배포할 수 있습니다. 도메인 컨트롤러 및 파일, 인쇄 또는 응용 프로그램 서버입니다. 점프 서버 아키텍처의 구현은 가능성이 가장 높은 대규모 환경에서는 보안 워크스테이션 및 서버 조합을 포함 합니다.  
  
관리 호스트의 디자인을 조직의 규모에 관계 없이 무단된 액세스 또는 도난 으로부터 물리적 컴퓨터를 보호 해야 하 고이 암호화 및 보호 관리 호스트의 드라이브에 BitLocker 드라이브 암호화를 사용 해야 . 관리 호스트에 BitLocker를 구현 하 여 호스트를 도난당 하거나 해당 디스크를 제거 하는 경우에 확실히 드라이브의 데이터가 권한이 없는 사용자에 액세스할 수 없는 합니다.  
  
### <a name="operating-system-versions-and-configuration"></a>운영 체제 버전 및 구성  
모든 관리 호스트를 실행할지 여부를 서버 또는 워크스테이션에 최신 운영 체제가이 문서의 앞부분에서 설명한 이유 때문에 조직에서 사용에서 합니다. 현재 운영 체제를 실행 하 여 관리 담당자는 새로운 보안 기능, 전체 공급 업체 지원 및 운영 체제에 도입 된 추가 기능에서 이점을 제공 합니다. 또한 새 운영 체제를 관리 하는 첫 번째 호스트를 배포 하 여를 평가 하는 경우 해야 익히는 새로운 기능, 설정 및 이후에 계획에 활용할 수 있는 관리 메커니즘을 제공 합니다. 운영 체제의 광범위 한 배포 합니다. 으로 조직에서 가장 복잡 한 사용자는 새 운영 체제를 사용 하 여 친숙 하 고 가장 배치 지원 하도록 사용자를 수도 있습니다.

### <a name="applocker"></a>AppLocker  
관리 호스트 및 가상 machinesshould 스크립트, 도구 및 AppLocker 또는 타사 응용 프로그램 제한 소프트웨어를 통해 응용 프로그램 허용 목록으로 구성 합니다. 모든 관리 응용 프로그램 또는 보안 설정을 준수 하지 않는 유틸리티를 업그레이드 하거나 안전한 개발 및 관리 사례를 준수 하는 도구를 사용 하 여 대체 해야 합니다. 신규 또는 추가 도구는 관리 호스트에 필요한 경우 응용 프로그램 및 유틸리티 철저히 테스트 해야, 및 도구 관리 호스트에 배포에 적합 한 경우 시스템의 허용 목록에 추가할 수 있습니다.  
  
### <a name="rdp-restrictions"></a>RDP 제한 사항  
계정 및 컴퓨터 수를 관리 되는 시스템에 대 한 원격 데스크톱 프로토콜 (RDP) 연결 제한에 포함 해야 특정 구성 관리 시스템의 아키텍처에 따라 다르지만 원격 데스크톱 게이트웨이 (RD 게이트웨이)를 사용 하는 등 도메인 컨트롤러에 대 한 액세스 및 권한이 있는 사용자 및 시스템에서 다른 관리 되는 시스템을 제어 하는 서버를 이동 합니다.  
  
권한 있는 사용자가 대화형 로그온을 허용 및 해야 제거 하거나 해야 서버 액세스를 위한 필요 하지 않은 다른 로그온 유형은 차단 합니다.  
  
### <a name="patch-and-configuration-management"></a>패치 및 구성 관리  
소규모 조직에서는 Windows 업데이트와 같은 제품에서 사용할 수 있습니다 또는 [Windows Server Update Services](https://technet.microsoft.com/windowsserver/bb332157) (WSUS) 대규모 조직에서는 enterprise 패치를 구현할 수 있습니다 하는 동안 Windows 시스템에 대 한 업데이트 배포를 관리 하 고 System Center Configuration Manager와 같은 소프트웨어를 관리 하는 구성입니다. 일반 서버 및 워크스테이션 모집단에 업데이트를 배포 하는 메커니즘에 관계 없이 보안 수준이 높은 시스템 도메인 컨트롤러와 인증 기관 관리 호스트에 대 한 별도 배포를 고려해 야 합니다. 일반 관리 인프라에서 이러한 시스템을 분리 하 여 관리 소프트웨어 또는 서비스 계정을 손상 된 경우에 이러한 타협 쉽게 확장할 수 없습니다 인프라의 가장 강력한 보안 시스템에 있습니다.  
  
보안 시스템에 대 한 수동 업데이트 프로세스를 구현 하지 않아야 하지만 보안 시스템 업데이트에 대 한 별도 인프라를 구성 해야 합니다. 매우 큰 조직의 경우에이 인프라 일반적으로 구현할 수 있습니다 전용된 WSUS 서버 및 Gpo를 통해 보안된 시스템에 대 한 합니다.  
  
### <a name="blocking-internet-access"></a>인터넷 액세스를 차단  
관리 호스트는 인터넷에 액세스 하 허용 해서는 안 없으며 조직 인트라넷을 찾아볼 수 있습니다. 웹 브라우저와 유사한 응용 프로그램 관리 호스트에 허용 해서는 안 됩니다. 경계 방화벽 설정, WFAS 구성 및 보안 호스트의 "블랙홀" 프록시 구성을 조합 하 여 보안 호스트에 대 한 인터넷 액세스를 차단할 수 있습니다. 관리 호스트에 사용 되 고 웹 브라우저를 방지 하기 위해 응용 프로그램 허용 목록을 사용할 수도 있습니다.  
  
### <a name="virtualization"></a>가상화  
가능한 경우에 관리 호스트로 가상 컴퓨터를 구현 하는 것이 좋습니다. 가상화를 사용 하 여 시스템 생성할 수 있습니다 사용자별 관리는 중앙 집중식으로 저장 되 고 관리 하 고는 쉽게 종료할 수 시기, 사용 중이 아님는 자격 증명은 하지 활성 상태로 유지 관리 시스템에서 확인 합니다. 가상 관리 호스트는 다시 설정 초기 스냅숏을 각 사용 후 가상 머신을 초기 상태로 유지 되도록 보장을 요구할 수 있습니다. 관리 호스트의 가상화를 위한 옵션에 대 한 자세한 내용은 다음 섹션에 제공 됩니다.  
  
## <a name="sample-approaches-to-implementing-secure-administrative-hosts"></a>관리 호스트를 보호 하는 샘플 구현 하는 방법  
어떻게를 디자인 하 고 호스트 관리 인프라를 배포 하는 것에 관계 없이 유지 해야 염두에서 "원칙에 대 한 만들기 보안 관리 호스트"이이 항목의 앞부분에 나오는 지침입니다. 여기에 설명 된 방법 중 각 "관리" 및 "productivity" 시스템 IT 직원이 사용한를 구분할 수 하는 방법에 대 한 일반 정보를 제공 합니다. 생산성 시스템은 전자 메일 확인, 인터넷을 탐색 하 고 Microsoft Office와 같은 일반적인 생산성 소프트웨어를 사용 하려면 IT 관리자를 사용 하는 컴퓨터. 관리 시스템은 강화 되어 IT 환경의 일상적인 관리에 대해 사용할 전용 된 컴퓨터.  
  
보안 관리 호스트를 구현 하는 가장 간단한 방법은 보안된 워크스테이션 관리 작업을 수행할 수 있습니다를 사용 하 여 IT 담당자를 제공 하는 것입니다. 워크스테이션이 전용 구현에서 각 관리 워크스테이션 관리 도구를 시작 하는 서버 및 기타 인프라 관리에 대 한 RDP 연결에 사용 됩니다. 관리 호스트는 전용 관리 서버 및 워크스테이션을 사용 하는 큰 규모의 더 복잡 한 인프라 분산 된 설계에서 이점을 얻을 수 있지만 워크스테이션 전용 구현 소규모 조직에서는 효과적일 수 있습니다으로 이 항목의 뒷부분에 나오는 "구현 보안 관리 워크스테이션 및 점프 서버"에서 설명합니다.  
  
### <a name="implementing-separate-physical-workstations"></a>별도 실제 워크스테이션 구현  
관리 호스트 구현 하는 한 가지 방법은 각 IT 사용자 두 워크스테이션을 발급 하는 것입니다. 워크스테이션 1 대는 전자 메일을 확인 하 고 생산성 응용 프로그램을 사용 하 여 두 번째 워크스테이션께 바 칩니다 엄격 하 게 관리 기능 등의 작업을 수행 하려면 "일반" 사용자 계정으로 사용 됩니다.  
  
생산성 워크스테이션에 대 한 권한 있는 계정을 사용 하 여 보안이 설정 되지 않은 컴퓨터에 로그온 하는 것이 아니라 일반 사용자 계정에는 IT 직원 지정할 수 있습니다. 관리 워크스테이션 엄격 하 게 제어 된 구성을 사용 하 여 구성 해야 하 고 IT 직원 관리 워크스테이션에 로그온 할 다른 계정을 사용 해야 합니다.  
  
스마트 카드를 구현한 경우 관리 워크스테이션 스마트 카드 로그온을 요구 하도록 구성 해야 하 고 IT 직원에 게 계정을 제공 해야 별도 대화형 로그온에 스마트 카드를 요구 하도록 구성 하 여 관리자에 대 한 합니다. 관리 호스트 여야 확정 된 이전에 설명한 대로, 하며 지정 된 IT 사용자만 관리 워크스테이션에 로컬로 로그온 할 수 있어야 합니다.  
  
#### <a name="pros"></a>전문가  
별도 실제 시스템을 구현 하 여 해당 역할에 대 한 각 컴퓨터는 적절 하 게 구성 하 고 IT 사용자 위험 관리 시스템을 노출할 수 없습니다 실수로 확인할 수 있습니다.  
  
#### <a name="cons"></a>단점  
  
-   별도 물리적 컴퓨터를 구현 하면 하드웨어 비용이 늘어납니다.  
  
-   메모리에서 자격 증명을 캐시 원격 시스템을 관리 하는 데 사용 되는 자격 증명을 사용 하 여 물리적 컴퓨터에 로그온 합니다.  
  
-   관리 워크스테이션 안전 하 게 저장 되어 있지 않으면 물리적 하드웨어 키로 거 또는 다른 실제 공격과 같은 메커니즘을 통해 손상에 취약할 수 있습니다.  
  
### <a name="implementing-a-secure-physical-workstation-with-a-virtualized-productivity-workstation"></a>가상화 된 생산성 워크스테이션을 사용 하 여 물리적 보안 워크스테이션을 구현합니다.  
이 경우 IT 사용자 지정 됩니다 일상적인 관리 기능을 수행할 수 있는 보안된 관리 워크스테이션 책임의 해당 범위 내에서 원격 서버 관리 도구 (RSAT) 또는 서버에 대 한 RDP 연결을 사용 하 여. IT 사용자 생산성 향상 작업 하는 데 필요한 가상 머신으로 실행 되는 원격 생산성 워크스테이션에 RDP를 통해 연결할 수 있습니다. 각 워크스테이션에 대 한 별도 자격 증명을 사용 해야 하 고 스마트 카드와 같은 컨트롤을 구현 해야 합니다.  
  
#### <a name="pros"></a>전문가  
  
-   관리 워크스테이션 및 생산성 워크스테이션 구분 됩니다.  
  
-   IT 직원이 생산성 워크스테이션에 연결할 보안 워크스테이션을 사용 하 여 별도 자격 증명 및 스마트 카드를 사용할 수 있습니다 및 권한 있는 자격 증명 보안 수준이 낮은 컴퓨터의 보관 되지 않습니다.  
  
#### <a name="cons"></a>단점  
  
-   디자인 및 구현 작업 옵션과 강력한 가상화 솔루션 구현에 필요 합니다.  
  
-   실제 워크스테이션 안전 하 게 저장 되어 있지 않으면, 하드웨어 또는 운영 체제를 손상 시키고 통신 인터 셉 션에 취약 하 게 하는 물리적 공격에 취약할 수 있습니다.  
  
### <a name="implementing-a-single-secure-workstation-with-connections-to-separate-productivity-and-administrative-virtual-machines"></a>구분 "Productivity" 및 "관리" Virtual Machines에 대 한 연결을 사용 하 여 단일 보안 워크스테이션을 구현 합니다.  
이 방법에서는 단일 실제 워크스테이션은 잠겨 앞에서 설명한 대로, 않으며 IT 사용자는 권한 있는 액세스를 제공 하지 않습니다 IT 사용자에 게 실행할 수 있습니다. 전자 메일 및 기타 생산성 응용 프로그램을 실행 하는 하나의 가상 컴퓨터와 사용자의 서버로 구성 되는 두 번째 가상 컴퓨터를 사용 하 여 IT 직원에 게 제공 전용된 서버에서 호스팅되는 가상 컴퓨터에 대 한 원격 데스크톱 서비스 연결 제공할 수 있습니다. 전용된 관리 호스트입니다.  
  
해야 스마트 카드 또는 기타 다단계 로그온 가상 컴퓨터용 물리적 컴퓨터에 로그온 하는 데 사용 되는 계정이 아닌 별도 계정을 사용 합니다. IT 사용자가 실제 컴퓨터에 로그온 한 후에 별도 계정이 원격 생산성 컴퓨터 및 원격 관리 컴퓨터에 연결 하는 스마트 카드에 연결할 생산성 스마트 카드를 사용할 수 있습니다.  
  
#### <a name="pros"></a>전문가  
  
-   IT 사용자는 단일 실제 워크스테이션을 사용할 수 있습니다.  
  
-   가상 호스트 및 가상 컴퓨터에 원격 데스크톱 서비스 연결을 사용 하 여 필요한 별도 계정에서 IT 사용자의 자격 증명은 로컬 컴퓨터의 메모리에 캐시 되지 않습니다.  
  
-   로컬 컴퓨터의 손상 가능성을 줄이고 관리 호스트로 폭 실제 호스트를 보호할 수 있습니다.  
  
-   가상 컴퓨터는 IT 사용자의 생산성 가상 컴퓨터 또는 관리 가상 컴퓨터가 손상 된 경우에는 "알려진된 정상" 상태를 재설정할 수 쉽게 있습니다.  
  
-   물리적 컴퓨터가 손상 된 경우에 권한 있는 자격 증명 없이 메모리에 캐시 됩니다 및 스마트 카드를 사용 하는 키 입력으로 거 하 여 자격 증명 손상을 방지할 수 있습니다.  
  
#### <a name="cons"></a>단점  
  
-   디자인 및 구현 작업 옵션과 강력한 가상화 솔루션 구현에 필요 합니다.  
  
-   실제 워크스테이션 안전 하 게 저장 되어 있지 않으면, 하드웨어 또는 운영 체제를 손상 시키고 통신 인터 셉 션에 취약 하 게 하는 물리적 공격에 취약할 수 있습니다.  
  
### <a name="implementing-secure-administrative-workstations-and-jump-servers"></a>보안 관리 워크스테이션 및 점프 서버 구현  
관리 워크스테이션을 보호 하는 대신 또는 이러한 함께 보안 점프 서버를 구현할 수 있습니다 및 관리자에 게 스마트 카드 및 RDP를 사용 하 여 관리 작업을 수행할 점프 서버에 연결할 수 있습니다.  
  
점프 서버에서 관리 되는 대상 서버로 점프 서버를 연결에서 제한을 구현할 수 있도록 원격 데스크톱 게이트웨이 역할을 실행 하도록 구성 되어야 합니다. 가능한 경우 또한 Hyper-v 역할 설치를 만듭니다 [개인용 가상 데스크톱](https://technet.microsoft.com/library/dd759174.aspx) 또는 다른 사용자 당 가상 머신을 점프 서버에 있는 해당 작업에 사용할 관리 사용자에 대 한 합니다.  
  
점프 서버에서 관리자가 사용자 당 가상 컴퓨터를 제공, 하 여 관리 워크스테이션에 대 한 물리적 보안을 제공 하 고 관리자에 게 다시 설정 하거나 사용 중이 아니면 해당 가상 머신을 종료 수 있습니다. 동일한 관리 호스트에 Hyper-v 역할 및 원격 데스크톱 게이트웨이 역할을 설치 하지 않으려는 경우에 별도 컴퓨터에 설치할 수 있습니다.  
  
가능한 경우 서버를 관리 하 원격 관리 도구를 사용 해야 합니다. 사용자의 가상 컴퓨터 (또는 관리에 대 한 사용자별 가상 컴퓨터를 구현 하지 않는 경우에 점프 서버)에서 원격 서버 관리 도구 (RSAT) 기능을 설치 해야 하 고 RDP를 통해 연결 해야 관리 담당자가 관리 작업을 수행할 가상 머신입니다.  
  
경우에서 관리 사용자를 직접 관리 하기 위해 대상 서버에 RDP를 통해 연결 해야 하는 경우 RD 게이트웨이 구성 해야 적절 한 사용자 및 컴퓨터는 대상에 대 한 연결을 설정 하는 데 사용 하는 경우에 연결할 수 있도록 서버입니다. 실행 서버 점프 하지 않고 일반 사용 하는 워크스테이션 및 구성원 서버는 같은 시스템 관리 시스템을 지정 되지 않은 도구 (또는 이와 유사한) rsat 금지 되어야 합니다.  
  
#### <a name="pros"></a>전문가  
  
-   점프 서버 만들기를 통해 "영역" (시스템 구성, 연결 및 보안 요구 사항이 유사한 모음) 네트워크에 특정 서버에 매핑할 관리 담당자가 각 영역의 관리 이루어집니다를 요구 하도록 지정 된 "영역" 서버에 보안 관리 호스트에서 연결 합니다.  
  
-   영역에 점프 서버를 매핑하여 연결 속성 및 구성 요구 사항에 대 한 세부적인 제어를 구현할 수 있습니다 하 고 허가 되지 않은 시스템에서 연결 하려는 시도 쉽게 식별할 수 있습니다.  
  
-   점프 서버의 관리자 당 가상 컴퓨터를 구현 하 여 종료 및 관리 작업을 완료할 때 알려진된 깨끗 한 상태로 가상 머신을 다시 설정을 적용 합니다. 관리 작업을 완료할 때 가상 머신의 종료 (또는 다시 시작)를 적용 하 여 가상 컴퓨터를 공격자에 의해 대상으로 지정할 수 없으며 자격 증명 도난 공격 가능 메모리에 캐시 된 자격 증명을 다시 부팅 이후 유지 되지 않는 때문에.
