# <a name="identifying-the-deployment-project-participants"></a>배포 프로젝트 참가자에 게 확인

>적용 대상: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Active Directory 도메인 서비스 (AD DS)에 대 한 배포 프로젝트를 설정 하는 첫 번째 단계 디자인을 설정 하는 및 배포 프로젝트 팀은 디자인 및 배포 단계가 Active Directory 프로젝트 관리 수 있는 전환 합니다. 또한 사용자 및 그룹 소유 하 고 배포 완료 된 후 디렉터리 유지 하기 위한 확정 되지 않은 것을 식별 해야 합니다.

-   [소유자와 관리자 설정](#BKMK_2)  
  
-   [빌딩 프로젝트 팀](#BKMK_3)  
  
## <a name="BKMK_1"></a>프로젝트 특정 역할 정의  
프로젝트 팀을 설정 하는 중요 한 단계 프로젝트 특정 역할을 하는 개인을 식별 하는 것입니다. 여기에 임원진, 프로젝트 설계자 프로젝트 관리자 포함 됩니다. 이러한 개인은 배포 프로젝트 Active Directory를 실행 합니다.  
  
후 프로젝트 설계자 및 프로젝트 관리자를 지정 이러한 개인 조직에서 통신 채널을 설정, 일정, 빌드하고 식별 하는 개인을 프로젝트 팀의 구성원 다양 한 소유자와 시작 됩니다.  
  
### <a name="executive-sponsor"></a>임원진  
AD DS 등 infrastructure 배포 조직에 광범위 한 영향을 미칠 수 있습니다. 이 이런 이유이 때문에 대 한 중요 있는 지원 배포 비즈니스 가치를 이해 하 고, 집행 수준 프로젝트를 지원 하며 조직에서 충돌을 해결 하는 데 도움이 집행 조직이 합니다.  
  
### <a name="project-architect"></a>프로젝트 설계자  
각 Active Directory 배포 프로젝트 Active Directory 디자인 및 배포 의사 과정을 관리 하는 프로젝트 설계자가 필요 합니다. 설계자 전문을 디자인 하 고 광고 DS 배포 하는 과정에 게 도움을 제공 합니다.  
  
> [!NOTE]  
> 조직에서 기존 없는 직원이 환경을 디렉터리 디자인을가지고 고용 외부 컨설턴트 Active Directory 디자인 및 배포 전문가 되 게 하는 것이 좋습니다.  
  
Active Directory 프로젝트 설계자의 책임은 다음과 같습니다.  
  
-   소유 Active Directory 디자인  
  
-   이해 하 고 녹화 키에 대해 설명 디자인 결정  
  
-   디자인 조직 비즈니스 요구를 충족 있는지 확인  
  
-   디자인, 배포 및 운영 팀 사이의 합의 설정  
  
-   이해 광고 DS 통합 응용 프로그램의 요구 사항  
  
최종 Active Directory 디자인 결정 기술 및 비즈니스 목표의 조합을 반영 해야 합니다. 따라서 프로젝트 설계자 맞추는 비즈니스 목표는 수 있도록 디자인을 검토 해야 합니다.  
  
### <a name="project-manager"></a>프로젝트 관리자  
프로젝트 관리자 비즈니스 단위 간에 및 기술 관리 그룹 간에 협력을 수 있습니다. 이상적, Active Directory 배포 프로젝트 관리자는 사람에 게 IT 부서 및 배포 하는 광고 DS 준비 하 고 그룹에 대 한 디자인 요구 운영 정책을 잘 조직 내 합니다. 프로젝트 관리자 디자인부터 구현까지 계속 및 전체 배포 프로젝트 감독 고 프로젝트 유지 budget와 일정에 있는지 확인 합니다. 프로젝트 관리자의 책임은 다음과 같습니다.  
  
-   기본 프로젝트를 예약 하 고 버짓 등 계획을 제공 합니다.  
  
-   진행 상황 Active Directory 디자인 및 배포 프로젝트 마이닝하고  
  
-   해당 하는 개인은 디자인 프로세스의 각 부분에 참여  
  
-   연락처에 대 한 Active Directory 배포 프로젝트 단일 지점으로  
  
-   디자인, 배포 및 운영 팀 간 통신 설정  
  
-   설정 및 배포 프로젝트 전체 행정 지원자와 통신 유지 관리  
  
## <a name="BKMK_2"></a>소유자와 관리자 설정  
된 Active Directory 배포 프로젝트에 소유자는 개인이 한 책임 관리를 작업이 완료 되는 해당 배포 및 specifications 조직의 요구를 충족 해당 Active Directory 설계 되어 있는지 확인 합니다. 소유자에 액세스할 수 있는 하거나 직접 디렉터리 infrastructure 조작 반드시 그렇지는 않습니다. 관리자는 배포 필요한 작업을 완료 담당 합니다. 관리자는 네트워크 액세스 및 사용 권한을 디렉터리 사항과 인프라를 조작 하는 데 필요한 가집니다.  
  
소유자의 역할은 전략 및 관리 합니다. 소유자는 숲 내에서 새로운 도메인 컨트롤러의 등의 Active Directory 디자인 구현 하는 데 필요한 작업 관리자에 게 통신 담당 합니다. 관리자는 디자인 사양에 따라 네트워크에서 디자인을 구현 합니다.  
  
큰 조직에 다른 개인 채우기 소유자와 관리자 역할; 그러나 일부 작은 조직에서 같은 개별 수 역할을 소유자와의 관리자 합니다.  
  
### <a name="service-and-data-owners"></a>소유자 서비스와 데이터  
매일 AD DS 관리 두 가지 유형의 소유자 포함 됩니다.  
  
-   디렉터리는 계속 작동 하 고 수준 서비스 계약에 설정한 목표는 유지 및 Active Directory 인프라에 계획 및 장기 유지 관리에 대 한 해야 하는 서비스 소유자  
  
-   디렉터리에 저장 된 정보를 관리 담당 하는 데이터 소유자 합니다. 사용자 컴퓨터 계정 관리 및 회원 서버, 워크스테이션 로컬 리소스 관리 포함 됩니다.  
  
많은 가능한 디자인 프로세스에 참여할 수 있도록 Active Directory 서비스와 데이터 소유자 초기 확인을 고려해 야 합니다. 배포 프로젝트 완료 된 후 서비스와 데이터 소유자 디렉터리의 장기 유지 관리 담당 되므로 것이 중요 이러한 개인 조직의 요구에 관한 입력을 제공 하 고 방법과 이유에 특정 디자인 결정을 알고 있어야 합니다. 서비스 소유자 숲 소유자, Active Directory 명명 DNS (도메인 시스템) 소유자와 사이트 토폴로지 소유자 포함 됩니다. 데이터 소유자 조직 단위 소유자 포함 됩니다.  
  
### <a name="service-and-data-administrators"></a>서비스 및 데이터 관리자  
AD DS 작업이 포함 두 가지 유형의 관리자: 관리자 및 관리자 데이터를 서비스입니다. 서비스 소유자 정책을 결정을 구현 유지 디렉터리 서비스와 인프라와 관련 된 일상적인 작업을 처리 하는 서비스 관리자입니다. 디렉터리 서비스를 관리 등 DNS DS AD 숲 수준의 설정의 구성을 제어 하 고 디렉터리는 항상 사용할 수 있도록에 필요한 다른 네트워크 서비스 호스트 하는 도메인 컨트롤러 관리 포함 됩니다.  
  
서비스 관리자는 초기 Windows Server 2008 Active Directory 배포 프로세스가 완료 되는 지속적인 Active Directory 배포 작업을 완료 해야 할 수도 있습니다. 예를 들어, 디렉터리 증가의 요청으로 서비스 관리자 추가 도메인 컨트롤러 및 설정 만들거나 제거 도메인 간에 신뢰 필요에 따라 합니다. 이런 이유이 때문에 대 한 Active Directory 배포 팀 서비스 관리자 포함 해야 합니다.  
  
조직에서 신뢰할 수 있는 사람에만 서비스 관리자 역할 지정할 주의 해야 합니다. 이러한 개인을 사용 하는 도메인 컨트롤러의 시스템 파일을 수정 하는 기능이 때문에 광고 DS의 동작을 변경할 수 있습니다. 조직에서 서비스 관리자 개인 익숙한 운영 및 보안 정책이 하는 네트워크에 되 고 해당 정책을 적용을 필요 이해 하는 있는지 확인 해야 합니다.  
  
데이터 관리자는 사용자가 도메인에 모두 AD DS 등 사용자 및 그룹 계정에에서 저장 된 데이터를 유지 하 고 유지 하는 도메인의 회원 하는 컴퓨터에 대 한 책임입니다. 데이터 관리자는 내에서 directory 개체의 하위 집합을 제어 하며 설치 또는 디렉터리 서비스의 구성 제어할 합니다.  
  
기본적으로 데이터 관리자 계정 제공 되지 않습니다. 조직에 대 한 관리할 리소스 하는 방법을 결정 하는 디자인 팀, 후 도메인의 소유자 데이터 관리자 계정을 만들고 하 고 개체 관리자 담당 하는 설정에 따라 적절 한 권한이 위임 해야 합니다.  
  
서비스에는 최소 인프라 계속 작동 하는지 확인 하는 데 필요한 조직에는 관리자의 수를 제한 하는 것이 좋습니다. 관리자 데이터는 대부분의 관리 작업을 완료할 수 있습니다. 서비스 관리자 디렉터리를 지 원하는 인프라 담당 있으므로 훨씬 더 넓게 기술이 필요 합니다. 관리자 데이터는 디렉터리의 각 부분 관리 하기 위한만 필요 합니다. 이러한 방식으로 작업 지정 분리로 인해 조직에 대해 비용을 절감 소수의 관리자 운영 및 유지 관리 전체 디렉터리 사항과 인프라 교육 하기 때문에 합니다.  
  
예를 들어, 서비스 관리자가 도메인에 숲을 추가 하는 방법을 파악 해야 합니다. 도메인 컨트롤러 서버 변환 하는 소프트웨어를 설치 하는 방법 및 Active Directory 환경에 도메인 컨트롤러를 원활 하 게 병합 수 있도록 DNS 환경 조작 하는 방법을 포함 됩니다. 만 데이터 관리자 담당 부서의 새로운 직원에 대 한 새로운 사용자 계정 만들기와 같은 특정 데이터를 관리 하는 방법을 알아야 합니다.  
  
광고 DS 배포 조정 및 다른 여러 그룹이 네트워크 인프라 작업과 관련된 간 통신 필요 합니다. 이러한 그룹 디자인 및 배포 프로세스 중에서 다양 한 그룹을 나타내는 담당 하는 서비스와 데이터 소유자 지정 해야 합니다.  
  
배포 프로젝트 완료 되 면 이러한 서비스와 데이터 소유자 관리 하 여 해당 그룹 인프라 부분의 대 한 책임을 지지 계속 합니다. 된 Active Directory 환경에서 다음이 소유자는 숲 소유자, AD DS 소유자 DNS, 사이트 토폴로지 소유자 및 OU 소유자 합니다. 이러한 서비스와 데이터 소유자의 역할을 다음 섹션에 설명 되어 있습니다.  
  
#### <a name="forest-owner"></a>숲 소유자  
일반적으로 숲 소유자가 담당자 Active Directory 배포 프로세스에 대 한 고 대 한 배포 완료 된 후 숲 내에서 제공 되는 서비스를 유지 하는 사람 조직에서 선임 정보 it 관리자입니다. 숲 소유자 할당 개인 핵심 담당자 조직 내가 네트워크 infrastructure 및 관리 요구 사항에 대 한 필요한 정보를 제공 하기 위해 수 있는 식별 하 여 다른 소유권 역할을 합니다. 숲 소유자는 다음 작업을 수행 합니다.  
  
-   숲 만드는 이렇게 배포  
  
-   각 도메인 숲에 필요한 도메인을 만드는 첫 번째 도메인 컨트롤러의 배포  
  
-   모든 숲 도메인에 있는 서비스 관리자가 그룹의 등록  
  
-   각 도메인 숲에 대해 OU 구조 디자인 만들기  
  
-   위임 OU 소유자에 게 권한 관리  
  
-   스키마에 대 한 변경  
  
-   숲 전체의 구성 설정 변경  
  
-   구현 도메인 사용자 계정 정책 세밀 암호 및 계정 잠금 정책 등을 포함 하 여 특정 그룹 정책 정책 설정  
  
-   도메인 컨트롤러에 적용 되는 비즈니스 정책 설정  
  
-   도메인 수준에 적용 되는 다른 그룹 정책 설정  
  
숲 소유자의 권한이 전체 숲 통해 합니다. 숲 소유자가 그룹 정책 및 비즈니스 정책 설정 하 고 된 서비스 관리자 사람을 선택 합니다. 숲 소유자 소유자는 서비스입니다.  
  
#### <a name="dns-for-ad-ds-owner"></a>DNS AD DS 소유자에 대 한  
DNS AD DS 소유자에는 기존 DNS 인프라를 파악 하 고 조직 기존 네임 스페이스 있는 개인입니다.  
  
DNS AD DS 소유자에은 다음을 수행 합니다.  
  
-   연락을 디자인 팀과 현재 DNS 인프라 담당 하는 IT 그룹 담당 하 여로  
  
-   새 Active Directory 네임 스페이스 생성 지원 하기 위해 조직 기존 DNS 이름에 대 한 정보를 제공 하며  
  
-   새 DNS infrastructure 디자인 팀의 배포 되 고 제대로 작동 하는지 확인 하려면 배포 팀과 함께 작동 합니다.  
  
-   DNS 서버가 서비스 및 DNS 데이터를 포함 하 여 광고 DS 인프라 DNS 관리  
  
DNS AD DS 소유자에 대 한 소유자는 서비스입니다.  
  
#### <a name="site-topology-owner"></a>사이트 토폴로지 소유자  
사이트 토폴로지 소유자가 개별 서브넷, 라우터와 슬로우 링크를 사용 하 여 연결 된 네트워크 영역의 지도 포함 하 여 조직의 네트워크의 실제 구조 잘 알고 있습니다. 사이트 토폴로지 소유자는 다음 작업을 수행 합니다.  
  
-   실제 네트워크 토폴로지와 광고 DS에 미치는 영향을 이해 합니다.  
  
-   이해 Active Directory 배포는 어떤 영향을 주나요 네트워크  
  
-   만들어야 Active Directory 논리 사이트 확인  
  
-   수정 서브넷 추가 되 면 사이트 개체 도메인 컨트롤러에 대 한 업데이트 또는 제거  
  
-   사이트 링크, 사이트 링크 다리 및 수동 연결 개체 만들기  
  
사이트 토폴로지 소유자 소유자는 서비스입니다.  
  
#### <a name="ou-owner"></a>OU 소유자  
OU 소유자는 디렉터리에 저장 된 데이터를 관리 합니다. 이 개별 익숙한 운영 및 네트워크에 장소에는 보안 정책을 수 있어야 합니다. OU 소유자 서비스 관리자에 위임 된 해당 작업을 수행할 수 고 할당 된 Ou에만 해당 작업을 수행할 수 있습니다. OU 소유자에 할당 될 수 있는 작업은 다음과 같습니다.  
  
-   할당 된 OU 자신의 모든 계정 관리 작업 수행  
  
-   워크스테이션 및 해당 할당된 OU의 회원 구성원 서버 관리  
  
-   내에서 자녀가 할당 된 OU 로컬 관리자에 게 권한을 위임  
  
OU 소유자 데이터 소유자가입니다.  
  
## <a name="BKMK_3"></a>빌딩 프로젝트 팀  
Active Directory 프로젝트 팀은 임시 그룹 Active Directory 디자인 및 배포 작업을 완료 하는 일을 담당입니다. Active Directory 배포 프로젝트 완료 되 면 소유자는 디렉터리에 대 한 책임 가정 및 프로젝트 팀 제거 수 있습니다.  
  
프로젝트 팀의 크기 조직 크기에 따라 달라 집니다. 작은 조직에서 한 사람 책임 프로젝트 팀의 여러 영역 커버 하 고 배포 개 이상의 단계에 포함 될 수 있습니다. 대기업 큰 팀 다른 사람으로 또는 다른 팀의 책임 서로 다른 영역을 다루는 필요할 수 있습니다. 모든 책임 영역 할당 된 고 조직 디자인 목표를 만족 팀의 크기 중요 하지 않습니다.  
  
### <a name="identifying-potential-forest-owners"></a>신원 파악이 가능한 잠재적 숲 소유자  
소유 하 고 네트워크에 사용자에 게 디렉터리 서비스를 제공 하는 데 필요한 리소스를 제어 하는 조직의 그룹 식별 합니다. 이러한 그룹 잠재적 숲 소유자 것으로 간주 됩니다.  
  
AD ds에서 서비스와 데이터 관리 분리 IT (그룹 하나 또는) 조직 로컬 관리자 각 그룹에서 자신의 그룹에 속해 있는 데이터를 관리 하는 동안 디렉터리 서비스를 관리 하는 인프라에 대해 수 있습니다. 잠재적 숲 소유자 배포 및 지 원하는 광고 DS 네트워크 인프라를 통해 권한이 필요 합니다.  
  
한 중앙 집중식된 인프라 IT 부서 조직의 IT 부서는 일반적으로 숲 소유자와 따라서 모든 이후 배포용 잠재적 숲 소유자 합니다. 번호를 포함 하는 조직의 그룹 독립 IT 인프라 다양 가능성 숲 소유자가 있습니다. 조직에 이미가 Active Directory infrastructure 장소에는 현재 숲 소유자의 새 배포 잠재적 숲 소유자 수도 있습니다.  
  
배포에 대 한 고려 하 고 각 숲에 대 한 숲 소유자 역할을 잠재적 숲 소유자 중 하나를 선택 합니다. 이러한 잠재적 숲 소유자는 또는 여부 자신의 숲 실제로 배포 될 경우 가기 (예: 다른 기존 숲 가입) 과정을 대신 사용 가능한 리소스를 사용 하는 더 나은 이며 여전히의 요구를 충족 결정 하기 위해 디자인 팀과 함께 작동 합니다. 숲 속의 소유자가 (또는 소유자) 조직에서 Active Directory 디자인 팀의 구성원 됩니다.  
  
### <a name="establishing-a-design-team"></a>설정 디자인 팀  
Active Directory 디자인 팀은에 대 한 Active Directory 논리 구조 디자인 결정 하는 데 필요한 모든 정보를 수집 합니다.  
  
디자인 팀의 책임은 다음과 같습니다.  
  
-   개수 숲 및 도메인이 필요 하 고 숲와 도메인 간의 관계는 결정  
  
-   디자인의 보안 및 관리 요구 사항을 충족 하는지 확인 하는 데이터 소유자와 협력  
  
-   현재 네트워크 infrastructure 디자인을 지원 하 고 디자인으로 네트워크 배포 기존 응용 프로그램 악영향 영향 구현 현재 네트워크 관리자에 사용  
  
-   디자인 기존된 보안 정책이 충족 하는지 확인 하는 조직의 보안 그룹 담당자 사용  
  
-   OU 구조 데이터 소유자에 게 적절 한 수준의 보호 하 고 적절 한 위임 기관의 허용 하는 디자인  
  
-   디자인 테스트를 배포 팀과 함께 작동 랩에서으로 작동 하는지 확인 하려면 환경을 계획 하 고 발생 하는 문제를 해결 하는 데 필요한 디자인을 수정  
  
-   대역폭이 오버 로드 것을 방지 하는 동안 숲의 복제 요구 사항을 충족 사이트 토폴로지 디자인을 작성 합니다. 사이트 토폴로지 디자인에 대 한 자세한 내용은 참조 [사이트 토폴로지에 대 한 Windows Server 2008 광고 DS 디자인](https://technet.microsoft.com/library/cc772013.aspx)합니다.  
  
-   디자인 올바로 구현 되도록 배포 팀과 함께 작동 합니다.  
  
디자인 팀 멤버 포함 됩니다.  
  
-   숲 소유자 잠재적  
  
-   프로젝트 설계자  
  
-   프로젝트 관리자  
  
-   개인 설정 하 고 네트워크에서 보안 정책이 관리 담당 합니다.  
  
논리 구조 디자인 프로세스 중 디자인 팀의 다른 소유자 식별합니다. 이러한 개인 식별 되는 즉시 디자인 프로세스에 참여 시작 해야 합니다. 배포 프로젝트 배포 팀에 전달는, 한 후 디자인 팀은을 위해 디자인 올바르게 구현는 배포 프로세스를 감독 합니다. 디자인 팀 하면 디자인 테스트에서 피드백에 따라 변경 됩니다.  
  
### <a name="establishing-a-deployment-team"></a>배포 팀 설정  
Active Directory 배포 팀 테스트 하 고 Active Directory 논리 구조 디자인을 구현 담당 합니다. 이 다음과 같은 작업이 포함 됩니다.  
  
-   충분히 생산 환경 에뮬레이트하 테스트 환경을 설정 하 고  
  
-   디자인 각 역할 소유자 목표에 맞는지 확인 하는 환경에서 제안된 된 숲 및 도메인 구조 구현 하 여 테스트  
  
-   개발 하 고 모든 마이그레이션 시나리오를 테스트 랩 환경에서 제안  
  
-   각 소유자 로그인 테스트 프로세스에 올바른 디자인 기능을 테스트 확인 하 않도록  
  
-   배포 작업이 시험 환경에서 테스트  
  
디자인 하 고 테스트 작업이 완료 되 면 배포 팀에서는 다음과 같은 작업을 수행 합니다.  
  
-   숲 만들고 Active Directory 논리 구조 디자인에 따라 도메인  
  
-   사이트와 필요한 사이트 링크 개체 사이트 토폴로지 디자인에 따라 만듭니다.  
  
-   DNS infrastructure AD DS을 지원 하도록 구성 된 모든 새로운 네임 스페이스 기존 네임 조직에 통합 되어 블  
  
Active Directory 배포 팀 멤버 포함 됩니다.  
  
-   숲 소유자  
  
-   DNS AD DS 소유자에 대 한  
  
-   사이트 토폴로지 소유자  
  
-   OU 소유자  
  
배포 팀은 운영 팀의 구성원 새로운 디자인 익숙한 되도록 배포 단계 서비스와 데이터 관리자가 작동 합니다. 이렇게 하면 배포 작업이 완료 되 면 소유 하 게 전환 하도록 합니다. 배포 프로세스를 완료 되 면 새 Active Directory 환경을 유지에 대 한 책임 운영 팀에 전달 합니다.
